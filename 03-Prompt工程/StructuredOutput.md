# StructuredOutput（结构化输出）

> 通过Prompt设计让AI生成JSON、XML、表格等结构化数据，实现与程序的无缝对接

---

## 一、结构化输出概述

### 1.1 什么是结构化输出

**结构化输出** 是指通过Prompt引导大语言模型生成具有固定格式、可被程序直接解析的数据（如JSON、XML、Markdown表格等），而非自由文本。

```
结构化输出的价值：

自由文本输出：
"这款产品的优点有很多。首先，它的设计非常精美，
 用户体验很好。其次，功能很强大，能满足各种需求。
 不过价格有点贵，可能不适合预算有限的用户。"
→ 程序难以提取关键信息

结构化输出（JSON）：
{
  "优点": ["设计精美", "用户体验好", "功能强大"],
  "缺点": ["价格较高"],
  "适用人群": "预算充足的用户",
  "总体评价": "推荐"
}
→ 程序可直接解析和使用
```

### 1.2 为什么需要结构化输出

| 场景 | 问题 | 结构化输出解决 |
|------|------|---------------|
| **API对接** | AI输出需要被下游系统消费 | JSON/XML直接解析 |
| **数据库存储** | 需要提取特定字段 | 固定字段名和类型 |
| **前端渲染** | 需要按组件展示 | 结构化数据映射到UI组件 |
| **工作流串联** | 多步骤AI处理 | 上一步输出作为下一步输入 |
| **结果验证** | 需要校验输出完整性 | _schema验证 |

## 二、常见结构化格式

### 2.1 JSON格式

最常用、最灵活的结构化格式。

```
JSON输出示例：

Prompt：
"请分析以下用户反馈，并以JSON格式输出分析结果：

反馈：'这个APP界面很好看，但是经常闪退，
       希望能尽快修复， otherwise 就要卸载了'

输出格式要求：
{
  \"sentiment\": \"正面/负面/混合\",
  \"sentiment_score\": 0-1之间的数字,
  \"key_issues\": [\"问题1\", \"问题2\"],
  \"key_praises\": [\"优点1\", \"优点2\"],
  \"urgency\": \"高/中/低\",
  \"action_required\": true/false,
  \"suggested_response\": \"建议的回复内容\"
}"

输出：
{
  "sentiment": "混合",
  "sentiment_score": 0.4,
  "key_issues": ["经常闪退"],
  "key_praises": ["界面好看"],
  "urgency": "高",
  "action_required": true,
  "suggested_response": "感谢您的反馈！界面设计得到您的认可我们非常开心。关于闪退问题，我们已记录并会优先修复，预计本周内发布更新。"
}
```

### 2.2 Markdown表格

适合对比、列表类输出。

```
表格输出示例：

Prompt：
"请对比以下三个方案，以Markdown表格输出：

方案A：自研AI客服系统
方案B：采购第三方SaaS
方案C：混合方案（自研+第三方）

对比维度：开发成本、维护成本、定制化程度、上线周期、数据安全

输出格式：
| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|"

输出：
| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 开发成本 | 高（需组建团队） | 低（按量付费） | 中 |
| 维护成本 | 高（持续投入） | 低（供应商维护） | 中 |
| 定制化程度 | 高 | 低 | 中 |
| 上线周期 | 6-12个月 | 1-2周 | 2-3个月 |
| 数据安全 | 高（数据自有） | 中（依赖第三方） | 高 |
```

### 2.3 XML格式

适合嵌套层次复杂的结构。

```
XML输出示例：

Prompt：
"请提取以下文章中的实体关系，以XML格式输出：

文章：'马云创立了阿里巴巴，阿里巴巴总部位于杭州。'

输出格式：
<entities>
  <entity type="人名">...</entity>
  <entity type="公司">...</entity>
  <entity type="地点">...</entity>
</entities>
<relations>
  <relation type="创立" from="..." to="..."/>
  <relation type="位于" from="..." to="..."/>
</relations>"

输出：
<entities>
  <entity type="人名">马云</entity>
  <entity type="公司">阿里巴巴</entity>
  <entity type="地点">杭州</entity>
</entities>
<relations>
  <relation type="创立" from="马云" to="阿里巴巴"/>
  <relation type="位于" from="阿里巴巴" to="杭州"/>
</relations>
```

### 2.4 YAML格式

适合配置类、层次结构数据。

```
YAML输出示例：

Prompt：
"请为以下产品生成功能配置说明，以YAML格式输出：

产品：智能客服机器人

输出格式：
product:
  name: ...
  features:
    - name: ...
      description: ...
      enabled: ..."

输出：
product:
  name: 智能客服机器人
  features:
    - name: 自动回复
      description: 基于知识库自动回答常见问题
      enabled: true
    - name: 意图识别
      description: 识别用户咨询意图并分类
      enabled: true
    - name: 多轮对话
      description: 支持上下文理解的多轮交互
      enabled: false
```

## 三、结构化输出设计技巧

### 3.1 明确的格式定义

```
格式定义的最佳实践：

✅ 好的格式定义：
"请以JSON格式输出，包含以下字段：
- title: 字符串，文章标题
- summary: 字符串，100字以内摘要
- tags: 字符串数组，最多5个标签
- sentiment: 枚举值，可选：positive/negative/neutral
- key_points: 字符串数组，核心观点列表"

❌ 差的格式定义：
"请以JSON输出"
→ 模型不知道包含哪些字段

❌ 过于严格：
"请输出JSON，字段必须是：title（字符串，不超过20个字符，
 必须包含关键词'AI'），summary（字符串，正好100个字符）..."
→ 约束过多，模型难以同时满足
```

### 3.2 字段类型说明

| 类型 | 说明 | 示例 |
|------|------|------|
| **string** | 字符串 | "title": "产品分析报告" |
| **number** | 数字 | "score": 85.5 |
| **integer** | 整数 | "count": 10 |
| **boolean** | 布尔值 | "is_valid": true |
| **array** | 数组 | "tags": ["AI", "产品"] |
| **object** | 对象 | "author": {"name": "张三", "age": 30} |
| **enum** | 枚举 | "priority": "high/medium/low" |

### 3.3 处理嵌套结构

```
嵌套JSON设计：

Prompt：
"请分析以下用户画像，以嵌套JSON输出：

用户：25岁女性，一线城市，月收入15000，
      喜欢购买护肤品和化妆品，月均消费3000元

输出格式：
{
  \"basic_info\": {
    \"age\": 数字,
    \"gender\": \"男/女\",
    \"city_tier\": \"一线/二线/三线\"
  },
  \"financial\": {
    \"monthly_income\": 数字,
    \"monthly_spending\": 数字,
    \"spending_ratio\": 数字（消费占收入比例）
  },
  \"preferences\": {
    \"categories\": [\"品类1\", \"品类2\"],
    \"price_sensitivity\": \"高/中/低\"
  },
  \"marketing_suggestions\": [\"建议1\", \"建议2\"]
}"
```

## 四、结构化输出的可靠性保障

### 4.1 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **格式不合法** | JSON缺少引号、括号不匹配 | 要求模型先输出格式说明，再输出数据 |
| **字段缺失** | 模型遗漏某些字段 | 明确标注"必须包含以下所有字段" |
| **类型错误** | 字符串写成了数字 | 在Prompt中明确每个字段的类型 |
| **额外内容** | 输出包含JSON外的解释文字 | 要求"只输出JSON，不要任何其他文字" |
| **中文编码** | 中文字符被转义 | 要求"保持中文字符原样，不要转义" |

### 4.2 输出校验流程

```
结构化输出校验流程：

模型输出
  │
  ▼
┌─────────────────┐
│ 1. 格式校验     │
│    - JSON/XML是否可解析 │
│    - 括号/引号是否匹配  │
└─────────────────┘
  │
  ▼
┌─────────────────┐
│ 2. 字段校验     │
│    - 必需字段是否都存在 │
│    - 字段类型是否正确   │
└─────────────────┘
  │
  ▼
┌─────────────────┐
│ 3. 值域校验     │
│    - 枚举值是否在范围内 │
│    - 数值是否在合理区间 │
└─────────────────┘
  │
  ▼
┌─────────────────┐
│ 4. 业务校验     │
│    - 数据是否符合业务逻辑│
└─────────────────┘
  │
  ▼
使用数据 / 返回错误要求重试
```

### 4.3 容错设计

```
Prompt中的容错引导：

"请以JSON格式输出，遵循以下规则：

1. 如果信息不足，对应字段填 null，不要编造
2. 如果不确定，confidence字段填低分
3. 如果无法判断，reason字段说明原因
4. 必须确保JSON格式合法，键名用双引号
5. 只输出JSON，不要添加任何解释文字

示例：
{
  \"result\": null,
  \"confidence\": 0.1,
  \"reason\": \"输入信息不足，无法判断\"
}"
```

## 五、AI产品经理的结构化输出关注点

### 5.1 产品中的结构化输出架构

```
AI产品中的结构化输出流程：

用户输入
  │
  ▼
Prompt模板（含格式定义）
  │
  ▼
大模型API
  │
  ▼
输出解析层
  ├── JSON/XML解析
  ├── 字段校验
  └── 默认值填充
  │
  ▼
业务逻辑层
  │
  ▼
前端展示 / 数据库存储 / 下游系统

产品决策点：
1. 选择什么格式？（JSON/表格/XML）
2. 字段如何设计？（业务需求驱动）
3. 校验规则是什么？（严格 vs 宽松）
4. 解析失败如何处理？（重试/降级/报错）
```

### 5.2 与开发团队协作

| 协作内容 | AI产品经理 | 开发工程师 |
|---------|-----------|-----------|
| **接口设计** | 定义输出字段和类型 | 实现解析和校验逻辑 |
| **错误处理** | 定义降级策略 | 实现异常处理 |
| **版本管理** | 评估字段变更影响 | 实现兼容性处理 |
| **性能优化** | 评估输出大小对延迟的影响 | 优化解析性能 |

1. 结构化输出是AI产品工程化的基础，实现AI能力与业务系统的桥接
2. JSON是最通用、最推荐的结构化输出格式
3. 格式定义要明确但不过度约束，给模型适当自由度
4. 必须设计输出校验流程，确保数据可靠性
5. AI产品经理需要与开发团队紧密协作，定义清晰的接口规范
