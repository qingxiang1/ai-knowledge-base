# Project05 - AI数据分析

> 这是一个适合学习 AI 数据分析产品原型的项目实战案例。当前项目采用 `React + TypeScript + Vite + Express` 实现，支持输入文本或 CSV 数据、选择分析类型并返回分析结果与关键洞察，但它还不是一个已经具备数据接入、可视化看板、指标体系、SQL 分析和报表协作能力的企业级分析平台。

## 一、先理解这个项目在练什么

这个项目最适合训练的是以下三类能力：

1. 如何把“大模型分析能力”包装成一个面向业务的分析工具。
2. 如何把用户输入从自由文本约束成明确的分析任务。
3. 如何理解数据分析原型与真正 BI/数据平台之间的差距。

如果你是初学者，这个项目能帮助你快速理解一个 AI 分析工具最小 MVP 的典型结构：

- 输入数据
- 选择分析类型
- 查看分析结果

如果你是技术人员，这个项目适合你练习：

- React 交互式标签切换
- 文本输入与结果展示
- 前后端接口串联
- mock/API 双模式切换
- 结构化分析结果设计

如果你是想转型 AI 产品经理，这个项目很适合你理解：

- AI 数据分析产品如何围绕“分析任务”来设计入口
- 为什么“能总结数据”不等于“能支撑业务决策”
- 为什么分析结果、洞察和图表通常需要一起设计

## 二、项目目标与用户场景

这个项目要模拟的问题可以概括为一句话：

> 用户把一段数据内容贴入系统后，选择分析任务类型，系统快速生成一版分析结果和关键洞察，帮助用户完成初步理解。

它适合模拟的典型场景包括：

- 运营同学快速看一份数据摘要
- 产品同学快速判断数据趋势
- 团队快速识别异常值
- 做一版初步相关性分析

这类场景很适合做成 AI 产品原型，因为它天然具备：

- 数据输入明确
- 分析任务可选
- 输出结果可解释

但要注意，当前项目更准确的定位是“分析结果生成原型”，而不是完整分析平台。

## 三、当前项目能力概览

根据真实代码，这个项目当前已经实现了：

- `数据摘要 / 趋势分析 / 异常检测 / 相关性分析` 四种分析入口
- 文本或 CSV 数据粘贴输入
- 前端结果展示
- 前端关键洞察展示
- 后端 mock/API 双模式
- 后端 mock 模式返回图表结构数据

但它当前也有非常明确的边界：

- 不支持文件上传
- 不支持数据库接入
- 不支持 SQL 查询
- 不支持图表真实渲染
- 前端当前没有展示后端返回的 `charts`
- API 模式下当前只返回文本结果，`insights` 和 `charts` 仍为空结构

因此，这个项目更适合被理解为：

> 一个把“分析任务”做成清晰界面和结果卡片的 AI 原型，而不是成熟 BI 产品。

## 五、技术栈与实现方式

### 1. 当前真实可运行部分

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 模型 SDK：OpenAI Node SDK
- 本地开发：前端 `5173`，后端 `3000`

### 2. 当前最值得注意的实现特点

这个项目虽然简单，但产品表达很清楚：

- 不让用户直接问“你帮我分析一下”
- 而是先约束分析任务类型

这其实是一个很重要的产品设计点，因为分析产品通常要通过任务模板来降低用户输入成本。

### 3. 当前没有落地的企业级分析能力

当前项目还没有实现：

- 实时数据源接入
- 指标计算逻辑
- 看板配置
- 权限隔离
- 图表交互
- 报表导出
- 多维下钻分析

所以你在阅读这个项目时，重点不是它“像不像完整 BI 工具”，而是它有没有把最小分析闭环做清楚。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置如下：

```env
OPENAI_API_KEY=
PORT=3000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：进入 mock 模式，适合演示和学习
- 配置 `OPENAI_API_KEY`：进入 API 模式，调用真实模型生成分析结果

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:3000/health`

项目在 `vite.config.ts` 中把 `/api` 代理到了 `http://localhost:3000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/components/DataAnalyzer.tsx`

这是整个项目最关键的前端文件。

它把数据分析工具最核心的体验拆得很清楚：

- 先选分析类型
- 再输入数据
- 然后展示结果和洞察

这个顺序很重要，因为它比“丢一堆数据给系统”更贴近真实业务使用方式。

### 2. `src/services/api.ts`

这个文件封装了 `analyzeData()` 请求。

它的价值在于：

- 统一接口调用
- 统一错误处理
- 给未来扩展埋点、鉴权和超时处理留空间

### 3. `src/types/index.ts`

这里定义了前端预期的数据分析返回结构：

- `result`
- `insights`
- `charts`

这说明项目设计上已经意识到一个很重要的事实：

> 数据分析产品不能只给一段分析文字，通常还要给洞察和图表。

只是当前前端还没有把 `charts` 这部分真正展示出来。

### 4. `server/routes/analysis.ts`

这是当前项目最值得认真看的后端文件。

它完成了以下事情：

- 根据分析类型生成不同指令语义
- mock 模式下返回结果、洞察和图表结构
- API 模式下接入真实模型

其中最值得注意的是：

- mock 模式结果更完整
- API 模式当前只返回 `result`

这说明项目还处在“界面结构先定好，但真实模型结构化输出还没补齐”的阶段。

### 5. `server/index.ts`

这个文件展示了最轻量的分析工具后端结构：

- 注册 `/api`
- 提供 `/health`
- 标注当前运行模式

对于学习者来说，这个结构足够简单，也足够实用。

## 八、从产品和技术角度，应该如何理解这个项目

### 1. 从产品角度看

这个项目已经具备分析产品原型最重要的三个感觉：

- 有明确分析任务入口
- 有结果输出
- 有关键洞察表达

这足以支撑一个数据分析方向的演示原型。

### 2. 从技术角度看

它当前主要完成的是：

- 前端分析任务切换
- 数据输入与结果展示
- 后端分析接口
- mock/API 双模式切换

而真正决定数据分析产品价值的底层能力还没有落地，例如：

- 图表引擎
- 数据模型
- 指标定义
- 多源数据接入
- 自动报表
- 下钻和筛选能力

### 3. 从学习角度看

这个项目很适合作为“AI 分析产品最小化表达”的练习样本：

1. 先把分析任务做成结构化入口。
2. 再把结果拆成文字、洞察、图表。
3. 最后再往真正的数据平台演化。

## 九、如果你是产品经理，应该重点观察什么

### 1. 数据分析产品不只是“总结一下数据”

很多人第一次做 AI 数据分析时，会把它理解成：

- 把数据扔进去
- 让模型说几句结论

但真正有价值的分析产品通常还要回答：

- 分析的是哪类问题
- 为什么得出这个结论
- 结论是否可信
- 用户下一步要做什么

当前项目已经通过“分析类型”这一步把问题先约束住了。

### 2. 洞察和图表通常要一起设计

后端 mock 模式已经返回了 `charts`，说明设计思路是对的。

只是当前前端还没渲染出来。

这正好提醒你一个很重要的产品问题：

> 后端有能力，不代表前端已经把价值完整交付给用户。

### 3. 企业分析系统真正难的是数据治理

真实企业里，分析产品最难的往往不是生成几句结论，而是：

- 数据从哪来
- 指标怎么算
- 图表给谁看
- 权限怎么控
- 口径是否统一

当前项目适合作为一个“分析表达层先完成，治理层还没展开”的学习样本。

## 十、建议你至少做一次改造

如果你要把这个项目真正写进作品集，建议至少做一次有价值的改造。

### 改造方向 1：把图表结果真正渲染出来

这是最自然的一步升级。

后端 mock 模式已经返回了图表结构，你可以在前端增加：

- 柱状图
- 折线图
- 饼图

至少把 `charts` 数据真正展示出来。

### 改造方向 2：补齐 API 模式结构化输出

当前真实模型模式下只返回 `result`。

你可以继续补：

- 让模型输出结构化 JSON
- 解析出 `insights`
- 解析出图表建议
- 增加失败兜底逻辑

### 改造方向 3：支持 CSV 文件上传

当前只能手动粘贴数据。

你可以继续补：

- 上传 CSV
- 解析表头
- 自动识别数值列
- 显示数据预览

### 改造方向 4：增加分析历史

当前每次分析结果只留在当前页面中。

你可以增加：

- 历史记录
- 再分析
- 对比前后结果

### 改造方向 5：增加指标模板

如果想往企业级方向推进，可以继续补：

- 销售分析模板
- 留存分析模板
- 漏斗分析模板
- 客服质量分析模板

## 十一、怎么把这个项目讲成作品集

面试时不要只说“我做了一个 AI 数据分析工具”，而要讲清楚你对分析产品结构的理解。

你可以这样表达：

> 我做了一个 AI 数据分析原型，前端提供数据摘要、趋势分析、异常检测和相关性分析等任务入口，后端实现了 mock/API 双模式分析接口。我在这个项目中重点关注分析产品从自由输入到结构化分析任务的转化过程，并识别了洞察展示、图表渲染、结构化输出和数据接入能力上的后续演化方向。

这种表达会比“做了个分析页面”更有产品含量。

## 十二、最小复盘模板

完成这个项目后，建议你至少回答下面几个问题：

1. 为什么分析产品需要先定义分析类型，而不是只让用户自由提问？
2. 当前 mock 模式和 API 模式的能力差异体现在哪里？
3. 后端已有 `charts`，前端没展示，这说明了什么产品状态？
4. 如果只能做一次升级，你会优先补图表、CSV 上传还是历史记录？为什么？
5. 企业分析产品最关键的价值是生成结论，还是统一数据口径和决策闭环？

如果这些问题你都能回答清楚，这个项目就真正沉淀成你的产品理解了。

## 十三、这个项目适合作为什么阶段的练习

这个项目很适合作为以下阶段的训练样本：

- 想从通用聊天产品转向分析类 AI 产品
- 想练习“任务型分析工具”的最小 MVP
- 想理解结构化分析结果如何设计
- 想向 AI 产品经理转型，需要一个兼具业务表达和技术边界的案例

如果你前面已经做过翻译、知识库、Agent、客服、会议纪要等项目，那么这个项目会帮助你进入“数据洞察类 AI 产品”的下一阶段。
