# 用户需求分析

> 系统化的需求挖掘与分析方法，从用户痛点出发，找到真正值得解决的问题

---

## 一、需求分析的核心认知

### 1.1 什么是真正的需求

**需求 = 用户想要达成的目标 - 现有解决方案的不足**

用户表达的往往是「解决方案」而非「真实需求」：

| 用户说的               | 真实需求       | 更好的解决方案  |
| ---------------------- | -------------- | --------------- |
| "我想要更快的马"       | 更快到达目的地 | 汽车            |
| "我想要更大的手机"     | 更好的阅读体验 | 平板电脑        |
| "我想要更长的电池续航" | 消除电量焦虑   | 快充 + 电池优化 |

### 1.2 AI产品需求分析的特殊性

```
传统产品需求：用户想要什么功能
AI产品需求：用户想要完成什么任务 + AI能如何帮助完成
```

AI产品需求分析需要额外考虑：

- **AI能力边界**：当前技术能否实现
- **人机协作模式**：AI做多少，人做多少
- **效果预期管理**：用户对AI准确性的容忍度
- **成本效益**：AI方案是否比传统方案更优

---

### 1.3 企业级需求判断框架

> 在真实企业环境里，需求分析不是“收集一堆声音”，而是对问题进行筛选、验证和取舍。

#### 1.3.1 需求判断的四个层次

| 层次   | 要回答的问题               | 常见错误                   |
| ------ | -------------------------- | -------------------------- |
| 用户层 | 用户遇到了什么问题？       | 只记录用户说法，不追问场景 |
| 业务层 | 这个问题是否值得优先解决？ | 只看投诉声量，不看业务价值 |
| 方案层 | AI 是否真的是更优解？      | 默认有 AI 就一定更先进     |
| 交付层 | 如何定义验收、风险和边界？ | 只提需求，不定义完成标准   |

#### 1.3.2 判断一个需求是否值得做

建议至少从以下 5 个角度判断：

1. **痛点强度**：问题是否高频、高损耗或高情绪
2. **目标用户规模**：影响的是核心用户还是边缘用户
3. **业务价值**：是否能提升收入、效率、留存或满意度
4. **技术可行性**：当前 AI / 非 AI 方案是否能稳定实现
5. **风险可控性**：错误输出、误导、越权和成本是否可接受

#### 1.3.3 什么时候不该上 AI

以下场景要特别谨慎：

- 问题规则明确、传统规则即可稳定解决
- 错误代价极高，且缺乏审核与兜底机制
- 用户诉求高度标准化，AI 反而增加成本与不确定性
- 当前没有可用数据、知识源或评测方式支撑验证

## 二、需求收集方法

### 2.1 用户访谈

#### 访谈准备

| 准备项   | 说明                  | 示例                             |
| -------- | --------------------- | -------------------------------- |
| 目标用户 | 明确访谈对象画像      | 使用AI写作工具的自媒体从业者     |
| 访谈目标 | 每次访谈聚焦1-2个问题 | 了解用户写作流程中的痛点         |
| 问题清单 | 开放式问题为主        | "描述一下你最近一次写作的全过程" |
| 样本数量 | 建议5-10人/批次       | 直到信息饱和                     |

#### 访谈技巧：5W1H + STAR

```
5W1H框架：
- What：用户在做什么？
- Why：为什么这么做？
- When：什么时候做？
- Where：在哪里做？
- Who：和谁一起做？
- How：具体怎么做的？

STAR法则（追问细节）：
- Situation：当时是什么情况？
- Task：你的任务是什么？
- Action：你具体做了什么？
- Result：结果如何？
```

#### 访谈问题示例（AI写作产品）

| 阶段 | 问题                                       | 目的       |
| ---- | ------------------------------------------ | ---------- |
| 开场 | "请描述一下你典型的工作日"                 | 建立上下文 |
| 探索 | "写作时最让你头疼的环节是什么？"           | 发现痛点   |
| 深挖 | "当时具体发生了什么？你怎么处理的？"       | 了解细节   |
| 验证 | "如果有工具能自动完成大纲，对你有帮助吗？" | 验证假设   |
| 收尾 | "还有什么我没问到但你觉得重要的？"         | 收集遗漏   |

### 2.2 问卷调查

#### 问卷设计原则

```
好的问卷 = 明确的目标 + 简洁的问题 + 合理的逻辑
```

| 原则     | 说明               | 反面示例                         | 正面示例                 |
| -------- | ------------------ | -------------------------------- | ------------------------ |
| 一题一问 | 每个问题只问一件事 | "你对产品的功能和价格满意吗？"   | "你对产品功能满意吗？"   |
| 避免引导 | 不暗示答案         | "大多数人都喜欢这个功能，你呢？" | "你对这个功能的看法是？" |
| 可量化   | 尽量用选择题       | "你觉得产品怎么样？"             | "请为产品打分（1-5分）"  |
| 控制长度 | 5分钟内完成        | 50道题的问卷                     | 10-15道核心问题          |

#### AI产品问卷示例

```markdown
## AI写作助手用户调研问卷

### 基本信息

1. 您的职业？ [单选]
   □ 自媒体从业者 □ 企业文案 □ 学生 □ 其他

2. 您每周写作频率？ [单选]
   □ 每天 □ 每周3-5次 □ 每周1-2次 □ 很少

### 痛点挖掘

3. 写作中最耗时的环节是？ [多选]
   □ 构思选题 □ 查找资料 □ 撰写正文 □ 修改润色

4. 您是否使用过AI写作工具？ [单选]
   □ 经常使用 □ 偶尔使用 □ 尝试过 □ 从未使用

### 需求验证

5. 以下功能中，您最看重哪些？ [排序]
   □ 自动生成大纲 □ 智能续写 □ 风格改写 □ 错别字检查

6. 您能接受AI生成内容的修改比例？ [单选]
   □ 几乎不用改 □ 小改即可 □ 需要大幅修改 □ 完全不可用
```

### 2.3 场景观察法

**适用场景**：用户无法准确表达需求，或行为与表达不一致时

**观察维度**：

| 维度     | 观察内容           | AI产品特别关注     |
| -------- | ------------------ | ------------------ |
| 行为路径 | 用户如何完成任务   | 哪些环节可以AI化   |
| 痛点时刻 | 用户何时表现出挫败 | AI能否缓解         |
| 变通方案 | 用户如何绕过限制   | 是否有更好的AI方案 |
| 情绪变化 | 用户的情绪起伏     | 体验优化点         |

### 2.4 竞品分析

**分析框架**：

```
竞品功能对比表：
| 功能 | 竞品A | 竞品B | 我们的机会 |
|------|-------|-------|-----------|
| 核心功能 | ✓ | ✓ | 差异化点 |
| 特色功能 | ✓ | ✗ | 可借鉴 |
| 缺失功能 | ✗ | ✗ | 机会点 |
```

## 三、需求分析方法

### 3.1 需求分类

```
需求分类矩阵：

            重要
              │
    紧急但不重要 │ 重要且紧急
    （快速处理） │ （立即解决）
              │
    ──────────┼──────────
              │
    不重要不紧急 │ 重要不紧急
    （考虑放弃） │ （规划实现）
              │
            不重要
```

### 3.2 KANO模型

| 需求类型 | 用户反应                     | 产品策略   | 示例（AI客服）   |
| -------- | ---------------------------- | ---------- | ---------------- |
| 基本型   | 没有会不满，有了觉得理所当然 | 必须满足   | 能回答常见问题   |
| 期望型   | 性能越好越满意               | 持续优化   | 回答准确率       |
| 兴奋型   | 超出预期，产生惊喜           | 差异化竞争 | 主动预测用户问题 |
| 无差异   | 有无都不在意                 | 不投入     | 花哨的动画效果   |
| 反向型   | 有了反而不满                 | 避免       | 过度热情的推销   |

### 3.3 需求优先级评估

#### RICE评分法

```
RICE Score = (Reach × Impact × Confidence) / Effort

- Reach（影响范围）：每月影响多少用户？
- Impact（影响程度）：0.25（低）、0.5（中）、1（高）、2（极高）、3（ massive）
- Confidence（信心指数）：50%、80%、100%
- Effort（投入成本）：需要多少人月？
```

#### AI产品优先级特殊考量

| 考量因素   | 说明                | 评估问题                       |
| ---------- | ------------------- | ------------------------------ |
| 技术可行性 | AI能否实现          | 当前模型能力能否达到可用水平？ |
| 数据可得性 | 是否有训练/评测数据 | 是否有足够的高质量数据？       |
| 成本效益   | 投入产出比          | 每次请求成本 vs 用户价值       |
| 风险可控   | 错误后果            | AI出错的影响是否在可接受范围？ |

---

### 3.4 需求验证与反证

#### 3.4.1 需求验证的常见证据

| 证据类型   | 说明                         | 例子                     |
| ---------- | ---------------------------- | ------------------------ |
| 定性证据   | 用户访谈、客服记录、销售反馈 | 用户频繁抱怨查找资料耗时 |
| 定量证据   | 数据报表、埋点、转化漏斗     | 某流程流失率高于预期     |
| 竞品证据   | 竞品是否已验证该需求         | 竞品上线后满意度显著提升 |
| 实验性证据 | 灰度、MVP、人工模拟流程      | 人工代跑验证用户愿意使用 |

#### 3.4.2 反证意识

需求分析时，除了找“支持这个需求存在”的证据，也要主动寻找反证：

- 这个问题是否只发生在少数极端用户身上？
- 用户说自己需要，是否真的会持续使用？
- 问题是否源于流程设计差，而非功能缺失？
- 用户当前是否已经有低成本替代方案？

## 四、需求文档化

### 4.1 用户故事（User Story）

**格式**：

```
作为 <角色>，
我希望 <功能/能力>，
以便 <价值/目标>。
```

**AI产品用户故事示例**：

```markdown
故事1：
作为 客服人员，
我希望 AI能自动回复常见问题，
以便 我能专注于处理复杂投诉。

验收标准：

- AI能覆盖80%的常见咨询
- 回答准确率 > 90%
- 用户可一键转人工
- 平均响应时间 < 2秒

故事2：
作为 内容创作者，
我希望 AI能根据我的写作风格生成初稿，
以便 我节省构思时间。

验收标准：

- 能学习并模仿用户过往文章风格
- 生成内容相关性 > 85%
- 用户修改率 < 50%
```

### 4.2 需求池管理

```markdown
## 需求池模板

| 编号    | 需求描述         | 来源     | 优先级 | 状态   | 负责人 | 备注             |
| ------- | ---------------- | -------- | ------ | ------ | ------ | ---------------- |
| REQ-001 | 智能客服自动回复 | 用户访谈 | P0     | 已排期 | 张三   | 技术可行性已确认 |
| REQ-002 | 写作风格学习     | 问卷调研 | P1     | 待评估 | 李四   | 需评估数据需求   |
| REQ-003 | 多语言支持       | 竞品分析 | P2     | 待讨论 | 王五   | Q3规划           |
```

---

### 4.3 企业级需求输出要求

#### 4.3.1 一份合格需求分析结果至少包含

| 模块     | 最低要求                           |
| -------- | ---------------------------------- |
| 问题定义 | 目标用户、场景、核心问题、业务影响 |
| 证据说明 | 至少包含访谈/数据/竞品中的两类证据 |
| 需求拆解 | 主需求、子需求、边界需求分层明确   |
| 优先级   | 有清晰排序依据，而非拍脑袋         |
| 建议方案 | 至少说明 AI / 非 AI 方案的取舍逻辑 |
| 风险说明 | 明确错误代价、成本风险与兜底思路   |

#### 4.3.2 从需求到 PRD 的衔接

建议在需求分析结束时输出以下内容，方便直接进入方案设计：

1. 核心用户故事
2. MVP 范围建议
3. 成功指标初稿
4. 关键风险清单
5. 待验证假设列表

## 五、AI产品需求分析实战

### 5.1 案例分析：AI客服系统需求分析

#### 背景

某电商网站希望引入AI客服，降低人工客服成本。

#### 需求分析过程

**Step 1：用户访谈**

- 访谈对象：10名客服、20名用户
- 关键发现：
  - 80%咨询是重复问题（订单状态、退换货政策）
  - 用户最在意的是「快速得到准确答案」
  - 复杂问题需要人工介入

**Step 2：需求分类**

| 需求             | 类型   | 优先级 | 实现方式        |
| ---------------- | ------ | ------ | --------------- |
| 自动回复常见问题 | 基本型 | P0     | 知识库 + RAG    |
| 理解用户意图     | 期望型 | P0     | 意图识别模型    |
| 多轮对话         | 期望型 | P1     | 对话管理        |
| 情绪识别         | 兴奋型 | P2     | 情感分析        |
| 主动推荐         | 兴奋型 | P2     | 用户画像 + 推荐 |

**Step 3：技术可行性评估**

| 需求     | 技术方案     | 准确率预估 | 成本/次 |
| -------- | ------------ | ---------- | ------- |
| 自动回复 | RAG + GPT-4  | 90%+       | ¥0.05   |
| 意图识别 | 微调分类模型 | 95%        | ¥0.01   |
| 多轮对话 | 对话状态跟踪 | 85%        | ¥0.03   |

**Step 4：MVP定义**

- 范围：仅覆盖Top 20常见问题
- 目标：解决60%咨询，准确率>90%
- 兜底：不满意一键转人工

### 5.2 常见陷阱

| 陷阱     | 表现                               | 避免方法           |
| -------- | ---------------------------------- | ------------------ |
| 技术驱动 | "我们有了大模型，做个聊天机器人吧" | 先找痛点，再看技术 |
| 过度设计 | 第一版就要支持多轮、多语言、多模态 | 先MVP验证核心假设  |
| 忽视成本 | 不考虑每次AI请求的成本             | 成本是需求的一部分 |
| 效果高估 | 认为AI能100%准确                   | 设计兜底和纠错机制 |
| 用户排斥 | 强制AI替代人工，用户反感           | 给用户选择权       |
