## 一、企业级使用说明

### 1.1 术语表的正确使用方式

| 使用场景      | 推荐方式                           | 目的                       |
| ------------- | ---------------------------------- | -------------------------- |
| 阅读技术方案  | 先看术语，再回到上下文判断业务含义 | 避免只记定义不理解业务影响 |
| 参加需求评审  | 准备“中文解释 + 产品影响”两套说法  | 让非技术同学也能理解       |
| 写 PRD / 方案 | 只保留必要术语，并给出中文说明     | 降低跨团队沟通成本         |
| 面试准备      | 用自己的话解释，并补一个真实例子   | 避免只会背名词             |

### 1.2 学术语时最容易犯的 4 个错误

1. 只记英文缩写，不理解它解决什么问题
2. 只会定义，不会说出对产品设计的影响
3. 混淆相近概念，例如 SFT、Fine-tuning、RLHF
4. 看到热门术语就写进方案，却无法说明业务必要性

### 1.3 企业最看重的术语能力

- 能解释术语，不等于能做产品判断；企业更看重“术语背后的业务含义”
- 能区分相近概念，说明你不会在评审中混淆方向
- 能把技术术语翻译成业务语言，说明你具备跨团队协作能力

## 四、企业级协作高频术语场景

### 4.1 需求评审中高频出现的术语

| 术语          | 为什么会在需求评审中出现   | 产品经理至少要讲清什么         |
| ------------- | -------------------------- | ------------------------------ |
| Token         | 影响输入输出长度和调用成本 | 为什么这个功能会更贵、限制更多 |
| Prompt        | 影响输出质量与交互方式     | 为什么这个场景需要特定提示设计 |
| Temperature   | 影响稳定性和创造性         | 为什么某些场景要更保守         |
| RAG           | 影响知识更新与准确率       | 为什么不能只靠模型记忆         |
| Hallucination | 影响风险与兜底设计         | 哪些场景不能接受错误生成       |

### 4.2 技术评审中高频出现的术语

| 术语              | 典型讨论点                       |
| ----------------- | -------------------------------- |
| Transformer       | 架构基础，关系到上下文与生成特性 |
| Embedding         | 检索、召回、聚类和推荐的基础     |
| Fine-tuning / SFT | 是否值得为垂直任务做定制         |
| Quantization      | 是否需要压缩模型降低部署成本     |
| MoE               | 是否适合高性能、大规模场景       |

### 4.3 面试和汇报时最值得优先掌握的表达方式

1. 不只说定义，要能讲“它为什么重要”
2. 不只说技术，要能讲“它对产品意味着什么”
3. 不只说名词，要能结合真实场景举例
