# Project08 - AI会议纪要

> 这是一个适合学习 AI 会议纪要产品原型的项目实战案例。当前项目采用 `React + TypeScript + Vite + Express` 实现，支持输入会议转录文本、填写会议标题和参会人信息，并生成摘要、关键要点、行动项和决议事项的展示结果，但它还不是一个已经具备音视频上传、语音转写、多人协同编辑和真实结构化抽取闭环的企业级会议系统。

## 一、先理解这个项目在练什么

这个项目最适合训练的是以下三类能力：

1. 如何把大模型能力包装成一个明确的办公场景产品。
2. 如何把长文本输入转化为结构化输出结果。
3. 如何理解“会议纪要原型”和“企业会议助手系统”之间的真实差距。

如果你是初学者，这个项目能帮助你快速理解一个 AI 办公工具最小 MVP 的典型结构：

- 有输入源
- 有处理按钮
- 有结构化结果区

如果你是技术人员，这个项目适合你练习：

- React 表单状态管理
- 长文本输入与结果渲染
- Express 接口拆分
- mock 模式与真实模型模式切换
- 结构化输出设计

如果你是想转型 AI 产品经理，这个项目最重要的价值在于帮助你理解：

- 办公类 AI 产品为什么强调“结构化结果”
- 什么能力只是演示层
- 什么能力才决定企业是否真正愿意用

## 二、项目目标与用户场景

这个项目要模拟的问题可以概括为一句话：

> 用户把会议转录文本粘贴进系统后，系统自动整理出可阅读、可分发、可执行的会议纪要。

它适合模拟的典型场景包括：

- 项目周会后的纪要整理
- 产品评审会后的结论提炼
- 跨部门协同会议的待办拆解
- 管理层会议后的关键决策沉淀

这类场景很适合做成 AI 产品原型，因为用户天然需要：

- 快速压缩长内容
- 提取重点
- 识别行动项
- 记录决策结果

但要注意，当前项目更准确的定位是“文本纪要生成原型”，不是完整的“会议记录平台”。

## 三、当前项目能力概览

根据真实代码，这个项目当前已经实现了：

- 输入会议标题
- 输入参会人员
- 粘贴会议转录文本
- 调用后端生成纪要
- 在前端展示摘要、关键要点、行动项、决议事项
- `OPENAI_API_KEY` 存在与否两种运行模式

但它当前也有非常明确的边界：

- 没有音频上传和语音转写
- 没有会议记录历史
- 没有多人协作编辑
- 没有纪要导出能力
- 没有真实任务分发和提醒
- API 模式下目前只把模型返回内容塞进 `summary`，并没有真正解析成完整结构化字段

因此，这个项目更适合被理解为：

> 一个强调“结构化纪要展示”的办公类 AI 原型，而不是成型的会议管理系统。

## 五、技术栈与实现方式

### 1. 当前真实可运行部分

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 模型 SDK：OpenAI Node SDK
- 本地开发：前端 `5173`，后端 `3000`

### 2. 当前最值得注意的实现特征

这个项目有一个非常典型的 AI 原型特征：

- mock 模式下返回的是完整结构化结果
- API 模式下虽然接入了真实模型，但还没有把模型输出真正解析成结构化字段

这说明当前项目非常适合学习一个现实问题：

> 很多 AI 产品原型在演示时结构很完整，但一旦切到真实模型输出，就会暴露“结构化抽取和格式控制”还没真正做完。

### 3. 当前没有落地的企业级会议能力

当前项目还没有实现：

- 会议录音导入
- 自动转写
- 发言人区分
- 历史纪要管理
- 协同修改
- 导出为文档或邮件
- 行动项同步到任务系统

所以你在阅读这个项目时，重点不应该是它“功能多不多”，而是它已经把最核心的“输入 -> 提炼 -> 输出”闭环做出来了。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置如下：

```env
OPENAI_API_KEY=
PORT=3000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：进入 mock 模式，适合演示和学习
- 配置 `OPENAI_API_KEY`：进入 API 模式，调用真实模型生成会议纪要内容

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:3000/health`

项目在 `vite.config.ts` 中把 `/api` 代理到了 `http://localhost:3000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/components/MeetingMinutes.tsx`

这是整个项目最关键的前端文件。

它把会议纪要产品最核心的交互拆得非常清楚：

- 标题输入
- 参会人输入
- 转录文本输入
- 一键生成
- 分区展示结果

特别值得学习的是结果展示结构：

- 摘要
- 关键要点
- 行动项
- 决议事项

这实际上就是一个办公类 AI 产品常见的结果组织方式。

### 2. `src/services/api.ts`

这个文件负责把前端调用统一封装为 `generateMinutes()`。

它的价值在于：

- 让组件逻辑更干净
- 请求入口更统一
- 后续便于增加鉴权、超时控制、重试和日志埋点

### 3. `src/types/index.ts`

这里定义了当前纪要结果的数据结构：

- `summary`
- `keyPoints`
- `actionItems`
- `decisions`

这说明这个项目的重点不是“自由文本生成”，而是“把会议内容组织成结构化产物”。

这也是会议纪要产品和普通聊天产品的重要区别。

### 4. `server/routes/minutes.ts`

这是当前项目最值得认真看的后端文件。

它很清楚地暴露了当前原型的真实状态：

- mock 模式：返回完整结构化纪要
- API 模式：调用真实模型，但暂时只把结果填进 `summary`

这会让你非常清楚地看到一个现实问题：

AI 产品最难的往往不是“接上模型”，而是“让模型稳定地产出符合前端结构的数据”。

### 5. `server/index.ts`

这个文件展示了一个非常轻量的 AI Demo 服务端形态：

- 注册 `/api`
- 提供 `/health`
- 根据环境变量切换 mock/api 模式

它非常适合新手理解最小 AI 办公工具后端通常长什么样。

## 八、从产品和技术角度，应该如何理解这个项目

### 1. 从产品角度看

这个项目已经具备会议纪要原型最重要的三个感觉：

- 有明确输入源
- 有一键提炼动作
- 有结构化产物输出

这已经足以支撑一个有说服力的办公场景演示。

### 2. 从技术角度看

它当前主要完成的是：

- 前端表单输入与结果渲染
- 后端纪要生成接口
- mock 结构化输出
- 真实模型接入

而真正决定企业会议系统价值的关键能力还没有落地，例如：

- 音视频处理
- 发言人识别
- 长文本切片与分段总结
- 结果校对与编辑
- 纪要版本管理
- 行动项分发与提醒

### 3. 从学习角度看

这个项目特别适合作为“办公类 AI 产品设计”的练习样本：

1. 先做一个能演示的会议纪要原型。
2. 再把输出从自由文本变成结构化结果。
3. 最后再往真实会议系统能力演化。

## 九、如果你是产品经理，应该重点观察什么

### 1. AI 会议纪要的核心不是摘要，而是结构化沉淀

很多初学者会把“总结一下会议内容”理解成会议纪要产品的全部价值。

实际上真正有价值的是把内容整理成：

- 可以快速浏览的摘要
- 可以传播的关键要点
- 可以执行的行动项
- 可以留档的决议事项

当前项目正好把这个结构展现得很直接。

### 2. 真实模型接入后，结构控制会立刻变成问题

mock 模式下，所有结构都很漂亮。

但一旦换成真实模型，如果没有：

- JSON 约束
- 输出校验
- 结构解析
- 错误兜底

前端就很难稳定拿到完整结果。

这也是这个项目最值得产品经理理解的技术边界之一。

### 3. 企业会议系统真正难的是工作流闭环

企业真正使用会议纪要系统时，通常不只关心“生成内容”，还关心：

- 纪要是否准确
- 谁来确认
- 行动项派给谁
- 什么时候提醒
- 是否进入项目系统或任务系统

当前项目正好停留在“内容生成”阶段，适合你学习它向“工作流产品”演化的方向。

## 十、建议你至少做一次改造

如果你要把这个项目真正写进作品集，建议至少做一次有价值的改造。

### 改造方向 1：让 API 模式输出真正结构化

这是最优先的一步。

你可以继续补齐：

- 要求模型输出 JSON
- 服务端解析 JSON
- 加入字段校验
- 出错时回退到安全兜底结构

完成这一步后，项目的真实模式才会和界面结构真正匹配。

### 改造方向 2：支持纪要历史记录

当前生成结果只停留在页面里，没有持久化能力。

你可以增加：

- 本地存储或数据库保存
- 历史列表
- 查看详情
- 再编辑

### 改造方向 3：补齐导出能力

会议纪要产品非常需要“生成后可分发”。

你可以补充：

- 导出为 Markdown
- 导出为 Word
- 复制纪要
- 一键发送邮件

### 改造方向 4：支持上传音频或文本文件

当前只能手动粘贴转录文本。

你可以升级为：

- 上传录音文件
- 上传会议文本
- 自动解析文本源

### 改造方向 5：把行动项接到任务系统

如果想往企业级方向推进，可以继续增加：

- 行动项负责人确认
- 截止时间编辑
- 同步到任务列表
- 会后提醒

## 十一、怎么把这个项目讲成作品集

面试时不要只说“我做了一个 AI 会议纪要工具”，而要讲清楚你对办公类 AI 产品的理解。

你可以这样表达：

> 我做了一个 AI 会议纪要产品原型，前端实现了会议信息输入、转录文本输入和结构化纪要展示，后端实现了 mock/API 双模式纪要生成。我在这个项目中重点关注会议内容从长文本到结构化产物的转化过程，并识别了真实模型接入后在 JSON 约束、字段解析、结果校验和后续任务流转上的关键差距，进一步设计了从原型向企业会议系统演化的方案。

这样的表达会比“做了个摘要工具”更有深度。

## 十二、最小复盘模板

完成这个项目后，建议你至少回答下面几个问题：

1. 为什么会议纪要产品比普通摘要工具更强调结构化输出？
2. 当前 mock 模式和 API 模式的能力差距体现在哪里？
3. 如果真实模型输出格式不稳定，前端会遇到什么问题？
4. 如果只能做一次升级，你会优先补结构化解析、历史记录还是导出能力？为什么？
5. 企业会议纪要系统最关键的价值是生成内容，还是推动后续执行？

如果这些问题你都能回答清楚，这个项目就不只是“能生成一段文字”，而是真正沉淀成你的产品能力了。
