## 一、结构化输出概述

### 1.1 什么是结构化输出

**结构化输出**是指通过Prompt或模型原生能力，引导大语言模型生成具有固定格式、固定字段、可被程序直接解析的数据，而非难以稳定消费的自由文本。

```text
结构化输出的价值：

自由文本输出：
"这款产品优点很多，界面精美、体验不错，但价格略高。"
→ 人能看懂，但程序难以稳定提取字段

结构化输出（JSON）：
{
  "advantages": ["界面精美", "体验良好"],
  "disadvantages": ["价格偏高"],
  "target_users": "预算相对充足的用户",
  "recommendation": "谨慎推荐"
}
→ 程序可直接解析、校验、存储和传递
```

### 1.2 为什么AI产品必须重视结构化输出

| 场景 | 典型问题 | 结构化输出解决什么 |
|------|---------|------------------|
| **API对接** | 下游系统无法稳定消费自然语言 | 统一字段、类型和格式 |
| **前端渲染** | 页面需要按组件展示内容 | 结构化字段直接映射UI |
| **工作流编排** | 多个模型步骤衔接不稳定 | 上一步输出直接作为下一步输入 |
| **数据存储** | 结果难以沉淀到数据库 | 字段可直接入库与检索 |
| **质量治理** | 输出质量无法自动校验 | 可做格式校验、值域校验、业务校验 |

### 1.3 结构化输出不是“更好看”，而是“更可控”

对企业来说，结构化输出的意义不在于把内容排版整齐，而在于它可以支撑以下 4 件事：

1. **让AI输出进入生产系统，而不是停留在演示层**
2. **让质量评测、异常监控、降级处理有抓手**
3. **让跨团队协作有统一接口，而不是依赖口头约定**
4. **让版本升级和字段变更可管理，而不是隐性破坏下游逻辑**

## 二、常见结构化格式

### 2.1 JSON格式

JSON是最常用、通用性最强的结构化格式，适合绝大多数AI产品场景。

```json
{
  "sentiment": "mixed",
  "sentiment_score": 0.4,
  "key_issues": ["经常闪退"],
  "key_praises": ["界面好看"],
  "urgency": "high",
  "action_required": true,
  "suggested_response": "感谢反馈，我们已记录闪退问题并优先排查。"
}
```

**适用场景**：
- 接口返回
- 工作流节点输入输出
- 结构化存储
- 自动评测与打分

### 2.2 Markdown表格

表格格式适合对比、汇总、评审记录等偏人工阅读场景。

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 开发成本 | 高 | 低 | 中 |
| 上线周期 | 长 | 短 | 中 |
| 定制化程度 | 高 | 低 | 中 |

**适用场景**：
- 向业务方或管理层汇报
- 方案对比
- 人工评审记录

### 2.3 XML格式

XML适合层次深、标签语义强、历史系统兼容要求高的场景，但在现代AI产品中通常不如JSON常用。

```xml
<entities>
  <entity type="person">马云</entity>
  <entity type="company">阿里巴巴</entity>
</entities>
<relations>
  <relation type="founder" from="马云" to="阿里巴巴" />
</relations>
```

### 2.4 YAML格式

YAML适合配置说明、规则编排和层次化配置输出。

```yaml
product:
  name: 智能客服机器人
  features:
    - name: 自动回复
      enabled: true
    - name: 多轮对话
      enabled: false
```

### 2.5 企业实践中的格式选择建议

| 目标 | 推荐格式 | 原因 |
|------|---------|------|
| **系统对接** | JSON | 生态成熟、解析简单、校验方便 |
| **人工评审** | Markdown表格 | 阅读友好、便于复盘 |
| **规则配置** | YAML | 可读性好、层次清晰 |
| **兼容旧系统** | XML | 适配部分存量系统要求 |

企业环境里，默认优先选 **JSON**。除非有明确兼容或阅读诉求，否则不要让输出格式过于分散。

## 三、结构化输出设计技巧

### 3.1 明确字段定义，而不是只说“请输出JSON”

```text
较好的写法：
"请以JSON格式输出，包含以下字段：
- title: 字符串，文章标题
- summary: 字符串，100字以内摘要
- tags: 字符串数组，最多5个
- sentiment: 枚举值，只能是 positive / negative / neutral
- key_points: 字符串数组"

较差的写法：
"请以JSON输出"
→ 模型不知道字段、类型、枚举和值域要求
```

### 3.2 字段设计应覆盖 5 个关键维度

| 维度 | 要回答的问题 | 示例 |
|------|-------------|------|
| **字段名** | 字段叫什么 | `priority` |
| **字段类型** | 是字符串、数值还是数组 | `string` / `number` / `array` |
| **是否必填** | 缺失是否允许 | `required: true` |
| **值域约束** | 合法值有哪些 | `high / medium / low` |
| **空值策略** | 信息不足时如何处理 | `null` 或 `unknown` |

### 3.3 处理嵌套结构时先想消费方

嵌套结构不是越深越好，而是要考虑消费方是否方便解析、渲染和检索。

```json
{
  "basic_info": {
    "age": 25,
    "gender": "女",
    "city_tier": "一线"
  },
  "financial": {
    "monthly_income": 15000,
    "monthly_spending": 3000
  },
  "preferences": {
    "categories": ["护肤品", "化妆品"],
    "price_sensitivity": "medium"
  }
}
```

设计时建议遵守：

1. **层级不宜过深，通常控制在 2 到 3 层**
2. **列表项结构尽量统一，避免同数组中字段不一致**
3. **字段命名保持稳定，避免今天用 `summary`、明天换成 `abstract`**

### 3.4 企业级Schema设计建议

#### 3.4.1 一份可上线的输出Schema至少要明确 6 件事

| 要素 | 说明 |
|------|------|
| `required` 字段 | 哪些字段缺失就不能进入下游系统 |
| 类型约束 | 每个字段的数据类型必须明确 |
| 枚举范围 | 状态、标签、优先级等不要自由发挥 |
| 空值策略 | 信息不足时填 `null`、`unknown` 还是空数组 |
| 默认值策略 | 哪些字段允许系统补默认值 |
| 版本兼容 | 新增字段是否影响旧版本解析逻辑 |

#### 3.4.2 设计Schema时的常见原则

- 对关键业务字段使用强约束，对描述性字段保留适度弹性
- 不要让字段命名承载过多业务解释，说明应写在文档或注释中
- 对可能演进的字段预留兼容空间，例如新增可选字段而不是改字段名

## 四、结构化输出的可靠性保障

### 4.1 常见问题与修复思路

| 问题 | 常见原因 | 修复思路 |
|------|---------|---------|
| **格式不合法** | 引号、括号、逗号缺失 | 使用模型原生结构化输出能力或加强格式约束 |
| **字段缺失** | Prompt未强调必填项 | 明确列出所有必填字段 |
| **类型错误** | 模型对类型理解不一致 | 显式写出类型和合法示例 |
| **额外内容** | 模型输出了解释性文本 | 要求“只输出结构化结果，不输出说明文字” |
| **业务值错误** | 字段语法正确但业务含义错 | 增加业务规则校验与样例 |

### 4.2 输出校验链路

```text
模型输出
  ↓
格式校验
  - 是否可解析
  - 括号/引号/分隔符是否正确
  ↓
字段校验
  - 必填字段是否齐全
  - 字段类型是否正确
  ↓
值域校验
  - 枚举值是否在范围内
  - 数值区间是否合理
  ↓
业务校验
  - 字段之间是否矛盾
  - 是否符合业务规则
  ↓
使用 / 重试 / 降级 / 转人工
```

### 4.3 容错设计

```text
建议在Prompt中明确以下规则：

1. 信息不足时，对应字段填 null，不要编造
2. 无法判断时，增加 reason 字段说明原因
3. 对高风险判断增加 confidence 字段
4. 必须只输出结构化内容，不附带解释性文本
5. 如果无法满足格式要求，返回统一错误对象
```

```json
{
  "result": null,
  "confidence": 0.1,
  "reason": "输入信息不足，无法判断"
}
```

### 4.4 企业级降级策略

| 场景 | 优先处理方式 | 说明 |
|------|-------------|------|
| **格式错误** | 自动重试一次 | 使用更强格式约束再次生成 |
| **字段缺失** | 规则补齐或返回错误对象 | 视字段关键程度决定 |
| **高风险字段异常** | 拒绝进入自动链路 | 如医疗、金融、法律建议 |
| **连续失败** | 降级到半结构化文本或转人工 | 保证系统可用性 |

## 五、AI产品经理的结构化输出关注点

### 5.1 产品中的结构化输出架构

```text
用户输入
  ↓
Prompt模板 / Schema定义
  ↓
模型调用
  ↓
输出解析层
  - 格式解析
  - 字段校验
  - 默认值填充
  ↓
业务逻辑层
  ↓
前端展示 / 数据库存储 / 工作流流转
```

### 5.2 AI产品经理要做的关键决策

1. **选择哪种格式最适合业务链路**
2. **定义哪些字段必须进入系统，哪些可选**
3. **明确失败后的重试、降级和转人工策略**
4. **确定如何评估输出成功率、解析成功率和业务可用率**

### 5.3 与开发团队协作的重点

| 协作内容 | AI产品经理负责什么 | 开发工程师负责什么 |
|---------|-------------------|-------------------|
| **接口设计** | 定义字段、含义、约束与版本策略 | 实现解析、映射与兼容逻辑 |
| **质量门禁** | 定义成功标准与验收阈值 | 建立校验、告警与失败处理 |
| **异常治理** | 定义重试、降级、转人工策略 | 实现异常链路与日志追踪 |
| **版本演进** | 评估字段变更的业务影响 | 实现新旧版本兼容 |

## 六、企业级输出治理框架

### 6.1 企业级结构化输出治理的 4 层框架

| 层级 | 核心问题 | 典型动作 |
|------|---------|---------|
| **定义层** | 输出什么、字段怎么定 | Schema、字段字典、样例 |
| **校验层** | 输出是否合格 | 格式校验、字段校验、业务校验 |
| **运行层** | 失败了怎么办 | 重试、降级、告警、转人工 |
| **演进层** | 版本怎么升级 | 兼容策略、灰度发布、变更公告 |

### 6.2 企业级指标体系建议

| 指标 | 定义 | 作用 |
|------|------|------|
| **解析成功率** | 输出可被程序成功解析的比例 | 衡量格式稳定性 |
| **字段完整率** | 必填字段齐全的比例 | 衡量结构完整性 |
| **业务可用率** | 结果可直接进入业务流程的比例 | 衡量真实可用性 |
| **重试率** | 需要再次生成的比例 | 衡量Prompt与Schema稳定性 |
| **降级率** | 进入降级链路的比例 | 衡量系统韧性与风险水平 |

### 6.3 高风险场景治理建议

在金融、医疗、法律、客服赔付等高风险场景中，结构化输出不仅要“格式对”，还要“责任清晰、边界明确”。

建议增加以下治理动作：

1. **高风险字段双重校验**
2. **敏感字段来源标注**
3. **关键结果人工复核**
4. **Prompt与Schema变更审批**

## 八、转行者实践建议

### 8.1 传统产品接口思维与AI接口思维的差异

| 维度 | 传统产品思维 | AI产品思维 | 转型要点 |
|------|-------------|-----------|---------|
| 输出类型 | 固定参数与固定接口 | 模型生成的概率性结果 | 需要额外做结构约束 |
| 校验重点 | 参数合法性 | 格式、字段、业务三层都要校验 | 不能只看是否能解析 |
| 异常处理 | 错误码即可 | 还要考虑重试、降级、转人工 | AI异常链路更复杂 |
| 版本演进 | API版本升级 | Prompt + Schema双版本治理 | 变更影响更隐蔽 |

### 8.2 实践练习

```markdown
## 练习1：输出Schema设计（30分钟）

为以下场景设计JSON Schema：
1. 用户反馈分析
2. 商品信息抽取
3. 会议纪要生成

要求：明确字段名、类型、必填项、枚举值、空值策略。

## 练习2：可靠性评测（30分钟）

使用同一Prompt运行5次，记录：
1. 解析成功率
2. 必填字段完整率
3. 枚举值错误率
4. 是否需要降级处理

## 练习3：企业级接口规范（1小时）

为一个AI产品撰写完整输出规范：
1. Schema定义
2. 校验规则
3. 重试与降级策略
4. 版本兼容说明
```

### 8.3 常见误区

| 误区 | 表现 | 正确做法 |
|------|------|---------|
| 格式定义过于模糊 | “请输出JSON” | 明确字段、类型、枚举和值域 |
| 只做解析不做业务校验 | JSON可解析就直接使用 | 增加字段和业务规则校验 |
| 忽视降级链路 | 失败直接报错 | 设计重试、兜底和转人工 |
| 忽视版本管理 | 字段随意修改 | 像管理API一样管理输出协议 |

## 十、企业级验收标准

### 10.1 学完本章后至少应做到

- [ ] 能为一个真实业务场景定义完整的结构化输出Schema
