# Agent产品设计

> Agent 是近几年 AI 产品里最容易让人兴奋、也最容易让团队走偏的方向之一。很多需求只要稍微复杂一点，就会有人说“这应该做成 Agent”。但现实是，真正适合 Agent 的场景并不那么多。很多项目最后失败，不是因为 Agent 技术不行，而是因为产品一开始就没有想清楚：用户到底是需要一个会持续执行任务的智能体，还是只需要一个能给建议的 Copilot，或者一个可控的工作流系统。真正成熟的 Agent 产品设计，不是追求“更像人、更自主”，而是判断在什么场景下，自主规划、工具调用、记忆和持续执行真的比更简单的方案更有价值。对转型中的 AI 产品经理来说，学会画清 Agent 的边界，是理解 AI 产品高级形态的关键一步。

## 一、为什么很多团队会把 Agent 做偏

### 1. 因为“自主执行”听起来很有吸引力

相比普通问答或 Copilot，Agent 最吸引人的地方在于：

- 能自己拆任务
- 能自己调用工具
- 能自己连续执行
- 看起来更像“数字员工”

这会让很多团队天然倾向于高估它的适用范围。

### 2. 因为很多需求表面复杂，实际上不需要 Agent

例如一些需求看起来包含多个步骤，但本质上可能只是：

- 固定流程
- 明确分支
- 已知规则
- 稳定输入输出

这类场景更适合：

- 工作流
- n8n / Dify / LangGraph
- 规则引擎
- Copilot + 表单

而不一定非要上 Agent。

### 3. 因为“能做出来”和“适合做产品”不是一回事

技术上可以让 Agent 跑起来，不代表：

- 用户真的信任它
- 业务真的需要它持续执行
- 企业真的能接受它的风险

所以 Agent 产品设计的第一课，不是想怎么把它做强，而是先想清楚值不值得做。

## 二、Agent 到底是什么

### 1. 一句话定义

如果用一句话定义：

**Agent 是一种围绕目标而非单次指令展开工作的 AI 产品形态，它能在一定边界内进行规划、调用工具、维护状态，并持续推进任务直到达到阶段性结果。**

关键词有五个：

- 目标
- 规划
- 工具
- 状态
- 持续推进

### 2. Agent 和聊天助手的区别

聊天助手通常是：

- 你问一句，它答一句
- 每轮交互相对独立
- 核心是语言生成

Agent 更强调：

- 以目标为中心
- 会拆解步骤
- 会决定下一步动作
- 可能多轮调用外部能力

### 3. Agent 和 Copilot 的区别

Copilot 更偏：

- 建议
- 草稿
- 辅助
- 用户主导

Agent 更偏：

- 连续执行
- 任务推进
- 状态保持
- 系统主导更多

### 4. Agent 和工作流的区别

工作流通常是：

- 预先定义路径
- 节点和分支大多已知
- 可预测性高

Agent 则通常面对：

- 任务路径不完全已知
- 需要动态决策
- 工具选择和顺序可能变化

所以真正的关键不是“步骤多不多”，而是“路径是否需要动态判断”。

## 三、什么时候真的该做 Agent

### 1. 当任务目标明确，但路径不完全固定时

这是 Agent 最典型的适用条件。

例如：

- 帮用户做一轮市场研究
- 自动排查某类线上故障
- 根据目标收集资料并生成分析草稿
- 跨多个系统完成一段复杂业务准备

### 2. 当任务需要多种工具组合使用时

如果任务需要：

- 检索知识
- 查数据库
- 发起工单
- 读取文件
- 生成内容

且这些动作顺序依赖中间结果，那么 Agent 的价值会更明显。

### 3. 当系统需要保持连续状态和阶段结果时

例如：

- 当前完成到哪一步
- 已经查过哪些信息
- 哪些动作失败过
- 下次恢复时从哪继续

这类问题越明显，越说明 Agent 可能有意义。

### 4. 当用户更想“委托任务”而不是“逐步操作”时

如果用户表达更接近：

- “帮我准备一份调研草稿”
- “帮我完成售前方案初稿”
- “帮我排查这个问题并给出建议”

而不是：

- “帮我写一句话”
- “帮我解释一个指标”

那就更接近 Agent 场景。

---

## 四、什么时候不该做 Agent

### 1. 当流程基本固定时

如果任务已经很明确地可以写成：

- 第一步
- 第二步
- 第三步

那么很多时候更适合工作流，而不是 Agent。

### 2. 当风险过高但治理能力不足时

如果场景涉及：

- 财务处理
- 审批动作
- 对外承诺
- 批量写操作

且团队还没有完善的审批、审计、回滚和权限体系，那就不适合直接做高自主 Agent。

### 3. 当用户其实更需要控制感而不是委托感时

例如：

- 写邮件
- 做演示文稿
- 写方案
- 回复客户

这类场景很多时候更适合 Copilot，因为用户希望自己掌控内容。

### 4. 当“复杂”只是需求描述复杂，而不是执行路径复杂时

很多需求其实只是业务概念复杂，但执行方式固定。

这类场景并不需要 Agent。

## 五、Agent 产品设计的核心不是“聪明”，而是“边界”

### 1. Agent 最大风险不是答错，而是做错

普通问答产品答错，通常影响是一次错误信息。

Agent 做错，可能意味着：

- 调错工具
- 改错数据
- 走错流程
- 通知错人
- 在错误前提下继续执行更多动作

### 2. 所以 Agent 设计第一步要定义边界

至少要定义：

- 它能处理哪些目标
- 它不能做哪些动作
- 哪些动作必须人确认
- 哪些工具只能只读
- 哪些场景必须立即停止

### 3. 一个好 Agent 不是“什么都敢做”

而是：

- 知道自己该做什么
- 知道自己不该做什么
- 知道何时请求人工介入

### 4. 企业里的 Agent 尤其需要保守设计

企业场景比个人实验更强调：

- 可控
- 可追责
- 可回滚
- 可审计

这会让“保守而稳定”的 Agent 比“大胆而酷炫”的 Agent 更有产品价值。

## 六、Agent 的目标设计怎么做

### 1. 目标要足够明确

用户给 Agent 的目标不能过度模糊。

例如：

- “帮我提升业务”太宽泛
- “帮我分析最近 30 天客户投诉，并输出一份问题归因草稿”就更可执行

### 2. 好目标通常包含三层信息

建议明确：

- 要达成什么结果
- 允许使用哪些资源
- 结果要以什么形式输出

### 3. 为什么目标设计这么重要

因为 Agent 的后续规划、工具选择和执行顺序，都依赖目标定义。

目标模糊时，Agent 往往会：

- 跑偏
- 过度探索
- 产出不符合用户预期

### 4. 产品上如何帮助用户表达目标

可以通过：

- 任务模板
- 目标输入结构化引导
- 范例目标
- 约束项选择

来降低用户提目标的难度。

---

## 七、规划、工具和记忆是 Agent 产品设计的三大核心

### 1. 规划设计

规划要回答：

- 当前任务分成哪些步骤
- 下一步为什么这样做
- 失败后是否重试或改道

产品上可考虑展示：

- 当前阶段
- 已完成步骤
- 下一步动作

### 2. 工具设计

Agent 的工具不应只是“越多越好”。

更重要的是：

- 工具职责清晰
- 只读和写操作区分明确
- 输入输出稳定
- 高风险工具有额外限制

### 3. 记忆设计

记忆可以帮助 Agent：

- 记住用户偏好
- 记住任务上下文
- 记住之前失败经验

但产品上要问清：

- 记什么
- 记多久
- 谁能看到
- 如何纠错和删除

### 4. 三者的关系

你可以把它们理解为：

- 目标决定方向
- 规划决定路径
- 工具决定动作
- 记忆决定连续性

缺其中任何一项，Agent 都很难稳定工作。

---

## 八、人机协作是 Agent 设计里不能省的一层

### 1. Agent 不是越少人介入越高级

很多团队会误以为：

- 人工确认越少，Agent 越先进

实际上，在很多企业场景里：

- 合适的人工介入
- 关键节点确认
- 异常时人工接管

才是产品成熟的表现。

### 2. 常见人机协作节点

通常包括：

- 目标确认
- 关键步骤确认
- 高风险动作确认
- 阶段结果复核
- 异常中断后的人工选择

### 3. 为什么这很重要

因为 Agent 场景往往牵涉连续动作。

如果前面一步方向错了，后面所有动作都可能在放大错误。

### 4. 人机协作体验怎么设计

更合理的设计通常是：

- 在关键节点给出简短可理解说明
- 告诉用户将要做什么
- 支持同意、修改、停止三种选择

---

## 九、Agent 产品的失败恢复和可观测性怎么设计

### 1. Agent 一定会失败，所以必须设计恢复机制

常见失败包括：

- 工具超时
- 工具返回异常
- 任务路径跑偏
- 中间结果不可靠
- 权限不足

### 2. 产品不能只显示“任务失败”

更成熟的做法是告诉用户：

- 失败发生在哪一步
- 当前已经完成了什么
- 是否可以重试
- 是否需要人工接手

### 3. Agent 的可观测性为什么重要

企业里尤其需要能追踪：

- 调了哪些工具
- 用了哪些输入
- 哪一步失败
- 为什么进入人工审核

### 4. 这不仅是运维问题，也是产品可信度问题

如果用户完全看不懂 Agent 在做什么，就很难真正信任它。

---

## 十、企业里的 Agent 为什么必须先做风险分级

### 1. 不同 Agent 任务的风险完全不同

例如：

- 资料整理 Agent 风险较低
- 工单分配 Agent 风险中等
- 自动执行审批动作的 Agent 风险很高

### 2. 风险分级会直接影响产品设计

例如：

- 低风险任务可以少确认、重效率
- 中风险任务要加阶段复核
- 高风险任务必须审批、审计、回滚

### 3. 产品经理至少要定义什么

至少包括：

- 风险等级
- 对应确认策略
- 对应工具权限
- 对应日志要求
- 对应上线范围

### 4. 没有风险分级，Agent 很容易“功能演示成功，生产落地失败”

这是企业 Agent 项目里非常常见的问题。

## 十一、一个完整案例：售前方案助手为什么可能适合做 Agent

### 1. 场景背景

某企业希望做一个售前方案助手，帮助方案经理完成：

- 收集客户背景
- 汇总历史案例
- 检索产品能力
- 生成初版方案结构

### 2. 为什么这个场景可能适合 Agent

因为它具备几个特征：

- 目标明确
- 路径不完全固定
- 需要多工具、多知识源协同
- 结果是阶段性产出，而不是单句回答

### 3. 这个场景的正确边界是什么

更合理的 Agent 设计通常是：

- 帮收集和整理信息
- 帮输出初版方案骨架
- 帮识别缺失信息

但不应直接：

- 对外发送正式方案
- 自动做商业承诺
- 绕过人工审核提交客户

### 4. 这说明什么

说明 Agent 更适合承担：

- 复杂准备工作
- 多源整合工作
- 草稿和建议型输出

而不是一开始就承担高责任结果。

## 十二、AI 产品经理如何评审一个 Agent 方案

### 1. 不要先看“会不会规划”，先看“是否真的需要 Agent”

首先要问：

- 这是动态路径问题，还是固定流程问题
- 用户需要委托任务，还是只是需要辅助
- 更简单的方案是否已足够

### 2. 再看目标、工具和边界

重点检查：

- 目标是否清晰
- 工具是否必要且受控
- 高风险动作是否有限制
- 失败是否可恢复

### 3. 再看状态和人机协作

要确认：

- 是否有任务状态表示
- 是否有关键节点确认
- 是否有人工接管机制
- 是否能回放执行链路

### 4. 最后看企业落地条件

至少要确认：

- 权限和审计是否可用
- 风险分级是否完成
- 是否有灰度发布方案
- 是否有回滚与停机策略

## 十四、常见误区补充

### 误区 1：任务复杂就一定应该做 Agent

错误。很多复杂任务其实只是固定流程复杂，更适合工作流。

### 误区 2：Agent 越自主越高级

错误。企业场景里，过度自主往往意味着更高风险和更差可控性。

### 误区 3：Agent 一定比 Copilot 更值钱

错误。很多高价值场景长期停留在 Copilot 形态，反而更稳、更省、更容易规模化。

### 误区 4：只要工具够多，Agent 就会更强

错误。工具越多，选择、权限和失败恢复复杂度也会迅速上升。

### 误区 5：Agent 的问题主要靠模型升级解决

错误。很多问题其实来自目标不清、边界不明、工具设计差和治理缺失。

## 十五、本章小结

如果用一句话总结：

**Agent 产品设计的本质，不是让 AI 看起来更像人，而是在真正需要动态规划和持续执行的场景中，以可控边界承接更复杂的任务委托。**

对 AI 产品经理来说，本章最重要的收获包括：

- 学会判断什么场景真的需要 Agent
- 学会围绕目标、规划、工具、记忆和状态做设计
- 学会在人机协作和企业治理中为 Agent 设定清晰边界
- 学会避免把工作流、Copilot 或聊天助手误做成 Agent
