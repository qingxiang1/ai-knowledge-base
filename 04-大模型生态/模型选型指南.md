- 建立从需求澄清、候选筛选、POC 验证到上线门禁的完整选型流程

## 一、选型决策框架

### 1.1 五维评估模型

```
AI产品经理模型选型五维评估框架：

┌─────────────────────────────────────────┐
│           模型选型五维雷达               │
│                                          │
│     成本效益 ◆                           │
│        ▲                                 │
│       /|\                                │
│      / | \                               │
│     /  |  \                              │
│ 技术能力 ◄───┼───► 生态成熟度            │
│     \  |  /                              │
│      \ | /                               │
│       \|/                                │
│        ▼                                 │
│     安全合规 ◆                           │
│        ▲                                 │
│        │                                 │
│     业务适配                             │
└─────────────────────────────────────────┘

五维度说明：
1. 技术能力：模型性能、多模态、上下文长度
2. 成本效益：API价格、部署成本、ROI
3. 生态成熟度：工具链、社区、文档、集成
4. 安全合规：数据隐私、行业监管、地理位置
5. 业务适配：语言、场景、延迟、定制化

权重分配建议：
- 初创公司/MVP：成本效益(40%) > 技术能力(25%) > 生态(15%) > 业务适配(15%) > 安全合规(5%)
- 中型企业：技术能力(30%) > 成本效益(25%) > 安全合规(20%) > 业务适配(15%) > 生态(10%)
- 大型企业/金融医疗：安全合规(35%) > 技术能力(25%) > 业务适配(20%) > 成本效益(15%) > 生态(5%)
```

### 1.2 选型决策流程

```
模型选型标准流程：

阶段1：需求澄清（1-2周）
├── 明确业务场景和核心诉求
├── 定义成功指标（准确率、延迟、成本）
├── 评估数据敏感性
├── 确定预算范围
└── 输出：《模型需求规格书》

阶段2：候选筛选（3-5天）
├── 基于需求初筛3-5个候选模型
├── 收集各模型技术规格
├── 初步成本估算
└── 输出：《候选模型清单》

阶段3：POC验证（2-4周）
├── 设计评测数据集（业务真实数据）
├── 多模型并行测试
├── 量化评估 + 人工评估
├── 压力测试和稳定性测试
└── 输出：《POC评测报告》

阶段4：决策评审（1周）
├── 综合评分排序
├── 风险评估
├── 成本效益分析
├── 技术团队评审
└── 输出：《模型选型决策书》

阶段5：落地实施（2-4周）
├── 接入开发
├── 监控体系搭建
├── A/B测试方案
├── 回退机制准备
└── 输出：《上线检查清单》
```

```
  □ 是否需要微调/定制？
```

## 三、POC评测方法

### 3.1 评测数据集设计

```
POC评测数据集构建原则：

数据集构成：
├── 业务真实数据（60%）
│   ├── 历史用户查询样本
│   ├── 典型业务场景案例
│   └── 边缘案例和bad case
├── 行业基准数据（30%）
│   ├── 公开评测集（如C-Eval、CMMLU）
│   ├── 领域专用测试集
│   └── 对抗性测试样本
└── 压力测试数据（10%）
    ├── 超长文本
    ├── 复杂多轮对话
    └── 高并发模拟

数据标注要求：
- 每个样本需有标准答案或评分标准
- 覆盖不同难度等级
- 包含正负样本
- 匿名化处理

示例（智能客服场景）：
| 类别 | 样本数 | 示例 |
|------|--------|------|
| 产品咨询 | 50 | "你们支持哪些支付方式？" |
| 售后问题 | 50 | "订单怎么退款？" |
| 投诉建议 | 30 | "你们服务太差了" |
| 闲聊干扰 | 20 | "今天天气怎么样" |
| 多轮对话 | 30 | 3-5轮上下文依赖 |
| 长文本理解 | 20 | 1000字+产品说明 |
```

### 3.2 评测指标体系

```
模型评测指标体系：

客观指标（可量化）：
┌─────────────────────────────────────────┐
│  准确性指标                              │
│  ├── 准确率（Accuracy）                  │
│  ├── F1 Score（分类任务）                │
│  ├── BLEU/ROUGE（生成任务）              │
│  └── 人工评分一致性                      │
├─────────────────────────────────────────┤
│  性能指标                                │
│  ├── 首token延迟（TTFT）                 │
│  ├── 吞吐率（tokens/秒）                 │
│  ├── 并发处理能力                        │
│  └── 稳定性（错误率）                    │
├─────────────────────────────────────────┤
│  成本指标                                │
│  ├── 单条请求成本                        │
│  ├── 月度预估成本                        │
│  └── 成本/性能比                         │
└─────────────────────────────────────────┘

主观指标（人工评估）：
├── 有用性（Helpfulness）
│   └── 回答是否解决了用户问题
├── 准确性（Accuracy）
│   └── 事实是否正确，逻辑是否严谨
├── 流畅性（Fluency）
│   └── 语言是否自然，表达是否清晰
├── 安全性（Safety）
│   └── 是否有害内容，是否拒绝不当请求
└── 风格适配（Style）
    └── 是否符合品牌调性

评分标准（1-5分）：
5分：完美，无需修改可直接使用
4分：良好，少量修改即可
3分：可用，需要一定修改
2分：较差，需要大量修改
1分：不可用，完全错误
```

## 四、常见选型误区

### 4.1 十大选型误区

```
AI产品经理常见选型误区：

误区1：盲目追求最强模型
✗ "必须用GPT-4o，因为是最强的"
✓ 评估：最强≠最适合，成本可能高10倍
✓ 做法：按场景选择，简单任务用小模型

误区2：只看基准测试分数
✗ "MMLU分数高就选它"
✓ 评估：基准测试≠业务表现
✓ 做法：用真实业务数据做POC

误区3：忽视长期成本
✗ "API调用一次才几分钱，很便宜"
✓ 评估：日调用百万次，月成本可达数十万
✓ 做法：做详细的成本模型和增长预测

误区4：忽略数据隐私
✗ "先上线再说，数据安全后面考虑"
✓ 评估：数据泄露可能致命
✓ 做法：选型第一步评估数据敏感性

误区5：单模型依赖
✗ "整个产品只用一个模型"
✓ 评估：单点故障，模型更新可能带来风险
✓ 做法：关键路径设计多模型备份

误区6：忽视延迟要求
✗ "模型效果好就行，延迟不重要"
✓ 评估：用户可接受的延迟通常<2秒
✓ 做法：根据场景定义延迟SLA

误区7：不考虑团队能力
✗ "选开源模型自己部署，省钱"
✓ 评估：运维大模型需要专业团队
✓ 做法：评估团队MLops能力

误区8：忽视供应商风险
✗ "OpenAI不会出问题"
✓ 评估：服务中断、政策变化、涨价都可能
✓ 做法：设计供应商切换方案

误区9：评测数据不客观
✗ "拿模型擅长的题目测试"
✓ 评估：这样得出的是虚假结论
✓ 做法：随机抽样，包含bad case

误区10：不做A/B测试就全量
✗ "POC效果好，直接全量上线"
✓ 评估：真实用户行为与测试不同
✓ 做法：灰度发布，持续监控
```

```
□ 灰度策略
  □ 定义灰度比例（1%→5%→20%→50%→100%）
  □ 定义升级/回退标准
  □ 准备回退方案

□ 文档沉淀
  □ 技术接入文档
  □ 运维手册
  □ 问题排查指南
  □ 模型更新流程
```

## 六、总结

```
模型选型核心原则：

1. 业务驱动：从需求出发，不为技术而技术
2. 数据说话：POC验证优于纸上谈兵
3. 成本意识：全生命周期成本，非仅API价格
4. 风险分散：避免单供应商依赖
5. 持续迭代：模型快速演进，保持灵活性

推荐工具和资源：
├── 模型对比：LMSYS Chatbot Arena
├── 价格查询：各厂商官网定价页
├── 性能测试：自建评测集 + 公开基准
├── 社区参考：HuggingFace、魔搭社区
└── 行业报告：Stanford HAI、机器之心

最后建议：
- 没有银弹，组合使用多个模型
- 保持对模型生态的持续关注
- 建立内部模型评测能力
- 与业务目标对齐，定期复盘
```

## 七、转行者实践建议

### 7.1 从"拍脑袋选型"到"数据驱动选型"

| 维度     | 传统产品思维   | AI产品思维              | 转型要点                    |
| -------- | -------------- | ----------------------- | --------------------------- |
| 选型依据 | 经验/口碑/关系 | POC实测+数据对比        | 必须用数据说话              |
| 评估方法 | 功能演示       | 五维评估+POC评测        | 系统化评估框架              |
| 成本意识 | 采购成本       | 全生命周期Token成本     | Token成本是变量，需精细管控 |
| 风险管理 | 合同约束       | 多模型备份+降级方案     | AI模型更新可能影响输出      |
| 决策流程 | 领导拍板       | 需求→筛选→POC→评审→落地 | 标准化选型流程              |

### 7.2 实践练习

```markdown

选择一个真实业务场景，走完完整选型流程：

2. 筛选3-5个候选模型
3. 设计POC评测数据集（至少50条）
4. 执行评测并记录结果
5. 撰写《POC评测报告》
6. 输出《模型选型决策书》

## 练习2：成本模型设计（1小时）

为一个AI产品设计详细的成本模型：

1. 估算日/月/年调用量
2. 计算不同模型方案的成本
3. 设计成本优化策略（模型分级/缓存等）
4. 建立成本监控方案

## 练习3：选型评审模拟（1小时）

模拟一次模型选型评审会：

1. 准备选型方案和POC数据
2. 列出优势和风险
3. 准备评审材料（PPT/文档）
4. 模拟回答技术团队和业务方的质疑
```

## 十、常见误区补充

| 误区                 | 问题                       | 正确做法                         |
| -------------------- | -------------------------- | -------------------------------- |
| 选型只看技术团队意见 | 容易忽略业务价值和用户体验 | 让产品、技术、法务、采购共同参与 |
| POC 样本过少         | 结论不稳定，无法上线       | 覆盖核心、边界、异常和高风险样本 |
| 只选主模型不选备份   | 生产风险过高               | 同时定义备份模型与切换条件       |
| 上线门禁模糊         | 发布决策随意               | 明确定义灰度、全量、回滚指标阈值 |
| 选型结论不沉淀       | 下次项目重复踩坑           | 建立内部模型选型知识库和模板     |
