## 一、FewShot概述

### 1.1 什么是FewShot

**FewShot（少样本提示）** 是指在Prompt中提供少量输入-输出示例（通常为1-5个），让模型从示例中学习任务模式，从而对新的输入生成正确的输出。

```
FewShot的核心思想：

模型不知道怎么做？ → 给它看几个例子！

Prompt结构：
示例1：输入A → 输出A
示例2：输入B → 输出B
示例3：输入C → 输出C

待处理：输入X → 模型输出X

模型从示例中抽象出模式，应用到新输入
```

### 1.2 FewShot vs ZeroShot

| 对比维度 | ZeroShot（零样本） | FewShot（少样本） |
|---------|-------------------|------------------|
| **示例数量** | 0个 | 1-5个（通常） |
| **适用场景** | 模型已知的通用任务 | 特定格式、复杂模式、新任务 |
| **输出稳定性** | 较低，格式可能不统一 | 较高，遵循示例格式 |
| **Prompt长度** | 短 | 较长 |
| **成本** | 低（Token少） | 较高（Token多） |

```
ZeroShot vs FewShot 示例：

任务：情感分析

ZeroShot：
"判断以下评论的情感倾向（正面/负面/中性）：
这款手机的续航太棒了！"
→ 模型可能输出："正面" 或 "这句话表达了正面的情感"
→ 格式不稳定

FewShot：
"请判断评论的情感倾向：

评论：这家餐厅的服务太差了 → 负面
评论：产品质量一般，无功无过 → 中性
评论：物流很快，包装也很精美 → 正面

评论：这款手机的续航太棒了！ →"
→ 模型输出："正面"
→ 格式稳定，与示例一致
```

## 二、FewShot的核心要素

### 2.1 示例的组成

一个完整的FewShot示例包含：

```
示例结构：

┌─────────────────────────────────────┐
│  输入（Input）                       │
│  "评论：这家餐厅的服务太差了"        │
├─────────────────────────────────────┤
│  输出（Output）                      │
│  "负面"                              │
├─────────────────────────────────────┤
│  推理过程（可选）                    │
│  "理由：'太差了'是负面词汇"          │
└─────────────────────────────────────┘
```

### 2.2 示例数量选择

| 示例数量 | 适用场景 | 优缺点 |
|---------|---------|--------|
| **1-shot** | 简单任务，格式引导 | Token消耗少，但模式学习可能不充分 |
| **3-shot** | 大多数任务的标准选择 | 平衡了学习效果和Token成本 |
| **5-shot** | 复杂任务，需要多样本 | 学习更充分，但Token消耗大 |
| **>5-shot** | 极少使用 | Token成本过高，收益递减 |

```
示例数量与效果关系：

效果
  │
  │        ╭────── 边际效益递减
  │     ╭──╯
  │  ╭──╯
  │╭─╯
  │╱
  └────────────────────────→ 示例数量
     1    3    5    7

最佳平衡点：通常3个示例
```

### 2.3 示例质量要求

```
高质量示例的特征：

1. 代表性
   └── 示例应覆盖任务的主要情况
   └── 避免所有示例都是同一类

2. 一致性
   └── 输入和输出格式统一
   └── 标注标准一致

3. 清晰性
   └── 输入明确无歧义
   └── 输出正确无争议

4. 多样性
   └── 示例之间应有差异
   └── 展示任务的不同方面

错误示例：
示例1："很好" → 正面
示例2："不错" → 正面
示例3："棒极了" → 正面
└── 问题：所有示例都是正面，模型无法学习区分

正确示例：
示例1："很好" → 正面
示例2："太差了" → 负面
示例3："一般般" → 中性
└── 覆盖了所有类别
```

## 三、FewShot设计模式

### 3.1 基础FewShot模式

```
基础模式：

任务说明 + 示例 + 待处理输入

Prompt：
"请将以下中文翻译成英文：

中文：你好 → 英文：Hello
中文：谢谢 → 英文：Thank you
中文：再见 → 英文：Goodbye

中文：很高兴认识你 → 英文："

输出：Nice to meet you
```

### 3.2 带推理的FewShot（FewShot-CoT）

```
FewShot + Chain of Thought：

Prompt：
"解决以下数学问题，并展示推理过程：

问题：小明有5个苹果，给了小红2个，还剩几个？
推理：小明原有5个，给出去2个，5-2=3，所以还剩3个。
答案：3

问题：一个长方形长8厘米，宽5厘米，面积是多少？
推理：长方形面积=长×宽，8×5=40，所以面积是40平方厘米。
答案：40

问题：火车时速120公里，行驶3小时，走了多远？
推理："

输出：
距离=速度×时间，120×3=360，所以走了360公里。
答案：360
```

### 3.3 对比式FewShot

```
对比式示例：

Prompt：
"请将口语化表达转换为正式商务用语：

口语：'这个东西挺好的' → 正式：'该产品品质优良'
口语：'价格有点贵' → 正式：'定价略高于市场平均水平'
口语：'发货很快' → 正式：'物流配送效率较高'

口语：'客服态度不错' → 正式："

输出：'客户服务团队专业素养良好'
```

### 3.4 多轮对话式FewShot

```
多轮对话示例：

Prompt：
"你是一位专业的客服助手，以下是优秀的客服对话示例：

用户：我想退货
客服：您好，很抱歉给您带来不好的体验。请问方便告知退货原因吗？
      我们会尽力帮您解决问题。如果确实需要退货，
      请提供订单号，我为您办理。

用户：这个怎么用？
客服：您好！我可以为您详细介绍使用方法。
      请问您想了解哪个功能呢？

用户：你们的价格比别家贵
客服："

输出：
'您好，感谢您关注我们的产品。我们的定价包含了优质的售后服务
和品质保障。如果您发现同款产品有更低价格，我们可以为您申请
价格保护。您看这样可以吗？'
```

## 四、FewShot进阶技巧

### 4.1 示例排序策略

```
示例排序的影响：

策略1：简单 → 复杂
示例1（简单）："好" → 正面
示例2（中等）："质量还可以" → 正面
示例3（复杂）："虽然价格偏高，但物有所值" → 正面
→ 帮助模型逐步理解复杂度

策略2：多样 → 集中
示例1：正面
示例2：负面
示例3：中性
→ 展示任务的完整范围

策略3：最近似优先
将最相似于待处理输入的示例放在最后
→ 模型更容易联想到相关模式

注意事项：
- 避免将最难的示例放在最后（模型可能模仿错误）
- 保持示例之间的逻辑顺序
```

### 4.2 负样本示例

```
使用负样本（错误示例）进行引导：

Prompt：
"请提取以下新闻中的关键人物，只输出人名：

新闻：马云创立了阿里巴巴公司。
错误输出：马云创立了阿里巴巴公司。（错误：包含了多余信息）
正确输出：马云

新闻：雷军在发布会上介绍了小米14。
错误输出：雷军，小米14（错误：包含了产品名）
正确输出：雷军

新闻：马斯克和扎克伯格约架的事情引起了热议。
输出："

→ 通过展示错误和正确，让模型更清楚边界
```

### 4.3 动态FewShot

```
动态FewShot（RAG式）：

场景：根据用户问题，从示例库中检索最相似的示例

流程：
用户问题 ──→ 向量检索 ──→ 获取Top-K相似示例 ──→ 拼接Prompt

示例库：
- "怎么退款？" → 退款流程...
- "什么时候发货？" → 发货政策...
- "质量有问题" → 售后处理...

用户："东西不想要了怎么退？"
检索结果："怎么退款？"（最相似）

拼接的Prompt：
"用户问题：怎么退款？
回答：退款流程...

用户问题：东西不想要了怎么退？
回答："
```

## 五、FewShot应用场景

### 5.1 常见应用场景

| 场景 | FewShot作用 | 示例 |
|------|------------|------|
| **格式统一** | 确保输出格式一致 | JSON、表格、特定标记格式 |
| **风格迁移** | 学习特定写作风格 | 商务、幽默、学术风格 |
| **分类任务** | 学习分类标准 | 情感分析、意图识别、优先级划分 |
| **信息抽取** | 学习抽取规则 | 实体识别、关系抽取、关键词提取 |
| **代码生成** | 学习代码模式 | 特定框架、编码规范 |
| **评估打分** | 学习评分标准 | 内容质量评分、创意评分 |

### 5.2 产品应用案例

```
案例：智能客服意图识别

场景：识别用户咨询的意图类型

FewShot Prompt：
"请判断用户咨询的意图类型（退款/物流/质量/其他）：

用户：'我买的衣服不合适，想退掉' → 退款
用户：'快递到哪里了？' → 物流
用户：'收到的商品有破损' → 质量
用户：'你们有优惠券吗？' → 其他

用户：'这个怎么比别家贵这么多？' →"

输出：其他

产品价值：
- 准确率从60%提升到90%+
- 无需训练分类模型
- 新增意图只需添加示例
```

## 六、FewShot的局限与注意事项

### 6.1 主要局限

| 局限 | 说明 | 解决方案 |
|------|------|---------|
| **上下文长度限制** | 示例过多会超出Token限制 | 精选最具代表性的示例 |
| **示例偏差** | 示例选择不当导致模型偏差 | 确保示例多样性和代表性 |
| **成本增加** | 每个请求携带示例增加Token消耗 | 使用更短的示例，控制数量 |
| **泛化不足** | 示例未覆盖的情况表现差 | 增加示例多样性，结合指令描述 |

### 6.2 最佳实践

```
FewShot最佳实践清单：

✅ 应该做的：
   - 提供3个高质量、多样化的示例
   - 确保示例格式与期望输出完全一致
   - 示例覆盖任务的主要场景
   - 使用清晰的输入输出分隔符
   - 测试不同示例组合的效果

❌ 不应该做的：
   - 使用过多示例（>5个通常无益）
   - 所有示例都是同一类型
   - 示例之间存在矛盾
   - 示例太长，消耗过多Token
   - 示例质量不高（标注错误）
```

## 七、AI产品经理的FewShot关注点

### 7.1 产品中的FewShot设计

```
AI产品中的FewShot架构：

用户输入 ──→ [意图识别] ──→ [匹配示例模板] ──→ [拼接FewShot Prompt]
                                              │
                                              ▼
                                        示例库管理
                                        ├── 按场景分类
                                        ├── 按意图分类
                                        └── 持续优化更新

产品决策点：
1. 哪些场景需要FewShot？
   └── 格式要求严格、任务复杂、ZeroShot效果不佳的场景

2. 示例从哪里来？
   └── 人工编写、历史数据挖掘、用户反馈收集

3. 示例如何更新？
   └── 定期评估效果、替换低效示例、补充新场景
```

### 7.2 效果评估与优化

| 评估维度 | 指标 | 优化方法 |
|---------|------|---------|
| **格式合规率** | 输出符合预期格式的比例 | 优化示例格式，增加格式约束 |
| **准确率** | 输出内容正确的比例 | 替换错误示例，增加边界示例 |
| **覆盖率** | 能正确处理的场景比例 | 补充缺失场景的示例 |
| **成本效率** | 单位Token的效果 | 精简示例长度，减少示例数量 |

## 九、转行者实践建议

### 9.1 传统产品 vs AI产品的示例思维差异

| 维度 | 传统产品思维 | AI产品思维 | 转型要点 |
|------|-------------|-----------|---------|
| 规则传达 | 文档描述规则 | 示例展示规则 | 从"描述怎么做"到"展示怎么做" |
| 边界定义 | 用文字说明边界 | 用正/负示例定义边界 | 示例比描述更直观 |
| 质量标准 | 验收标准文档 | 输入输出示例对 | 示例就是"活的验收标准" |
| 迭代方式 | 更新文档 | 更新示例库 | 示例库需要持续维护 |

### 9.2 实践练习

```markdown
## 练习1：FewShot vs ZeroShot对比（20分钟）

选择一个任务（如情感分析/意图识别），分别用ZeroShot和FewShot测试：
1. 记录两者的输出格式差异
2. 记录两者的准确率差异
3. 计算FewShot额外消耗的Token数量
4. 评估FewShot的性价比

## 练习2：示例库设计（1小时）

为一个AI产品场景设计FewShot示例库：
1. 选择场景（客服意图识别/内容分类/信息抽取）
2. 设计3-5个高质量示例
3. 确保示例覆盖主要类别
4. 测试示例效果，记录Bad Case

## 练习3：动态FewShot方案设计（1小时）

设计一个动态FewShot系统：
1. 如何构建示例库？
2. 如何根据用户输入检索最相似的示例？
3. 如何控制Token消耗？
4. 如何评估效果？
```

### 9.3 常见误区

| 误区 | 表现 | 正确做法 |
|------|------|---------|
| 示例越多越好 | 放10+个示例 | 3个高质量示例通常足够，过多浪费Token |
| 示例类别单一 | 全是正面示例 | 确保示例覆盖所有类别 |
| 忽视示例格式 | 示例格式不一致 | 示例格式必须与期望输出完全一致 |
| 示例一成不变 | 从不更新示例库 | 定期评估效果，替换低效示例 |
