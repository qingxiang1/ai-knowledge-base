## 一、Prompt概述

### 1.1 什么是Prompt

**Prompt（提示词）** 是指用户输入给大语言模型的指令或问题，用于引导模型生成特定的输出。Prompt是人与大模型交互的主要接口，其质量直接决定了模型输出的质量。

```
Prompt的本质：

用户意图 ──→ Prompt ──→ 大语言模型 ──→ 输出结果
              │
              └── 任务定义
              └── 上下文信息
              └── 输出格式要求
              └── 约束条件
```

### 1.2 为什么Prompt工程重要

| 原因 | 说明 |
|------|------|
| **模型能力依赖Prompt** | 同样的模型，不同的Prompt可能产生天壤之别的结果 |
| **成本效益** | 优化Prompt比微调模型成本低得多 |
| **快速迭代** | 修改Prompt即可快速验证想法 |
| **通用技能** | 适用于所有大模型，不受模型更新影响 |
| **产品竞争力** | 优秀的Prompt是AI产品的核心壁垒之一 |

```
Prompt质量对输出的影响：

差Prompt："写个产品介绍"
→ 输出：泛泛而谈，缺乏重点

好Prompt："你是一位资深产品经理，为一款面向25-35岁都市白领的智能手环撰写产品介绍，
          突出健康监测和长续航两个核心卖点，字数控制在300字以内，语气专业且有温度"
→ 输出：精准、有针对性、符合场景
```

## 二、Prompt的组成要素

### 2.1 核心组成要素

一个完整的Prompt通常包含以下要素：

```
Prompt组成要素：

┌─────────────────────────────────────────┐
│  1. 角色设定 (Role)                      │
│     "你是一位资深数据分析师..."          │
├─────────────────────────────────────────┤
│  2. 任务指令 (Instruction)               │
│     "请分析以下用户行为数据..."          │
├─────────────────────────────────────────┤
│  3. 上下文信息 (Context)                 │
│     "背景：我们是一款电商APP..."         │
├─────────────────────────────────────────┤
│  4. 输入数据 (Input Data)                │
│     "以下是近30天的用户数据：..."        │
├─────────────────────────────────────────┤
│  5. 输出格式 (Output Format)             │
│     "请以JSON格式输出，包含以下字段..."  │
├─────────────────────────────────────────┤
│  6. 约束条件 (Constraints)               │
│     "字数不超过500字，避免使用专业术语"  │
├─────────────────────────────────────────┤
│  7. 示例 (Examples)                      │
│     "例如：输入X → 输出Y"               │
└─────────────────────────────────────────┘
```

### 2.2 各要素详解

#### 角色设定（Role）

为模型设定一个专业身份，使其从特定视角回答问题。

```
角色设定示例：

❌ 无角色：
"解释一下区块链技术"

✅ 有角色：
"你是一位有10年经验的区块链技术专家，正在给一位完全不懂技术的投资人解释区块链。
请用通俗的比喻和生活中的例子来说明。"

效果差异：
- 无角色：技术术语堆砌，难以理解
- 有角色：比喻生动，通俗易懂
```

#### 任务指令（Instruction）

明确告诉模型需要完成什么任务。

```
任务指令设计原则：

1. 使用动词开头
   ✅ "分析..." "总结..." "比较..." "生成..."
   ❌ "关于..." "我想..."

2. 具体明确
   ✅ "列出5个提高用户留存的具体策略"
   ❌ "说说怎么留住用户"

3. 避免歧义
   ✅ "从成本、效果、实施难度三个维度评估"
   ❌ "评估一下这个方案"
```

#### 上下文信息（Context）

提供任务相关的背景信息，帮助模型理解场景。

```
上下文信息示例：

场景：电商APP用户流失分析

Prompt：
"背景信息：
- 我们是一款面向Z世代的潮流电商APP
- 主要用户群体为18-25岁的大学生和年轻白领
- 近三个月DAU下降了15%
- 主要竞品是得物和小红书商城

基于以上背景，分析可能导致用户流失的原因。"
```

#### 输入数据（Input Data）

提供给模型处理的具体内容。

```
输入数据格式：

方式1：直接嵌入
"请总结以下文章：
[文章内容]"

方式2：标记分隔
"### 待处理文本 ###
[文本内容]
### 文本结束 ###

请对以上文本进行情感分析。"

方式3：结构化输入
"数据：
- 日期：2024-01-01
- 销售额：¥100,000
- 订单数：500
- 退款率：5%"
```

#### 输出格式（Output Format）

明确指定模型输出的格式和结构。

```
输出格式示例：

1. 结构化格式
"请以以下JSON格式输出：
{
  \"key_points\": [\"要点1\", \"要点2\"],
  \"sentiment\": \"positive/negative/neutral\",
  \"confidence\": 0-1
}"

2. 列表格式
"请用 bullet points 列出：
- 每个点不超过20字
- 最多列出5个点"

3. 表格格式
"请以表格形式对比A和B方案：
| 维度 | A方案 | B方案 |"

4. 特定风格
"请用Markdown格式输出，包含：
## 标题
### 子标题
- 要点"
```

#### 约束条件（Constraints）

对输出内容施加限制条件。

```
约束条件类型：

长度约束：
- "字数控制在200字以内"
- "不超过3个段落"

内容约束：
- "不要提及竞争对手名称"
- "避免使用'最好''第一'等绝对化表述"

风格约束：
- "语气要亲切，像朋友聊天"
- "使用专业但易懂的语言"

格式约束：
- "每个要点开头用emoji"
- "使用中文输出"
```

## 三、Prompt设计原则

### 3.1 核心设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **清晰明确** | 指令无歧义，模型能准确理解意图 | ❌"写个好点的产品文案" → ✅"为25-30岁女性用户撰写护肤品文案，突出天然成分，语气温柔" |
| **具体详细** | 提供足够的细节和约束 | ❌"分析一下" → ✅"从用户增长、留存、变现三个维度分析，每个维度给出3个关键发现" |
| **结构化** | 使用标记、编号、分隔符组织内容 | 使用###、---、编号等分隔不同部分 |
| **示例驱动** | 提供输入输出示例 | "例如：输入'好评' → 输出{'情感':'正面','分数':0.9}" |
| **迭代优化** | 根据输出效果持续调整Prompt | 记录不同版本的效果，持续优化 |

### 3.2 常见设计模式

```
Prompt设计模式：

模式1：RTF模式（Role-Task-Format）
┌─────────────────────────────────────┐
│ Role：你是一位资深UX设计师         │
│ Task：为新手用户设计 onboarding 流程 │
│ Format：分步骤说明，每步包含界面元素 │
└─────────────────────────────────────┘

模式2：CO-STAR模式
┌─────────────────────────────────────┐
│ Context：提供背景信息               │
│ Objective：明确任务目标             │
│ Style：指定写作风格                 │
│ Tone：设定语气                      │
│ Audience：明确目标受众              │
│ Response：定义输出格式              │
└─────────────────────────────────────┘

模式3：APE模式（Action-Purpose-Expectation）
┌─────────────────────────────────────┐
│ Action：请撰写一封用户召回邮件      │
│ Purpose：目的是让30天未登录用户回流 │
│ Expectation：期望打开率提升10%      │
└─────────────────────────────────────┘
```

## 四、Prompt类型

### 4.1 按任务类型分类

| 类型 | 说明 | 示例 |
|------|------|------|
| **问答型** | 直接回答问题 | "什么是AARRR模型？" |
| **生成型** | 创作内容 | "写一篇关于AI产品设计的公众号文章" |
| **分析型** | 分析数据或文本 | "分析这段用户反馈的情感倾向" |
| **转换型** | 格式或语言转换 | "将以下中文翻译成英文" |
| **推理型** | 逻辑推理和决策 | "基于以下数据，预测下季度销量" |
| **代码型** | 生成或解释代码 | "用Python写一个快速排序算法" |

### 4.2 按复杂度分类

```
Prompt复杂度层级：

Level 1：简单Prompt
"翻译：Hello World"
└── 直接指令，无额外要求

Level 2：增强Prompt
"你是一位专业翻译，将以下英文翻译成地道的中文：
Hello World
要求：保持原意，语言自然流畅"
└── 增加角色和约束

Level 3：结构化Prompt
"### 角色 ###
你是一位资深技术文档翻译

### 任务 ###
翻译以下技术文档段落

### 输入 ###
Hello World

### 要求 ###
1. 术语准确
2. 符合中文技术文档规范
3. 保留原文格式"
└── 完整要素，结构化组织

Level 4：Few-shot Prompt
"### 示例 ###
英文：The quick brown fox
中文：敏捷的棕色狐狸

### 待翻译 ###
Hello World"
└── 增加示例，引导模型学习模式
```

## 五、Prompt工程基础技巧

### 5.1 分隔符使用

使用明确的分隔符区分Prompt的不同部分。

```
分隔符示例：

### 角色设定 ###
你是一位资深产品经理

---

### 任务 ###
分析以下用户反馈数据

### 数据 ###
"""
1. 用户A：APP闪退严重，无法使用
2. 用户B：希望增加夜间模式
3. 用户C：加载速度太慢了
"""

### 输出要求 ###
- 分类整理反馈
- 标注优先级
- 给出处理建议
```

### 5.2 清晰 vs 简洁的平衡

```
平衡原则：

过于简单（信息不足）：
"分析数据" 
→ 模型不知道分析什么、怎么分析

过于复杂（信息冗余）：
"你是一位有15年经验的产品经理，曾经在阿里巴巴、腾讯工作过，
精通数据分析、用户研究、市场分析...
（大段无关背景）
请分析数据..."
→ 关键信息被淹没

最佳实践：
"你是一位资深产品经理（角色简洁）

背景：我们是做SaaS工具的创业公司，ARR 1000万（关键上下文）

任务：分析以下客户流失原因，按影响程度排序（明确指令）

数据：..."
```

### 5.3 避免常见错误

| 错误类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| **指令模糊** | "写个好点的文案" | "撰写吸引25-30岁女性的护肤品文案，突出天然成分" |
| **角色冲突** | "你既是专家又是新手" | 明确一个主要角色 |
| **约束矛盾** | "详细但简短" | "详细但控制在300字以内" |
| **示例不足** | 只给一个模糊示例 | 提供2-3个清晰、多样的示例 |
| **忽略边界** | 不说明不做什么 | 明确排除项："不要提及价格" |

## 六、AI产品经理的Prompt关注点

### 6.1 Prompt在产品中的位置

```
AI产品架构中的Prompt：

用户输入 ──→ [Prompt工程层] ──→ 大模型API ──→ 输出处理 ──→ 用户看到的结果
              │
              ├── 系统Prompt（隐藏）
              │   └── 角色设定、安全约束、输出格式
              │
              ├── 动态Prompt（半隐藏）
              │   └── 根据上下文拼接的模板
              │
              └── 用户Prompt（可见）
                  └── 用户的直接输入

AI产品经理的核心工作：
- 设计系统Prompt
- 优化动态Prompt模板
- 引导用户输入更好的Prompt
```

### 6.2 Prompt设计的度量指标

| 指标 | 说明 | 测量方法 |
|------|------|---------|
| **输出准确率** | 模型输出符合预期的比例 | 人工标注或自动评估 |
| **格式合规率** | 输出格式符合要求的比例 | 格式校验 |
| **用户满意度** | 用户对输出的满意程度 | 用户反馈收集 |
| **Token效率** | 达到同等效果所需的最少Token | 对比实验 |
| **鲁棒性** | 输入变化时输出的稳定性 | 变异输入测试 |

### 6.3 与团队协作要点

| 协作方 | 沟通要点 |
|--------|---------|
| **算法工程师** | "当前Prompt在XX场景下输出不稳定，需要调整模型参数还是优化Prompt？" |
| **产品经理** | "这个功能的系统Prompt需要包含哪些安全约束？" |
| **运营团队** | "活动文案的Prompt模板需要支持哪些变量？" |
| **测试团队** | "这些边界 case 的Prompt表现如何？" |

## 八、转行者实践建议

从传统产品或非技术岗位转型AI产品的同学，Prompt工程是你最需要优先掌握的实操技能。

### 8.1 传统产品 vs AI产品的交互设计差异

| 维度 | 传统产品思维 | AI产品思维 | 转型要点 |
|------|-------------|-----------|---------|
| 交互方式 | 按钮/表单/菜单 | 自然语言对话 | 从"UI交互"到"语言交互" |
| 功能定义 | PRD定义功能逻辑 | 系统Prompt定义AI行为 | 从"功能规格"到"行为规范" |
| 边界控制 | 代码限制操作范围 | Prompt约束输出范围 | 从"硬约束"到"软约束" |
| 效果评估 | 功能是否正常 | 输出是否准确、安全、一致 | 从"功能测试"到"效果评估" |
| 迭代方式 | 修改代码 | 修改Prompt | 从"代码迭代"到"Prompt迭代" |

### 8.2 实践练习

```markdown
## 练习1：Prompt要素拆解（20分钟）

拆解以下Prompt的7个组成要素：
"你是一位资深产品经理，擅长用户研究。
 请分析以下用户反馈，挖掘真实需求。
 背景：我们是一款面向Z世代的潮流电商APP。
 用户反馈：'APP经常闪退，太烦了'
 请以JSON格式输出，包含：表面需求、深层需求、痛点、建议。
 字数控制在200字以内，避免使用专业术语。"

答案：
1. 角色：资深产品经理
2. 任务：分析用户反馈，挖掘需求
3. 上下文：Z世代潮流电商APP
4. 输入：用户反馈内容
5. 格式：JSON
6. 约束：200字以内、避免专业术语
7. 示例：无（Zero-shot）

## 练习2：系统Prompt设计（1小时）

为一个AI产品场景设计系统Prompt：
1. 选择场景（智能客服/AI写作助手/代码助手）
2. 按照角色+职责+风格+安全+边界设计
3. 测试3个典型输入，评估输出质量
4. 记录需要优化的点

## 练习3：Prompt优化迭代（1小时）

选择一个日常工作中需要用AI完成的任务：
1. 写出初版Prompt
2. 测试输出，记录问题
3. 按照迭代优化法逐步改进
4. 记录每个版本的效果变化
```

### 8.3 常见误区

| 误区 | 表现 | 正确做法 |
|------|------|---------|
| Prompt太简单 | "帮我写个文案" | 包含角色、任务、格式、约束等完整要素 |
| 忽视系统Prompt | 只关注用户输入 | 系统Prompt决定AI行为基线，必须精心设计 |
| 一次成型心态 | 写完不测试 | Prompt需要迭代优化，至少3轮测试 |
| 忽视Token成本 | Prompt写很长 | 控制Prompt长度，优化Token消耗 |
| 忽视安全约束 | 不设安全边界 | 必须在系统Prompt中设定安全护栏 |
