## 一、转化分析基础概念

### 1.1 定义与标准

```
转化分析定义体系：

核心定义
├── 基本定义
│   ├── 转化率 = 完成目标行为的用户数 / 参与转化的用户数
│   ├── 时间范围：根据业务特点选择（会话级、天级、周级等）
│   └── 计算维度：按用户唯一标识（user_id）去重
│
├── 转化类型
│   ├── 注册转化
│   │   └── 从访问到完成注册的转化
│   ├── 付费转化
│   │   └── 从访问到完成付费的转化
│   └── 功能使用转化
│       └── 从访问到完成核心功能使用的转化
│
└── 转化口径
    ├── 严格口径
    │   ├── 定义：按会话计算，用户必须在同一会话内完成转化
    │   ├── 优势：反映真实即时转化
    │   └── 适用：短决策链路产品
    └── 宽松口径
        ├── 定义：按用户计算，用户可以在任意时间完成转化
        ├── 优势：转化率高，适合长决策链路
        └── 适用：长决策链路产品

相关概念
├── 转化成本
│   └── 获取一次转化所需的成本
├── 转化价值
│   └── 每次转化带来的平均价值
└── 转化周期
    └── 用户从首次接触到完成转化的平均时间
```

### 1.2 计算方式

```python
"""
转化计算方法

不同场景下的转化计算
"""

class ConversionCalculation:
    """转化计算"""
    
    @staticmethod
    def get_calculation_methods() -> dict:
        """
        获取计算方法
        
        Returns:
            方法定义
        """
        return {
            "基础计算": {
                "公式": "转化率 = 完成目标行为的用户数 / 参与转化的用户数",
                "说明": "统计每个转化步骤的转化情况",
                "SQL 示例": """
                WITH conversion_steps AS (
                    SELECT 
                        user_id,
                        MIN(CASE WHEN event_name = 'step1' THEN event_time END) as step1_time,
                        MIN(CASE WHEN event_name = 'step2' THEN event_time END) as step2_time,
                        MIN(CASE WHEN event_name = 'step3' THEN event_time END) as step3_time
                    FROM user_events
                    WHERE event_name IN ('step1', 'step2', 'step3')
                    GROUP BY user_id
                )
                SELECT 
                    COUNT(DISTINCT user_id) as step1_users,
                    COUNT(DISTINCT CASE WHEN step2_time IS NOT NULL THEN user_id END) as step2_users,
                    COUNT(DISTINCT CASE WHEN step3_time IS NOT NULL THEN user_id END) as step3_users,
                    COUNT(DISTINCT CASE WHEN step2_time IS NOT NULL THEN user_id END) * 1.0 / COUNT(DISTINCT user_id) as step2_conversion,
                    COUNT(DISTINCT CASE WHEN step3_time IS NOT NULL THEN user_id END) * 1.0 / COUNT(DISTINCT CASE WHEN step2_time IS NOT NULL THEN user_id END) as step3_conversion
                FROM conversion_steps
                """
            },
            "会话级计算": {
                "说明": "用户必须在同一会话内完成所有步骤",
                "适用": "短决策链路产品"
            },
            "用户级计算": {
                "说明": "用户可以在任意时间完成步骤",
                "适用": "长决策链路产品"
            }
        }
    
    @staticmethod
    def get_conversion_variants() -> dict:
        """
        获取转化衍生指标
        
        Returns:
            衍生指标
        """
        return {
            "总体转化率": {
                "定义": "完成最终步骤的用户数 / 进入第一步的用户数",
                "公式": "最终步骤用户数 / 第一步用户数",
                "意义": "反映整体转化效率"
            },
            "步骤转化率": {
                "定义": "完成某步骤的用户数 / 进入该步骤的用户数",
                "公式": "步骤完成用户数 / 步骤进入用户数",
                "意义": "反映各步骤转化效率"
            },
            "流失率": {
                "定义": "1 - 转化率",
                "公式": "1 - 转化率",
                "意义": "反映各步骤流失情况"
            }
        }
```

```python
# 转化常见误区
"""
转化常见误区：

1. 唯转化论
   ├── 问题：只关注转化数字，忽视质量
   ├── 后果：用户增长但价值不增长
   └── 解决：结合时长、价值综合评估

2. 口径混乱
   ├── 问题：不同团队转化定义不同
   ├── 后果：数据对不上，争论不休
   └── 解决：统一口径，文档化，定期对齐

3. 刷量造假
   ├── 问题：通过激励刷出虚假转化
   ├── 后果：数据好看，实际无价值
   └── 解决：定义严格活跃标准，监控异常行为

4. 忽视结构
   ├── 问题：只看总数，不看构成
   ├── 后果：掩盖老用户流失问题
   └── 解决：拆分新老用户、分层分析

5. 过度反应
   ├── 问题：单日波动就紧张，频繁调整
   ├── 后果：策略摇摆，团队疲惫
   └── 解决：关注趋势而非单日，设置合理阈值
"""
```
