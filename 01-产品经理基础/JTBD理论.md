## 一、JTBD理论概述

### 1.1 什么是JTBD

**Jobs-to-be-Done（待完成的工作）** 是一个理解用户需求的理论框架，由 Clayton Christensen 提出。核心观点是：**用户不是在购买产品，而是在「雇用」产品来完成某个工作**。

### 1.2 核心思想

```
传统思维：用户想要什么产品？
    ↓
JTBD思维：用户想完成什么工作？

案例对比：
传统：用户想要一台电钻
JTBD：用户想要在墙上打一个孔

传统：用户想要一个AI写作工具
JTBD：用户想要高效地产出高质量内容
```

### 1.3 用户「雇用」决策

用户在决定是否「雇用」一个产品时，会考虑：

| 考量维度 | 问题 | 示例 |
|---------|------|------|
| 功能性 | 能完成任务吗？ | AI能写出可用的文章吗？ |
| 情感性 | 让我感觉如何？ | 用这个工具有面子吗？ |
| 社会性 | 别人怎么看？ | 领导会认可我用的工具吗？ |
| 情境性 | 在我当前场景下能用吗？ | 能在地铁上用吗？ |

## 二、JTBD的核心框架

### 2.1 Job的构成

```
Job = 动词 + 对象 + 情境（可选）

功能性Job：
- 写文章（而不是：用AI写作工具）
- 查询物流（而不是：打开客服页面）

情感性Job：
- 显得专业（而不是：购买专业软件）
- 减轻焦虑（而不是：设置提醒）

社会性Job：
- 让团队认可（而不是：使用协作工具）
- 在社交媒体上分享（而不是：使用某个App）

情境性Job：
- 在通勤路上完成工作
- 在没有网络的飞机上使用
```

### 2.2 JTBD描述模板

```markdown
## JTBD描述模板

"当 [情境]，我想要 [动机]，以便 [预期结果]。"

## 示例

场景1：AI写作
"当 deadline逼近但我还没动笔，我想要快速生成初稿，
以便 在截止时间前完成任务，避免被领导批评。"

场景2：AI客服
"当 我查询订单问题等待太久，我想要立即得到答案，
以便 不用浪费时间，快速解决我的问题。"

场景3：AI学习助手
"当 我看不懂专业术语，我想要有人用简单的话解释，
以便 我能继续学习，不被卡住。"
```

### 2.3 功能性与情感性Job

```
Job分类矩阵：

              情感需求
                 │
                 │
    基础型        │      社交型
    Job           │       Job
    （完成任务）   │      （身份认同）
                 │
   ──────────────┼─────────────
                 │
    效率型        │      体验型
    Job           │       Job
    （省时省力）   │      （享受过程）
                 │
                 │
              功能需求

AI产品常见Job分布：
- AI写作工具：效率型Job为主
- AI社交App：体验型Job为主
- AI客服系统：基础型Job为主
```

## 三、JTBD分析流程

### 3.1 整体流程

```
识别Job → 理解动机 → 分析情境 → 评估需求 → 转化为产品功能
    │                                            │
    └────────────────────────────────────────────┘
```

### 3.2 Step 1：识别Job

#### Job采集方法

| 方法 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| 用户访谈 | 深度理解Job | 深入、具体 | 样本小 |
| 问卷调研 | 验证普遍性 | 样本大 | 浅层 |
| 行为数据分析 | 发现真实Job | 客观 | 需要数据积累 |
| 客服/评论分析 | 发现痛点 | 成本低 | 分散 |

#### 访谈技巧：聚焦于「工作」而非「产品」

**错误示范**：
```
Q：你为什么用这个AI工具？
A：因为它功能强大，有xxx功能。
```

**正确示范**：
```
Q：告诉我一个你最近使用AI工具的场景。
A：上周我要写一个方案...
Q：当时的情况是怎样的？
A：第二天就要交了，我还没开始...
Q：你当时最想要的是什么？
A：有人能帮我快速出个初稿...
Q：如果有魔法棒，你希望它怎么帮你？
A：直接帮我写完最好了，或者帮我理清思路也行...
```

### 3.3 Step 2：理解动机

#### 马斯洛需求层次在JTBD中的应用

```
                    自我实现
                       ▲
                       │
                 尊重需求 ▲
                  （被认可、被尊重）
                       ▲
                       │
                 社交需求 ▲
                 （归属感、联系）
                       ▲
                       │
                 安全需求 ▲
                 （保障、确定性）
                       ▲
                       │
                 基本需求 ▲
                （功能、性能）
                       ▲
                       │
                       │
```

#### 深层动机挖掘

| 表层表达 | 深层Job |
|---------|--------|
| "这个工具很好用" | 节省时间、降低学习成本 |
| "感觉不够专业" | 需要被他人认可 |
| "不想让同事知道我在用AI" | 担心被视为不努力 |
| "AI写的太机械了" | 需要个性化表达 |
| "等AI回答好慢" | 效率被拖累 |

### 3.4 Step 3：分析情境

#### 情境维度

```markdown
## 情境分析框架

### 时间情境
- 什么时间做这个工作？
- 紧迫程度如何？
- 周期性还是临时性？

### 地点情境
- 在哪里做？
- 环境条件如何？（安静/嘈杂、网络状况）

### 社交情境
- 独自完成还是协作？
- 结果会被谁看到？
- 需要向谁汇报？

### 技术情境
- 使用什么设备？
- 可用哪些工具？
- 有哪些限制？

### 情绪情境
- 做这项工作时的情绪状态？
- 期望的结果是什么？
- 失败的后果是什么？
```

### 3.5 Step 4：评估需求

#### Job评分标准

| 评估维度 | 问题 | 评分（1-5） |
|---------|------|------------|
| 重要性 | 这个Job对用户有多重要？ | 1-5 |
| 紧迫性 | 用户现在有多需要解决这个Job？ | 1-5 |
| 现有方案满意度 | 当前方案对这个Job的满足程度？ | 1-5 |
| 频次 | 用户多久遇到一次这个Job？ | 1-5 |
| 付费意愿 | 用户愿意为此付多少钱？ | 1-5 |

#### 机会识别

```markdown
## Job优先级矩阵

高重要性
    │
    │  ★ 核心Job     ★ 战略Job
    │  （必须做）     （差异化）
    │
 ───┼────────────────────────────
    │
    │  ○ 基础Job      ○ 潜力Job
    │  （不投入）     （观察）
    │
    │
低重要性

高现有方案满意度  ←→  低现有方案满意度
```

## 四、AI产品中的JTBD应用

### 4.1 AI产品常见Job分析

#### AI写作工具Job Map

```markdown
## AI写作工具的Job Map

### 核心Job
| Job | 重要性 | 现有方案 | 痛点 |
|-----|--------|---------|------|
| 快速生成初稿 | ★★★★★ | 手动写2小时 | 效率低 |
| 改善写作质量 | ★★★★★ | 自己改多遍 | 费时费力 |
| 获取灵感思路 | ★★★★☆ | 搜索+思考 | 分散精力 |
| 保持风格一致 | ★★★☆☆ | 刻意注意 | 容易走偏 |
| 避免低级错误 | ★★★☆☆ | 检查2遍 | 容易遗漏 |

### 情感Job
| Job | 重要性 | 现有方案 | 痛点 |
|-----|--------|---------|------|
| 显得专业 | ★★★★☆ | — | 担心AI味太重 |
| 减轻写作焦虑 | ★★★★★ | — | deadline压力 |
| 获得成就感 | ★★★☆☆ | — | 自我表达 |
```

### 4.2 从Job到功能

```
Job → 功能映射示例：

Job: "快速生成初稿"
│
├── 功能1: AI生成文章
│   └── 触发条件：用户输入主题/关键词
│
├── 功能2: 续写扩写
│   └── 触发条件：用户写了开头
│
├── 功能3: 大纲生成
│   └── 触发条件：用户有想法但无结构
│
└── 功能4: 模板套用
    └── 触发条件：用户有固定格式需求

注意：同一个Job可以有多种满足方式，
      需要根据用户情境选择最优方案。
```

### 4.3 AI产品设计中的JTBD思考

| JTBD问题 | AI产品设计考量 |
|---------|--------------|
| 用户真正想完成的工作是什么？ | 不是「用AI」，而是「完成任务」 |
| 用户在什么情境下会雇用AI？ | 效率压力、知识盲区、时间紧迫 |
| 用户愿意为这个Job付多少钱？ | 决定商业模式 |
| 用户「解雇」AI的原因是什么？ | 质量不达标、错误代价高、信任缺失 |
| AI比现有方案好在哪里？ | 速度、成本、可扩展性 |

## 六、JTBD实战案例

### 6.1 案例：AI客服系统JTBD分析

#### 背景
某电商平台希望升级客服系统，引入AI能力。

#### JTBD分析过程

**Step 1：识别核心Job**

```markdown
## 用户访谈摘要

用户A（退货咨询）：
"我就是想知道能不能退，怎么退，多久能收到钱。
最好一句话就能说清楚，别让我等。"

用户B（订单查询）：
"我下单后就想知道什么时候到。
不用一直刷物流页面，有消息主动告诉我。"

用户C（投诉）：
"上次收到的东西有问题，打电话没人接，
后来等了2天才有人回复，态度还很差。"
```

**Step 2：Job分类与优先级**

```markdown
## AI客服Job Map

### P0 Job（必须满足）
| Job | 重要性 | 现有痛点 |
|-----|--------|---------|
| 快速得到准确答案 | ★★★★★ | 等待时间长 |
| 24小时可用 | ★★★★★ | 夜间无人响应 |
| 不用重复描述问题 | ★★★★☆ | 转人工要重新说 |

### P1 Job（重要）
| Job | 重要性 | 现有痛点 |
|-----|--------|---------|
| 问题被真正解决 | ★★★★★ | 回答但没解决 |
| 可以联系人工 | ★★★★☆ | 找不到入口 |
| 记录我的偏好 | ★★★☆☆ | 每次都要重新说 |

### P2 Job（加分）
| Job | 重要性 | 现有痛点 |
|-----|--------|---------|
| 主动提醒我问题进展 | ★★★☆☆ | 要自己追问 |
| 预测我可能的问题 | ★★★☆☆ | 被动响应 |
```

**Step 3：转化为产品功能**

```markdown
## Job → 功能映射

Job: "快速得到准确答案"
├── 功能1: 智能FAQ匹配
│   └── 输入问题 → 直接给答案
│
├── 功能2: 知识库RAG问答
│   └── 自然语言查询 → 精准答案
│
└── 功能3: 上下文理解
    └── 多轮对话，不用重复描述

Job: "问题被真正解决"
├── 功能1: 解决状态追踪
│   └── 记录问题→处理→闭环
│
├── 功能2: AI+人工协作
│   └── AI处理常见问题，人工处理复杂问题
│
└── 功能3: 满意度回访
    └── 问题解决后主动确认

Job: "可以联系人工"
├── 功能1: 明显的转人工入口
│   └── 任何时刻都能转
│
├── 功能2: 智能转接
│   └── 根据问题类型转给对应客服
│
└── 功能3: 转接上下文同步
    └── 人工能看到AI对话历史
```

## 七、JTBD与产品迭代

### 7.1 迭代优先级

```
JTBD驱动的迭代逻辑：

1. 核心Job满足了吗？
   └── 未满足 → 优先级最高
   
2. 核心Job的满足程度可以提升吗？
   └── 可以 → 持续优化
   
3. 次要Job可以满足吗？
   └── 有资源 → 逐步扩展
   
4. 新Job出现了？
   └── 评估 → 决定是否满足
```

### 7.2 监测指标

```markdown
## JTBD健康度指标

| Job | 监测指标 | 目标值 |
|-----|---------|--------|
| 快速得到答案 | 首轮解决率 | > 70% |
| 问题被解决 | 最终解决率 | > 90% |
| 联系人工 | 转人工率 | < 20% |
| 不用重复 | 上下文丢失率 | < 5% |
```

## 九、转行者实践建议

从传统产品转型AI产品的同学，在应用JTBD时需要特别注意以下要点：

### 9.1 传统产品 vs AI产品的JTBD差异

| 维度 | 传统产品JTBD | AI产品JTBD | 转型要点 |
|------|-------------|-----------|---------|
| Job来源 | 用户明确表达 | 用户可能自己都没意识到 | 用「魔法棒」问题挖掘潜在需求 |
| 解决方案 | 功能堆叠 | AI能力匹配 | 先判断AI能否完成这个Job |
| 情感维度 | 品牌认同 | 对AI的信任与依赖 | 重视「信任建立」这个隐性Job |
| 竞争维度 | 功能对比 | 效果+体验+成本综合对比 | 不只比功能，更要比AI效果 |
| 迭代方式 | 需求驱动 | 数据+反馈双轮驱动 | 建立AI效果反馈闭环 |

### 9.2 实践练习

```markdown
## 练习1：Job重新定义（30分钟）

选择一个你熟悉的产品，用JTBD重新定义用户需求：

传统描述 → JTBD描述
"用户需要一个AI翻译功能" → "当用户阅读外文资料时，想要快速理解内容，以便不中断阅读节奏"

练习产品建议：
- ChatGPT / Kimi / 文心一言
- Notion AI / 飞书智能助手
- Midjourney / Stable Diffusion

## 练习2：AI产品Job Map（2小时）

选择一个AI产品，完成完整的Job Map：
1. 列出5-8个功能性Job
2. 识别2-3个情感性Job
3. 评估每个Job的重要性和现有方案满意度
4. 找到机会点（高重要性 + 低满意度）

## 练习3：竞品Job对比（1小时）

选择2-3个同类AI产品，对比它们对同一组Job的满足程度：
- 示例：对比 ChatGPT vs Claude vs 文心一言 在「快速获取准确答案」这个Job上的表现
```

### 9.3 常见误区

| 误区 | 表现 | 正确做法 |
|------|------|---------|
| 把AI当Job | "用户想要一个AI助手" | AI是解决方案，不是Job |
| 忽略情感Job | 只关注功能需求 | AI产品的信任、安全感是关键Job |
| 照搬传统框架 | 直接套用传统产品的JTBD模板 | 加入AI特有的维度（信任度、兜底需求等） |
| 忽视情境变化 | 认为AI产品的使用情境与传统产品相同 | AI产品可能在碎片化、移动化场景下使用 |
