# 09 - 工作流编排

> 本章聚焦 AI 工作流从“能串节点”走向“能稳定上线”。内容覆盖 Dify、Coze、Flowise、n8n 等低代码平台，也覆盖 LangChain、LangGraph 等代码框架，以及企业级工作流设计和自动化方案。

---

## 本章定位

`09-工作流编排` 解决的不是“会拖几个节点”，而是以下 4 个更关键的问题：

1. 如何把 Prompt、模型、工具、知识库和人工审核编排成稳定业务流程
2. 如何在效果、成本、时延、可靠性之间做工作流级权衡
3. 如何区分低代码平台与代码框架的适用边界
4. 如何为企业级 AI 自动化建立监控、回滚、审计与扩展机制

---

## 企业级学习原则

1. 先定义业务闭环，再决定节点与平台
2. 先保障失败恢复，再追求流程复杂度
3. 先控制成本和时延，再扩大工作流规模
4. 始终把工作流看作生产系统，而不是演示流程

## 关键术语速查

| 术语              | 说明                                               |
| ----------------- | -------------------------------------------------- |
| Workflow          | 将多个 AI 或系统节点按逻辑顺序编排的自动化流程     |
| Node              | 工作流中的处理单元，如模型调用、工具调用、条件判断 |
| State             | 工作流执行时传递的数据和上下文                     |
| Branch            | 根据条件进入不同执行路径的分支机制                 |
| Retry             | 节点失败后的重试机制                               |
| Human-in-the-loop | 关键环节保留人工审核或确认                         |
| Graph             | 以图结构描述节点关系的工作流模型                   |
| Trigger           | 启动工作流的条件，如时间、事件或 API 调用          |

## 学习建议

1. 先从一个真实业务流程出发，再选择平台和框架
2. 低代码平台适合验证与运营提效，复杂系统仍需代码框架兜底
3. 工作流设计时优先考虑失败恢复与人工接管
4. 节点越多不代表越强，通常也意味着更高成本和更高延迟
5. 企业工作流必须有观测、审计、限流和回滚能力
