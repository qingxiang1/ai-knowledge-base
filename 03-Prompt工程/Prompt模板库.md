# Prompt模板库

> 经过验证的高质量Prompt模板，覆盖产品经理日常工作场景，可直接复用或按需调整

---

## 一、需求分析类

### 1.1 用户需求分析

```
模板：用户需求深度分析

角色：你是一位有8年经验的用户研究专家，擅长定性分析和需求挖掘。

任务：对以下用户反馈进行深度分析，挖掘真实需求和痛点。

用户反馈：
{{user_feedback}}

请按以下结构输出：
1. 表面需求：用户明确表达的需求
2. 深层需求：用户未明说但真实的需求（使用5 Whys方法）
3. 痛点分析：当前解决方案的不足
4. 机会点：潜在的产品机会
5. 优先级建议：按影响程度和可行性排序
6. 验证建议：如何验证这些需求的真伪

要求：
- 每个分析点提供具体依据
- 区分事实和推测
- 使用用户原话作为支撑
```

### 1.2 竞品分析

```
模板：竞品功能对比分析

角色：你是一位资深产品分析师，擅长竞品分析和市场研究。

任务：对以下竞品进行功能对比分析。

我方产品：{{my_product}}
竞品列表：{{competitor_list}}
分析维度：{{analysis_dimensions}}

请按以下结构输出：
1. 竞品概览：各竞品定位、目标用户、核心优势
2. 功能对比矩阵：以表格形式对比各维度功能
3. 差异化分析：我方产品的优势和劣势
4. 功能缺口：竞品有但我方没有的功能（评估是否需要跟进）
5. 竞争策略建议：基于分析的差异化策略

要求：
- 客观中立，避免主观偏见
- 每个结论提供证据支撑
- 标注信息来源和置信度
```

### 1.3 用户画像生成

```
模板：用户画像构建

角色：你是一位用户研究专家，擅长基于数据构建用户画像。

任务：根据以下信息构建详细的用户画像。

可用数据：
{{user_data}}

请输出以下画像维度：
1. 基础信息：年龄、性别、职业、收入、地域
2. 行为特征：使用习惯、消费行为、决策方式
3. 心理特征：价值观、动机、痛点、期望
4. 场景描述：典型使用场景（按频率排序）
5. 需求优先级：核心需求 vs 期望需求 vs 兴奋需求
6. 一句话描述：用一句话概括这个用户群体

要求：
- 基于提供的数据，不编造
- 数据不足的地方标注"待验证"
- 提供画像的应用建议
```

## 二、产品设计类

### 2.1 PRD撰写辅助

```
模板：PRD框架生成

角色：你是一位资深产品经理，擅长撰写清晰、可执行的PRD。

任务：为以下功能撰写PRD框架。

功能概述：{{feature_description}}
目标用户：{{target_user}}
业务目标：{{business_goal}}

请按以下结构输出PRD：

# {{feature_name}} 产品需求文档

## 1. 背景与目标
- 背景：为什么要做这个功能？
- 目标：具体的业务目标和用户价值
- 成功指标：如何衡量功能成功？

## 2. 需求范围
- In Scope：本次迭代包含的内容
- Out of Scope：明确不包含的内容

## 3. 用户故事
- 作为[用户角色]，我希望[需求]，以便[价值]
- 至少覆盖核心场景和边界场景

## 4. 功能详述
- 流程图：核心业务流程
- 页面/模块说明：每个页面的功能和交互
- 规则逻辑：业务规则、计算公式、状态流转

## 5. 非功能需求
- 性能要求
- 兼容性要求
- 安全要求

## 6. 数据埋点
- 需要采集的事件和属性
- 分析维度

## 7. 风险评估
- 技术风险
- 业务风险
- 应对策略

要求：
- 逻辑清晰，无歧义
- 开发可直接基于PRD进行技术方案设计
- 标注需要进一步确认的内容
```

### 2.2 交互设计评估

```
模板：交互设计可用性评估

角色：你是一位资深UX设计师，精通可用性原则和交互设计规范。

任务：评估以下交互设计的可用性问题。

设计描述：{{design_description}}
用户场景：{{user_scenario}}

请按以下结构输出评估结果：
1. 整体评价：设计的优点和明显问题
2. 可用性检查（对照尼尔森十大原则）：
   - 系统状态可见性
   - 系统与现实世界匹配
   - 用户控制与自由
   - 一致性与标准
   - 错误预防
   - 识别而非回忆
   - 灵活性与效率
   - 美学与极简
   - 错误诊断与恢复
   - 帮助文档
3. 具体问题列表：
   | 严重程度 | 问题描述 | 违反原则 | 改进建议 |
4. 优化方案：针对高优先级问题的具体优化建议
5. 参考案例：业界优秀实践参考

要求：
- 每个问题说明对用户体验的具体影响
- 改进建议要具体可执行
```

### 2.3 产品方案评审

```
模板：产品方案结构化评审

角色：你是一位产品总监，正在评审团队提交的产品方案。

任务：对以下产品方案进行结构化评审。

方案概述：{{solution_summary}}
目标用户：{{target_user}}
预期效果：{{expected_outcome}}
资源投入：{{resource_investment}}

请从以下维度进行评审：

1. 价值评估
   - 用户价值：解决了什么真实问题？
   - 业务价值：对业务指标的预期影响？
   - 价值与投入比：ROI是否合理？

2. 可行性评估
   - 技术可行性：技术难度和风险
   - 资源可行性：人力、时间是否充足
   - 合规可行性：是否涉及法律/合规风险

3. 完整性评估
   - 需求是否清晰完整？
   - 边界场景是否考虑？
   - 异常流程是否覆盖？

4. 风险评估
   - 主要风险点
   - 风险发生概率和影响
   - 风险应对策略

5. 评审结论
   - 通过/有条件通过/不通过
   - 关键修改意见
   - 后续跟进事项

要求：
- 评审意见要具体，避免泛泛而谈
- 区分"必须修改"和"建议优化"
- 保持建设性，给出改进方向
```

## 三、数据分析类

### 3.1 数据解读

```
模板：业务数据解读报告

角色：你是一位资深数据分析师，擅长从数据中发现业务洞察。

任务：解读以下业务数据，输出分析报告。

数据背景：{{data_context}}
数据内容：{{data_content}}
分析目标：{{analysis_goal}}

请按以下结构输出：
1. 数据概览：关键指标和趋势
2. 核心发现：3-5个最重要的发现
3. 深度分析：
   - 是什么：数据表现
   - 为什么：原因分析（提供多个假设）
   - 意味着什么：业务影响
4. 对比分析：与历史数据/行业基准/竞品的对比
5. 行动建议：基于数据的 actionable insights
6. 数据局限：当前数据的局限性和需要补充的数据

要求：
- 用数据说话，每个结论有数据支撑
- 区分相关性和因果性
- 不确定的地方明确标注
```

### 3.2 AB测试分析

```
模板：AB测试效果分析

角色：你是一位数据科学家，精通实验设计和统计检验。

任务：分析以下AB测试结果，判断实验是否有效。

实验背景：{{experiment_background}}
实验设计：
- 对照组：{{control_group}}
- 实验组：{{treatment_group}}
- 样本量：{{sample_size}}
- 运行时长：{{duration}}

实验数据：{{experiment_data}}

请按以下结构输出：
1. 实验设计评估
   - 随机性检查
   - 样本量是否充足
   - 实验时长是否合理

2. 统计检验
   - 核心指标变化及置信区间
   - P值和统计显著性
   - 效应量（实际业务意义）

3. 细分分析
   - 不同用户群体的效果差异
   - 是否存在异质性处理效应

4. 业务解读
   - 结果的业务含义
   - 是否达到预期

5. 决策建议
   - 是否全量上线
   - 如果不上线，原因和改进方向
   - 如果上线，预期影响和监控计划

要求：
- 统计方法要严谨
- 区分统计显著和业务显著
- 考虑实验的潜在偏误
```

## 四、文档写作类

### 4.1 产品说明书

```
模板：用户产品说明书

角色：你是一位技术文档工程师，擅长将复杂功能转化为用户易懂的说明。

任务：为以下产品功能撰写用户说明书。

功能说明：{{feature_description}}
目标用户：{{target_user}}

请按以下结构输出：

# {{feature_name}} 使用指南

## 功能介绍
- 一句话说明这个功能是什么
- 能解决什么问题
- 适合什么场景使用

## 使用前准备
- 前提条件
- 权限要求
- 相关配置

## 操作步骤
1. 步骤一：...
   - 截图/示意图位置
   - 注意事项
2. 步骤二：...
   ...

## 常见问题
Q1：...
A1：...

## 相关资源
- 视频教程链接
- 相关功能链接

要求：
- 语言简洁，避免技术术语
- 步骤清晰，可一步步跟随操作
- 预判用户可能遇到的问题
```

### 4.2 汇报材料

```
模板：产品汇报PPT大纲

角色：你是一位产品总监，擅长向管理层汇报产品工作。

任务：为以下主题准备汇报PPT大纲。

汇报主题：{{topic}}
汇报对象：{{audience}}
汇报时长：{{duration}}
核心信息：{{key_messages}}

请按以下结构输出PPT大纲：

1. 封面
   - 标题、汇报人、日期

2. 背景与目标（1页）
   - 为什么要做这个汇报？
   - 希望达成什么目标？

3. 核心内容（3-5页）
   - 每页一个核心观点
   - 观点+数据+案例的结构
   - 建议的图表类型

4. 关键结论（1页）
   - 3个核心结论
   - 每个结论的支撑依据

5. 下一步计划（1页）
   - 具体行动项
   - 负责人和时间
   - 需要的支持

6. 附录（可选）
   - 详细数据
   - 补充材料

要求：
- 每页标注建议的图表/可视化方式
- 标注每页预计讲解时长
- 考虑受众关注点，调整内容侧重
```

## 五、沟通协作类

### 5.1 跨部门沟通

```
模板：跨部门协作需求沟通

角色：你是一位擅长跨部门协作的产品经理，能够用对方听得懂的语言沟通需求。

任务：将以下产品需求转化为面向{{target_department}}的沟通材料。

需求概述：{{requirement_summary}}
背景说明：{{background}}
期望交付：{{expected_delivery}}
时间节点：{{timeline}}

请输出：
1. 沟通开场：如何引起对方兴趣和重视
2. 需求说明：
   - 用对方部门的语言描述需求
   - 说明对对方部门的价值和影响
3. 协作方式：
   - 需要对方提供的具体支持
   - 协作流程和对接人
4. 风险与应对：
   - 可能遇到的阻力
   - 应对策略
5. 跟进计划：
   - 后续沟通节奏
   - 里程碑检查点

要求：
- 站在对方角度思考，强调双赢
- 具体明确，避免模糊表述
- 预留讨论空间，避免命令式语气
```

### 5.2 用户沟通

```
模板：用户反馈回复

角色：你是一位用户运营专家，擅长真诚、专业地回复用户反馈。

任务：针对以下用户反馈撰写回复。

用户反馈：{{user_feedback}}
用户类型：{{user_type}}
反馈渠道：{{channel}}

请输出：
1. 回复内容（可直接发送）
   - 表达感谢和理解
   - 回应用户具体问题
   - 说明后续行动
   - 适当的结束语

2. 内部处理建议
   - 问题分类和优先级
   - 责任部门
   - 处理时限
   - 是否需要升级

要求：
- 语气真诚，避免官话套话
- 不承诺无法实现的事项
- 负面反馈也要转化为正向表达
```

## 六、模板使用指南

### 6.1 如何使用模板

```
使用步骤：

1. 选择模板
   - 根据当前任务类型选择对应模板
   - 阅读模板说明，确认适用性

2. 填充变量
   - 将 {{variable_name}} 替换为实际内容
   - 删除不适用的部分

3. 调整优化
   - 根据具体场景增减内容
   - 调整语气和风格

4. 测试验证
   - 用示例数据测试输出质量
   - 根据测试结果微调

5. 保存复用
   - 将调整后的版本保存为团队模板
   - 记录使用效果和优化建议
```

### 6.2 模板管理建议

| 管理维度 | 建议 |
|---------|------|
| **分类** | 按工作场景分类（需求、设计、分析、文档、沟通） |
| **版本** | 记录模板版本和更新历史 |
| **效果** | 追踪模板使用效果，定期淘汰低效模板 |
| **共享** | 建立团队模板库，促进经验复用 |
| **权限** | 核心模板由专人维护，避免随意修改 |

**关键认知**：
1. 模板是Prompt工程的最佳实践沉淀，能显著提升工作效率
2. 使用模板时需要根据具体场景调整，不要生搬硬套
3. 建立团队级模板库，促进知识共享和协作效率
4. 定期回顾和更新模板，淘汰过时或低效的内容
5. 好的模板应该经过实际验证，有明确的使用场景和效果数据
