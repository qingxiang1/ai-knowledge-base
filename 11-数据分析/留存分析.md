## 一、留存分析基础概念

### 1.1 定义与标准

```
留存分析定义体系：

核心定义
├── 基本定义
│   ├── 留存率 = 某时段内新增用户在后续时段内仍然活跃的比例
│   ├── 时间范围：根据业务特点选择（次日、7日、30日等）
│   └── 计算维度：按用户唯一标识（user_id）去重
│
├── 留存类型
│   ├── 新增留存
│   │   └── 以新增用户为基准计算留存
│   ├── 活跃留存
│   │   └── 以活跃用户为基准计算留存
│   └── 回流留存
│       └── 以回流用户为基准计算留存
│
└── 留存口径
    ├── 严格口径
    │   ├── 定义：完成核心业务操作
    │   ├── 优势：反映真实产品价值
    │   └── 适用：产品成熟期
    └── 宽松口径
        ├── 定义：任意页面访问
        ├── 优势：数字好看
        └── 适用：产品早期

相关概念
├── 流失率
│   └── 1 - 留存率
├── 回流率
│   └── 流失用户中再次活跃的比例
└── 活跃渗透率
    └── 活跃用户 / 总注册用户
```

```python

"""

class RetentionCalculation:
    """留存计算"""

    def get_calculation_methods() -> dict:
        """
        获取计算方法
        
        Returns:
            方法定义
        """
        return {
            "基础计算": {
                "公式": "留存率 = 后续时段活跃用户数 / 基准时段新增用户数",
                "说明": "统计基准时段新增用户在后续时段的留存比例",
                "SQL 示例": """
                WITH new_users AS (
                    SELECT user_id, MIN(DATE(event_time)) as first_date
                    FROM user_events
                    GROUP BY user_id
                ),
                retention AS (
                    SELECT 
                        n.first_date,
                        COUNT(DISTINCT CASE WHEN DATEDIFF(DATE(e.event_time), n.first_date) = 1 THEN e.user_id END) as d1_retained,
                        COUNT(DISTINCT CASE WHEN DATEDIFF(DATE(e.event_time), n.first_date) = 7 THEN e.user_id END) as d7_retained,
                        COUNT(DISTINCT CASE WHEN DATEDIFF(DATE(e.event_time), n.first_date) = 30 THEN e.user_id END) as d30_retained,
                        COUNT(DISTINCT n.user_id) as total_new
                    FROM new_users n
                    LEFT JOIN user_events e ON n.user_id = e.user_id
                    GROUP BY n.first_date
                )
                SELECT 
                    first_date,
                    d1_retained / total_new as d1_retention,
                    d7_retained / total_new as d7_retention,
                    d30_retained / total_new as d30_retention
                FROM retention
                """
            },
            "分层计算": {
                "核心留存": "完成核心业务的用户留存",

                "公式": "次日活跃用户数 / 新增用户数",
                "意义": "反映产品首日体验"
            },
```

```python
# 留存常见误区
"""
留存常见误区：

1. 唯留存论
   ├── 问题：只关注留存数字，忽视质量
   ├── 后果：用户增长但价值不增长
   └── 解决：结合时长、价值综合评估

2. 口径混乱
   ├── 问题：不同团队留存定义不同
   ├── 后果：数据对不上，争论不休
   └── 解决：统一口径，文档化，定期对齐

3. 刷量造假
   ├── 问题：通过激励刷出虚假留存
   ├── 后果：数据好看，实际无价值
   └── 解决：定义严格活跃标准，监控异常行为

4. 忽视结构
   ├── 问题：只看总数，不看构成
   ├── 后果：掩盖老用户流失问题
   └── 解决：拆分新老用户、分层分析

5. 过度反应
   ├── 问题：单日波动就紧张，频繁调整
   ├── 后果：策略摇摆，团队疲惫
   └── 解决：关注趋势而非单日，设置合理阈值
"""
```
