# Project02 - AI翻译助手

> 这是一个适合学习 AI 翻译产品原型的项目实战案例。当前项目采用 `React + TypeScript + Vite + Express` 实现，支持多语言互译、翻译风格切换和 mock/API 双模式演示，但它还不是一个已经具备术语库、文档翻译、批量翻译和翻译质量评估能力的企业级翻译平台。

## 一、先理解这个项目在练什么

这个项目最适合训练的是以下三类能力：

1. 如何把一个很常见的 AI 能力做成清晰、直接、可演示的单点工具。
2. 如何围绕“输入文本 -> 选择参数 -> 获取结果”设计一个最小可用产品闭环。
3. 如何理解翻译类 AI 产品从 Demo 到企业级工具之间的能力差距。

如果你是初学者，这个项目能帮助你快速理解一个 AI 工具型产品最小 MVP 的典型结构：

- 输入区
- 参数区
- 结果区

如果你是技术人员，这个项目适合你练习：

- React 表单与状态管理
- 前后端接口串联
- mock 模式与真实模型模式切换
- 文本类工具的前端交互细节

如果你是想转型 AI 产品经理，这个项目很适合用来理解：

- 单点 AI 工具如何定义核心价值
- 体验层设计如何影响产品易用性
- 为什么“能翻译”不等于“翻得适合业务场景”

## 二、项目目标与用户场景

这个项目要模拟的问题可以概括为一句话：

> 用户输入一段文本，指定源语言、目标语言和翻译风格后，系统快速返回一版可阅读的翻译结果。

它适合模拟的典型场景包括：

- 日常中英文内容互译
- 技术文档快速翻译
- 海外资料初步理解
- 不同表达风格的文案转换

这类场景很适合做成 AI 工具原型，因为它天然具备：

- 明确输入
- 明确输出
- 用户即时反馈

但要注意，当前项目更准确的定位是“文本翻译原型”，而不是完整翻译工作台。

## 三、当前项目能力概览

根据真实代码，这个项目当前已经实现了：

- 源语言与目标语言选择
- 一键交换源语言和目标语言
- `正式 / 口语 / 技术` 三种翻译风格
- 文本翻译接口调用
- mock/API 双模式切换
- 后端返回备选翻译字段

但它当前也有非常明确的边界：

- 不支持自动识别语言
- 不支持文档上传翻译
- 不支持长文分段处理
- 不支持术语库
- 不支持翻译历史
- 前端当前没有展示后端返回的 `alternatives`

因此，这个项目更适合被理解为：

> 一个聚焦“单段文本翻译体验”的 AI 工具原型，而不是完整翻译产品。

## 五、技术栈与实现方式

### 1. 当前真实可运行部分

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 模型 SDK：OpenAI Node SDK
- 本地开发：前端 `5173`，后端 `3000`

### 2. 当前最值得注意的实现特点

这个项目虽然小，但有几个很适合学习的点：

- 前端支持多语言切换
- 前端支持翻译方向互换
- 前端支持风格参数
- 后端支持 mock 模式降级

这说明它不是一个只会“把文本发给模型”的最粗糙 Demo，而是已经开始考虑工具产品的可用性。

### 3. 当前没有落地的企业级翻译能力

当前项目还没有实现：

- 行业术语库
- 公司品牌语气统一
- 多版本翻译结果比较
- 文档批量翻译
- 人工校对流
- 质量评分与回译校验

所以你在阅读这个项目时，重点不应该是它“功能多不多”，而是它是否把最核心的翻译体验闭环做清楚了。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置如下：

```env
OPENAI_API_KEY=
PORT=3000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：进入 mock 模式，适合演示和学习
- 配置 `OPENAI_API_KEY`：进入 API 模式，调用真实模型进行翻译

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:3000/health`

项目在 `vite.config.ts` 中把 `/api` 代理到了 `http://localhost:3000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/components/Translator.tsx`

这是整个项目最关键的前端文件。

它把翻译工具最核心的体验拆得很清楚：

- 输入原文
- 选择源语言
- 选择目标语言
- 选择翻译风格
- 查看结果

同时还有一个很有工具感的细节：

- 支持一键交换源语言和目标语言

这类细节虽然实现简单，但对工具产品体验很重要。

### 2. `src/services/api.ts`

这个文件负责封装 `translateText()` 请求。

它的价值在于：

- 让组件层不直接写 `fetch`
- 统一错误处理
- 为后续增加鉴权、超时或日志埋点留出空间

### 3. `src/types/index.ts`

这个文件定义了翻译请求和响应结构。

特别值得注意的是响应里有：

- `translatedText`
- `alternatives`

这说明后端设计上已经考虑了“多版本翻译候选”的可能性，只是前端当前还没有把这部分能力展示出来。

### 4. `server/routes/translate.ts`

这是当前项目最值得认真看的后端文件。

它做了几件关键的事：

- 校验输入文本不能为空
- 根据是否有 `OPENAI_API_KEY` 切换 mock/API 模式
- 根据 `style` 选择不同翻译风格指令
- 把源语言和目标语言映射成中文语义标签

mock 模式下，它并不是简单返回固定一句话，而是会根据语言和风格生成不同的演示结果，这一点很适合学习。

### 5. `server/index.ts`

这个文件展示了一个最轻量的 AI 工具型后端形态：

- 注册 `/api`
- 提供 `/health`
- 展示当前运行模式

对新手来说，这个结构足够简单，也足够实用。

## 八、从产品和技术角度，应该如何理解这个项目

### 1. 从产品角度看

这个项目已经具备翻译产品原型最重要的三个感觉：

- 参数明确
- 结果即时
- 交互简单

这足以支撑一个很清晰的单点工具演示。

### 2. 从技术角度看

它当前主要完成的是：

- 前端翻译交互
- 风格参数透传
- 后端翻译接口
- mock/API 双模式

而真正决定企业翻译工具价值的底层能力还没有落地，例如：

- 专业术语控制
- 文本上下文一致性
- 长文分片翻译
- 多版本候选排序
- 翻译质量校验

### 3. 从学习角度看

这个项目很适合作为“AI 单点工具产品化”的练习样本：

1. 先把一个能力做成清晰可用的工具。
2. 再增强输入参数和交互体验。
3. 最后再往专业场景和企业需求扩展。

## 九、如果你是产品经理，应该重点观察什么

### 1. 翻译产品真正的差异，不只在模型能力

很多人会以为翻译产品的核心只是“模型翻得准不准”。

实际上真实用户还会关心：

- 风格是否合适
- 术语是否统一
- 结果是否可直接使用
- 是否支持业务上下文

当前项目虽然简单，但已经开始从“风格控制”切入这一层。

### 2. 工具产品很依赖细节体验

像这个项目里的这些点都很重要：

- 语言切换
- 方向互换
- 加载状态
- 结果只读展示

这些细节决定了用户是否愿意频繁使用。

### 3. 企业翻译系统真正难的是一致性和流程

真实企业不会只要求“翻译出来”。

它们通常更关心：

- 品牌表达是否统一
- 技术术语是否统一
- 是否需要人工复核
- 是否能批量处理内容

当前项目适合你用来理解“基础翻译工具”向“专业翻译系统”演化的方向。

## 十、建议你至少做一次改造

如果你要把这个项目真正写进作品集，建议至少做一次有价值的改造。

### 改造方向 1：展示备选翻译

这是一个非常自然且成本较低的升级。

后端已经返回了 `alternatives`，你可以在前端增加：

- 备选翻译列表
- 一键选用
- 不同风格比较

### 改造方向 2：支持自动识别语言

当前必须手动选择源语言。

你可以继续补齐：

- 自动检测输入语言
- 语言检测结果提示
- 自动填充源语言

### 改造方向 3：增加翻译历史

当前翻译结果停留在当前页面状态里。

你可以增加：

- 历史记录列表
- 最近翻译缓存
- 一键再次编辑

### 改造方向 4：支持文档翻译

如果想往企业级方向推进，可以继续增加：

- 上传文档
- 分段翻译
- 导出翻译结果

### 改造方向 5：增加术语库或场景模板

这是把“通用翻译”升级为“业务翻译工具”的关键。

可以考虑增加：

- 技术术语模板
- 商务邮件模板
- 产品文档模板
- 公司品牌语气模板

## 十一、怎么把这个项目讲成作品集

面试时不要只说“我做了一个翻译工具”，而要讲清楚你对工具产品化的理解。

你可以这样表达：

> 我做了一个 AI 翻译助手原型，前端实现了多语言选择、翻译风格切换和翻译方向互换，后端实现了 mock/API 双模式翻译接口。我在这个项目中重点关注翻译类工具从“模型输出”到“用户可用结果”的转化过程，并识别了风格控制、术语一致性、备选翻译展示和历史沉淀等后续演化方向。

这种表达会比单纯说“做了个翻译页面”更有产品含量。

## 十二、最小复盘模板

完成这个项目后，建议你至少回答下面几个问题：

1. 为什么翻译产品不能只看“能不能翻译”？
2. 当前项目里，哪些体验细节最有助于工具化？
3. 后端已经返回 `alternatives`，前端没展示，这说明了什么产品状态？
4. 如果只能做一次升级，你会优先补自动识别语言、翻译历史还是术语库？为什么？
5. 企业翻译系统最关键的价值是速度、准确率，还是一致性？

如果这些问题你都能回答清楚，这个项目就真正沉淀成你的产品理解了。

## 十三、这个项目适合作为什么阶段的练习

这个项目很适合作为以下阶段的训练样本：

- 刚开始做 AI 工具型产品
- 想练习“输入参数 + 结构清晰结果”的工具体验
- 想理解单点 AI 能力如何做成最小 MVP
- 想向 AI 产品经理转型，需要一个容易讲清楚的入门案例

如果你前面已经做过 AI 写作助手，那么这个项目可以帮助你进一步理解“生成类工具”和“翻译类工具”在产品设计上的差异。
