# Project03 - 企业知识库

> 这是一个适合学习 AI 知识库产品原型的项目实战案例。它当前采用 `React + TypeScript + Vite + Express` 实现前后端分离，支持文档上传、文档列表、问答交互和 mock/API 双模式演示，但还不是一个已经完整打通真实 RAG 流程的企业级知识库系统。

## 一、先理解这个项目在练什么

这个项目最适合练的是三件事：

1. 你如何把“上传资料 + 提问问答”做成一个最小可演示的 AI 产品原型。
2. 你如何把前端页面、后端接口、文件上传和模型调用串成一个完整闭环。
3. 你如何识别一个“看起来像知识库”的 Demo，距离真正的企业级 RAG 系统还有哪些关键差距。

如果你是初学者，这个项目可以帮助你建立 AI 知识库产品的基本认知。

如果你是技术人员，这个项目适合你用来练习：

- React 前端交互组织
- Express 接口拆分
- 文件上传流程
- mock 模式与真实 API 模式切换
- 从 Demo 继续演化到真实 RAG 系统的改造路径

如果你是想转型 AI 产品经理，这个项目最值得观察的地方不是“它有多复杂”，而是：

- 知识库产品最小 MVP 应该先做到什么
- 什么功能只是演示层
- 什么能力才是真正决定企业可用性的底层能力

## 二、项目目标与用户场景

这个项目要解决的问题可以概括为一句话：

> 让用户上传企业资料后，可以围绕资料发起问答，快速得到一个看起来像“企业知识助手”的产品原型。

它适合模拟的典型场景包括：

- 新员工查找制度、流程和操作规范
- 客服团队查询产品知识、售后说明和常见问题
- 销售团队查询方案资料、报价说明和行业案例
- 内部团队围绕项目文档、需求文档、培训资料进行问答

但要注意，当前项目更偏向“知识库产品原型”，而不是“已经打通真实文档检索链路的生产系统”。

## 三、当前项目能力概览

结合真实代码看，这个项目当前已经具备以下能力：

- 前端展示聊天区、文档列表和上传弹窗
- 支持上传 `pdf`、`docx`、`md`、`txt` 文件
- 支持查看已上传文档列表并删除文档
- 支持向后端提交问题并展示回答结果
- 支持展示“参考来源”和相关度
- 支持 `OPENAI_API_KEY` 存在与否两种运行模式

但它当前也有非常明确的边界：

- 文档数据保存在服务端内存对象中，服务重启后不会保留
- 上传后的 `chunk_count` 是随机生成的演示值，不是真实分块结果
- 没有真正把上传文档解析、嵌入、入向量库、检索召回串起来
- 即使配置了 `OPENAI_API_KEY`，当前问答路由也是直接调用模型回答，而不是基于上传文档做真实 RAG 检索增强
- “参考来源”当前也是演示型返回，不是来自真实检索结果

这恰恰是这个项目最值得学习的地方：你能非常清楚地看到一个 AI Demo 的最小闭环，以及它与企业级系统之间的落差。

## 五、技术栈与实现方式

当前项目的技术栈可以拆成两层来理解。

### 1. 当前真正已经跑起来的实现

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 文件上传：Multer
- 接口通信：基于浏览器 `fetch` 的简单封装
- 本地开发：前端 `5173`，后端 `8000`

### 2. 当前“预留了依赖，但还未完整落地”的方向

从 `server/package.json` 可以看到服务端已经安装了：

- `langchain`
- `langchain-openai`
- `chromadb`

这说明项目设计意图确实是朝知识库/RAG 方向走的，但从当前路由代码看，真实检索链路还没有接上，所以你应该把它理解为：

> 一个带有 RAG 演化方向的知识库原型，而不是一个已经完成真实 RAG 实现的知识库成品。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置很简单：

```env
OPENAI_API_KEY=
PORT=8000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：系统进入 mock 模式，适合演示和学习
- 配置 `OPENAI_API_KEY`：问答接口会调用真实模型，但当前仍不是严格意义上的真实文档 RAG

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:8000/health`

项目在 `vite.config.ts` 中配置了代理，因此前端访问 `/api` 时会自动转发到 `http://localhost:8000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/App.tsx`

这里决定了整个页面骨架：

- 左侧主区域是 `ChatInterface`
- 右侧侧栏是 `DocumentList`
- 顶部按钮打开 `UploadModal`
- 上传完成后通过 `refreshDocs` 触发文档列表刷新

你可以把它理解为一个非常典型的 AI 工具页布局：

- 顶部是全局动作
- 中间是核心交互
- 右侧是上下文资产

### 2. `src/components/ChatInterface.tsx`

这个组件体现了知识库问答前端最基本的交互闭环：

- 用户输入问题
- 本地先插入用户消息
- 调用 `askQuestion()`
- 返回后插入助手消息
- 渲染参考来源
- 异常时显示错误提示

这部分很适合新手理解“AI 聊天页面最小可用交互”应该如何组织。

### 3. `src/components/DocumentList.tsx`

这里体现的是知识库产品的第二条主线：文档资产管理。

它当前实现了最基础的三件事：

- 拉取文档列表
- 展示标题与状态
- 删除文档

虽然能力很轻，但对知识库产品来说这是必须存在的，因为知识问答不是孤立能力，它一定依赖“资料资产管理”。

### 4. `src/components/UploadModal.tsx`

上传弹窗展示了知识库最重要的输入入口：

- 选择文件
- 输入标题
- 输入描述
- 调用上传接口

这部分虽然简单，但它决定了未来是否能扩展为：

- 批量上传
- 文件夹上传
- URL 导入
- 飞书/Notion/Confluence 同步

### 5. `server/routes/documents.ts`

这个文件非常值得认真看，因为它清楚暴露了项目的“当前真实能力”和“未来可升级方向”。

当前真实实现是：

- 使用 `multer` 接收上传文件
- 将文件临时存入 `uploads/`
- 用内存对象 `documents_db` 维护文档记录
- 返回文档元数据给前端

但它还没有完成真实知识库系统必须具备的动作：

- 解析文件正文
- 文本切块
- 生成 embedding
- 写入向量数据库
- 依据真实处理结果回填状态

所以你读这个文件时，最重要的不是“它做了什么”，而是“它还缺什么”。

### 6. `server/routes/chat.ts`

这是当前项目最关键的后端文件之一。

它实现了两种模式：

- mock 模式：返回固定结构的演示答案和演示来源
- API 模式：调用 OpenAI 直接回答问题

但它目前没有真正把上传文档检索进来，因此要特别注意：

- `doc_ids` 只是参与返回数据拼装，并没有驱动真实检索
- `top_k` 也没有进入真实向量搜索流程
- “参考来源”并不是从真实向量召回结果中生成

如果你能看懂这里，你就会理解一个很重要的产品判断：

> 很多知识库 Demo 展示的是“AI 回答界面”，但真正难的是“知识入库、检索、引用、权限、更新、追溯”这一整套底层能力。

### 7. `src/services/api.ts`

这个文件负责统一封装前端 API 请求。

它的价值在于：

- 让组件层不用直接写 `fetch`
- 把上传、列表、删除、提问等动作集中管理
- 为后续接入鉴权、重试、错误码映射留下空间

## 八、从产品和技术角度，应该如何理解这个项目

这个项目非常适合作为一个“知识库产品演化样本”来学习。

### 1. 从产品角度看

它已经具备知识库产品的三个基础面：

- 资料进入系统
- 用户围绕资料发起问题
- 系统返回答案和来源

这意味着它已经具备了演示型 MVP 的基本形态。

### 2. 从技术角度看

它当前主要完成的是：

- 页面搭建
- 接口串联
- 文件上传
- 模型调用
- mock 降级

而真正决定企业知识库质量的关键能力，还没有落地完整版本，例如：

- 文档解析质量
- 分块策略
- 向量检索效果
- 重排序策略
- 权限隔离
- 数据持久化
- 知识更新机制

### 3. 从学习角度看

这个项目最大的价值不是“直接上线”，而是让你建立一条非常清晰的认知链路：

1. 先做一个能演示的知识库原型。
2. 再识别哪些地方只是前台展示。
3. 最后继续往真实 RAG、企业级治理和多数据源接入演化。

## 九、如果你是产品经理，应该重点观察什么

如果你想从这个项目里学习 AI 产品设计，建议重点观察下面几个问题。

### 1. 上传文档不等于知识库完成

很多初学者会把“文件能上传”误当成“知识库已经建好”。

实际上中间还隔着很多关键环节：

- 文档解析是否成功
- 文档如何切块
- 分块粒度是否合理
- 向量是否生成
- 检索是否准确
- 回答是否带引用
- 引用是否可追溯

### 2. 问答页面不等于真实 RAG

当前项目的聊天界面已经很像一个知识助手，但这并不意味着它已经实现真实的知识增强问答。

这会帮助你形成一个很重要的判断：

> AI 产品中“体验层可演示”与“底层能力可交付”往往不是同一阶段完成的。

### 3. 知识库产品真正难的是数据治理

一个真正可落地的企业知识库，难点通常不在聊天框，而在：

- 文档来源杂乱
- 数据持续更新
- 同名知识冲突
- 权限隔离复杂
- 回答可信度难评估

这个项目现在正好适合作为“前台原型已搭好，后台治理还没展开”的学习样本。

## 十、建议你至少做一次改造

如果你准备把这个项目真正写进作品集，建议不要只停留在“把它跑起来”，而是至少完成一次有价值的改造。

### 改造方向 1：把上传文档真正接入 RAG

这是优先级最高的改造。

你可以继续补齐：

- 文件内容解析
- 文本分块
- embedding 生成
- ChromaDB 入库
- 查询时按 `doc_ids` 检索
- 将真实检索片段返回前端展示

完成这一步后，这个项目才会真正接近“知识库”而不是“问答 Demo”。

### 改造方向 2：补齐文档处理状态流转

当前文档状态展示已经存在，但还比较演示化。

你可以继续补充：

- `pending -> processing -> completed/failed`
- 失败原因展示
- 重试处理
- 文档更新时间
- 文档分块数量真实统计

### 改造方向 3：支持基于选中文档提问

当前前端虽然保留了 `doc_ids` 参数能力，但并没有形成完整的文档筛选问答体验。

你可以增加：

- 勾选文档
- 仅针对已选文档提问
- 在回答来源里高亮命中文档

这会让产品更像企业内部实际会使用的知识助手。

### 改造方向 4：把数据从内存迁移到持久化存储

当前文档记录和对话记录都在内存中，服务重启后会丢失。

你可以考虑补齐：

- PostgreSQL 存文档元数据
- 对话记录持久化
- 对象存储保存原始文件
- 向量库持久化

### 改造方向 5：补齐企业级能力

如果你要把这个项目往企业级方向推进，可以继续增加：

- 用户登录与权限控制
- 多租户隔离
- 数据源同步
- 回答引用跳转
- 审计日志
- 命中率和问答效果分析

## 十一、怎么把这个项目讲成作品集

如果你面试时要介绍这个项目，不要只说“我做了一个企业知识库”，而要讲清楚产品边界和你的认知深度。

你可以这样表达：

> 我做了一个企业知识库原型，前端实现了文档上传、文档列表和问答交互，后端完成了文件上传接口、问答接口和 mock/API 双模式切换。我在这个项目中重点拆解了知识库产品从演示型 MVP 到真实 RAG 系统的演化路径，识别了文档解析、向量检索、引用溯源、数据持久化和权限治理等关键差距，并设计了后续改造方案。

这样的表达会比“我做了一个 AI 聊天页面”更有说服力。

## 十二、最小复盘模板

你做完这个项目后，至少应该回答下面几个问题：

1. 当前项目里，哪些能力是真实实现的，哪些只是演示型实现？
2. 为什么“文档上传成功”不等于“知识库已经可用”？
3. 当前问答流程为什么还不能算完整 RAG？
4. 如果你只能做一次升级，你会优先补齐哪一段链路？
5. 如果把它用在企业场景，最先会遇到什么数据治理问题？

如果你能把这几个问题回答清楚，这个项目就不只是“跑通过”，而是真正进入你的产品与技术理解里了。

## 十三、这个项目适合作为什么阶段的练习

这个项目非常适合作为以下阶段的训练样本：

- 刚开始学 AI 应用开发，想先做一个能演示的知识库原型
- 已经了解聊天产品，想进一步理解知识库产品结构
- 想从 Demo 过渡到 RAG 系统，但还不清楚差距在哪里
- 想向 AI 产品经理转型，需要一个能同时看产品结构与技术边界的案例

如果你前一个项目已经做过 AI 写作助手，那么这个项目正好帮助你从“单轮内容生成”走向“文档资产 + 检索问答”的下一阶段。
