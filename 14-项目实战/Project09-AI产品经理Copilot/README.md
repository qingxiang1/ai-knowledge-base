# Project09 - AI产品经理Copilot

> 这是一个适合学习 AI 产品经理辅助工具原型的项目实战案例。当前项目采用 `React + TypeScript + Vite + Express` 实现，前端提供 PRD 生成和竞品分析两个标签页，目标是把产品经理常见的文档工作做成 AI 辅助工具，但它还不是一个已经完整打通需求管理、竞品资料沉淀、协作评审和工作流编排的产品经理工作台。

## 一、先理解这个项目在练什么

这个项目最适合训练的是以下三类能力：

1. 如何把“产品经理日常文档工作”拆成可以被 AI 辅助的明确任务。
2. 如何把大模型输出包装成结构化文档结果，而不是只返回一段自由文本。
3. 如何识别一个 AI Copilot 原型和真正工作台系统之间的差距。

如果你是初学者，这个项目能帮助你理解一个 AI Copilot 产品最小 MVP 的典型形态：

- 有明确任务入口
- 有参数输入表单
- 有可阅读的生成结果

如果你是技术人员，这个项目适合你练习：

- 多标签页单页应用
- 表单状态管理
- 结构化结果渲染
- 前后端接口对齐
- mock 模式与真实模型模式切换

如果你是想转型 AI 产品经理，这个项目很有代表性，因为它直接对应你熟悉的工作内容：

- 生成 PRD
- 做竞品分析
- 未来可以继续扩展用户故事、方案评估、需求拆解等任务

## 二、项目目标与用户场景

这个项目要模拟的问题可以概括为一句话：

> 当产品经理输入产品背景、目标用户和功能信息后，系统能帮助生成 PRD 草稿，或者围绕指定竞品输出一份结构化分析结果。

它适合模拟的场景包括：

- 新项目立项时快速生成 PRD 初稿
- 做方案比选时先产出一版竞品分析框架
- 在头脑风暴阶段快速整理产品思路
- 把零散描述先整理成可讨论的文档雏形

这类场景非常适合 AI 化，因为它们天然具备：

- 输入信息半结构化
- 输出内容模板化
- 大量重复性整理工作

但要注意，当前项目更准确的定位是“产品文档辅助原型”，而不是完整的产品管理平台。

## 三、当前项目能力概览

根据真实代码，这个项目前端当前已经实现了：

- `PRD 生成` 标签页
- `竞品分析` 标签页
- PRD 参数输入
- 竞品分析参数输入
- 结构化结果展示

根据后端真实接线状态，这个项目当前的能力边界需要分开看：

### 1. 前端已具备的能力

- 展示 PRD 结果标题与章节
- 展示竞品分析摘要、维度对比和优化建议

### 2. 服务端当前真实挂载的能力

`server/index.ts` 当前只挂载了：

- `server/routes/copilot.ts`

这意味着目前真正暴露出来的接口是：

- `POST /api/prd`
- `POST /api/user-stories`

### 3. 当前存在的实际对齐问题

前端调用的是：

- `POST /api/prd`
- `POST /api/competitor`

而 `POST /api/competitor` 这个接口实际上定义在：

- `server/routes/pm.ts`

但 `pm.ts` 当前并没有在 `server/index.ts` 里注册。

这意味着一个非常关键的真实结论：

> 当前项目的 PRD 生成链路可以工作，但“竞品分析”标签页按现状大概率无法正常调用后端接口。

这也是这个项目最值得学习的地方之一：你能看到一个典型的 AI 原型在“界面完成了，但前后端接线还没完全打通”时会出现什么问题。

## 五、技术栈与实现方式

### 1. 当前真实可运行部分

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 模型 SDK：OpenAI Node SDK
- 本地开发：前端 `5173`，后端 `3000`

### 2. 当前最值得注意的实现特点

这个项目有两个很典型的 AI 文档工具特征：

- PRD 输出采用结构化章节形式展示
- 竞品分析输出采用摘要 + 对比 + 建议的形式展示

这说明它的目标不是“聊天问答”，而是“工作产物生成”。

### 3. 当前实现中的关键不一致

从代码看，后端存在两套思路：

- `copilot.ts`：更偏大模型生成完整文档字符串
- `pm.ts`：更偏前端需要的结构化对象格式

但当前入口只挂了前者，这会造成：

- 前端 PRD 类型与后端 `copilot.ts` 返回结构并不完全一致
- 前端竞品分析接口在现有接线路径下无法工作

这也说明本项目当前最重要的问题不是“模型没接上”，而是“接口契约还没统一”。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置如下：

```env
OPENAI_API_KEY=
PORT=3000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：进入 mock 模式
- 配置 `OPENAI_API_KEY`：进入 API 模式，调用真实模型生成内容

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:3000/health`

项目在 `vite.config.ts` 中把 `/api` 代理到了 `http://localhost:3000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/components/PMCopilot.tsx`

这是整个项目最关键的前端文件。

它把产品经理 Copilot 的两个核心场景拆成两个标签页：

- PRD 生成
- 竞品分析

这是一种非常典型的 Copilot 产品设计方式：

- 不做纯聊天
- 直接围绕具体任务入口提供辅助

这也是这个项目最值得学习的地方之一。

### 2. `src/services/api.ts`

这个文件定义了前端当前希望调用的能力：

- `generatePRD()`
- `generateUserStories()`
- `analyzeCompetitors()`

这可以帮助你直接看出前端对后端的“产品能力期待”。

### 3. `src/types/index.ts`

这里定义了前端所期待的响应结构：

- PRD：`title + sections`
- 竞品：`summary + comparison + suggestions`

这说明前端明显更希望后端返回的是“结构化对象”，而不是一整段 Markdown 字符串。

### 4. `server/routes/copilot.ts`

这是当前真正被挂到入口上的后端路由文件。

它的特点是：

- 提供 `POST /prd`
- 提供 `POST /user-stories`
- 返回结果更偏“整块内容生成”

这部分很适合你理解“大模型直接生成文档”这种最初级的 Copilot 接法。

### 5. `server/routes/pm.ts`

这个文件虽然没有接到入口，但非常值得看。

因为它展示了另一种更贴近前端页面的实现思路：

- PRD 返回结构化章节
- 竞品分析返回结构化对比结果

换句话说，它更像是“前端真正想要的接口形态”。

因此这个文件很适合被看作：

- 当前项目的另一版实现
- 或者未来重构时更应该靠拢的接口契约样式

### 6. `server/index.ts`

这个文件暴露了当前项目最关键的接线问题：

- 只注册了 `copilotRoutes`
- 没有注册 `pm.ts`

你只要看懂这个文件，就能理解为什么界面看起来很完整，但某些功能可能实际不可用。

## 八、从产品和技术角度，应该如何理解这个项目

### 1. 从产品角度看

这个项目已经具备 Copilot 产品原型最重要的感觉：

- 围绕具体工作任务提供辅助
- 输入信息后快速生成工作产物
- 输出结果可直接用于修改和讨论

这比“通用聊天助手”更接近真实办公场景。

### 2. 从技术角度看

它当前主要完成的是：

- 前端任务入口设计
- 表单输入与结果展示
- 后端 PRD 生成功能
- 用户故事生成能力
- 结构化竞品分析的原型接口设计

而真正决定产品经理工作台价值的关键能力还没有落地，例如：

- 历史文档管理
- 多轮编辑
- 版本差异对比
- 需求协作评审
- 输出导出
- 数据源沉淀

### 3. 从学习角度看

这个项目非常适合作为“AI Copilot 产品化”的学习样本：

1. 先做具体任务入口，而不是万能聊天框。
2. 再把结果做成结构化产物。
3. 最后继续补齐工作台、流程和协作能力。

## 九、如果你是产品经理，应该重点观察什么

### 1. Copilot 产品要围绕任务，而不是围绕模型

当前项目最值得肯定的一点是它把任务入口定义得很明确：

- 生成 PRD
- 做竞品分析

这比“问我任何问题”更像一个真实工具。

### 2. 接口契约不统一，会直接破坏用户体验

这个项目最大的现实问题之一，不是生成质量，而是前后端能力契约还没有统一。

这会直接导致：

- 某些页面能显示但调不通
- 某些返回结构前端处理不了
- UI 看起来完成了，但实际不可交付

这对产品经理是一个很重要的提醒：

> AI 产品的可用性，很多时候卡在接口契约和结果结构，而不是只卡在模型本身。

### 3. 产品经理工具真正难的是沉淀与协作

真实企业里，产品经理不会只生成一版 PRD 就结束。

真正难的是：

- 如何修改
- 如何评审
- 如何记录版本
- 如何共享给设计、研发和业务

当前项目非常适合作为“从单点生成工具走向协作工作台”的起点样本。

## 十、建议你至少做一次改造

如果你要把这个项目真正写进作品集，建议至少做一次高价值改造。

### 改造方向 1：统一前后端接口契约

这是当前优先级最高的改造。

你可以选择两种方向之一：

- 把 `pm.ts` 挂到 `server/index.ts`
- 或者把 `copilot.ts` 的返回结构改成前端当前需要的形态

同时统一：

- 字段命名
- 返回结构
- 错误处理

### 改造方向 2：补齐竞品分析可用链路

当前前端已经有“竞品分析”页，但后端接线不完整。

你可以补齐：

- `POST /api/competitor`
- 统一竞品分析结构
- 增加空状态与异常兜底

### 改造方向 3：支持用户故事生成入口

当前后端已经有 `/user-stories`，但前端还没有单独入口。

你可以继续增加：

- 用户故事标签页
- 功能描述输入
- Story 列表展示
- 优先级与点数展示

### 改造方向 4：增加文档历史与导出能力

当前结果只停留在页面展示层。

你可以增加：

- 历史记录保存
- 复制 Markdown
- 导出为文档
- 对比不同版本内容

### 改造方向 5：把 Copilot 做成完整工作台

如果想往企业级方向推进，可以继续增加：

- 需求拆解
- 里程碑规划
- 会议纪要联动
- 用户反馈归纳
- 与知识库联动

## 十一、怎么把这个项目讲成作品集

面试时不要只说“我做了一个 AI 产品经理工具”，而要讲清楚你对 Copilot 产品形态的理解。

你可以这样表达：

> 我做了一个 AI 产品经理 Copilot 原型，前端围绕 PRD 生成和竞品分析两个典型任务设计了结构化输入与结果展示，后端接入了 PRD 生成和用户故事生成能力。我在这个项目中重点识别了 AI Copilot 从单点生成到工作台产品化过程中的关键问题，尤其是接口契约统一、结果结构化、历史沉淀和协作扩展等能力，并针对当前前后端接线不一致的问题设计了后续改造方案。

这种表达能明显体现你不仅会“接模型”，还理解“工具产品化”。

## 十二、最小复盘模板

完成这个项目后，建议你至少回答下面几个问题：

1. 为什么 Copilot 产品更适合做任务入口，而不是万能聊天框？
2. 当前项目里，PRD 生成和竞品分析的真实可用性差异在哪里？
3. `copilot.ts` 和 `pm.ts` 并存说明了什么问题？
4. 如果只能做一次升级，你会优先统一接口契约还是扩新功能？为什么？
5. 产品经理工作台真正的长期价值来自生成能力，还是来自沉淀与协作？

如果这些问题你都能回答清楚，这个项目就真正沉淀成你的产品理解了。

## 十三、这个项目适合作为什么阶段的练习

这个项目很适合作为以下阶段的训练样本：

- 想从通用 AI 工具转向垂直角色 Copilot 产品
- 想练习“任务型 AI 工具”的界面与接口设计
- 想理解 AI 产品经理工具的最小 MVP 形态
- 想向 AI 产品经理转型，需要一个和自己工作最贴近的案例

如果你前面已经做过写作、知识库、Agent、客服、会议纪要等项目，那么这个项目会帮助你进入“角色型 Copilot 产品”这一阶段。
