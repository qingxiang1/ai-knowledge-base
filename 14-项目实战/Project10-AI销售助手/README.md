# Project10 - AI销售助手

> 这是一个适合学习 AI 销售辅助产品原型的项目实战案例。当前项目采用 `React + TypeScript + Vite + Express` 实现，前端提供 `销售话术` 和 `客户画像` 两个标签页，目标是把销售准备和客户分析做成 AI 辅助工具，但它还不是一个已经具备 CRM、线索管理、商机推进、客户跟进记录和销售流程自动化能力的企业级销售系统。

## 一、先理解这个项目在练什么

这个项目最适合训练的是以下三类能力：

1. 如何把销售团队常见的准备工作拆成可由 AI 辅助的明确任务。
2. 如何把“销售说什么”和“客户怎么看”做成两个不同的产品入口。
3. 如何识别销售辅助原型和真正销售工作台之间的差距。

如果你是初学者，这个项目能帮助你快速理解一个 AI 销售工具最小 MVP 的典型形态：

- 有明确任务入口
- 有结构化输入表单
- 有可阅读的辅助结果

如果你是技术人员，这个项目适合你练习：

- React 多标签页状态管理
- 前后端接口串联
- 结构化结果渲染
- mock/API 双模式理解

如果你是想转型 AI 产品经理，这个项目非常适合你理解：

- AI 如何切入销售工作流
- 销售工具为什么要围绕场景来设计
- 为什么“会生成文案”不等于“能帮助成交”

## 二、项目目标与用户场景

这个项目要模拟的问题可以概括为一句话：

> 当销售人员输入产品信息或客户线索后，系统帮助生成销售话术，或者输出客户画像与沟通策略，提升销售前期准备效率。

它适合模拟的典型场景包括：

- 冷启动外呼前准备话术
- 产品演示前梳理重点卖点
- 跟进客户前整理沟通策略
- 根据客户描述快速形成基础画像

这类场景很适合做成 AI 原型，因为它天然具备：

- 输入上下文明确
- 输出结果偏模板化
- 用户能快速判断是否有帮助

但要注意，当前项目更准确的定位是“销售辅助原型”，不是完整销售系统。

## 三、当前项目能力概览

根据真实代码，这个项目前端当前已经实现了：

- `销售话术` 标签页
- `客户画像` 标签页
- 产品信息输入
- 客户信息输入
- 结果卡片展示

但结合后端真实接口看，这个项目有几个非常关键的能力边界。

### 1. 前端希望得到的结果

前端当前期望的返回结构包括：

- 销售话术：`script + keyPoints + objections`
- 客户画像：`profile + painPoints + needs + approach`

### 2. 后端当前真实返回的结果

`server/routes/sales.ts` 当前更偏向返回：

- 话术接口：以整段 `script` 文本为主
- 客户分析接口：以整段 `analysis` 文本为主

### 3. 当前最关键的接口契约问题

前端发送给 `/script` 的字段是：

- `productName`
- `productFeatures`
- `targetCustomer`
- `scenario`

但后端实际读取的是：

- `product`
- `customerType`
- `scenario`

前端发送给 `/analyze` 的字段是：

- `customerInfo`
- `industry`
- `companySize`

但后端实际读取的是：

- `customerInfo`
- `analysisType`

这意味着一个非常重要的真实判断：

> 当前项目的界面看起来已经完成，但前后端接口字段和结果结构并没有真正对齐，按现状部分功能大概率只能拿到不完整或不符合前端预期的结果。

这也是这个项目最值得学习的地方之一。

## 五、技术栈与实现方式

### 1. 当前真实可运行部分

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 模型 SDK：OpenAI Node SDK
- 本地开发：前端 `5173`，后端 `3000`

### 2. 当前最值得注意的实现特点

这个项目最值得学习的一点，是它把销售辅助拆成两个不同工作对象：

- 面向“销售自己”的话术准备
- 面向“客户理解”的画像分析

这比只做一个万能聊天框更接近真实业务场景。

### 3. 当前没有落地的企业级销售能力

当前项目还没有实现：

- 客户线索池
- 跟进记录
- CRM 数据接入
- 商机阶段管理
- 销售漏斗
- 销售团队协同

所以你在阅读这个项目时，重点应该放在“销售辅助切入口是否设计合理”，而不是它是否像一个完整 CRM。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置如下：

```env
OPENAI_API_KEY=
PORT=3000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：进入 mock 模式
- 配置 `OPENAI_API_KEY`：进入 API 模式，调用真实模型生成销售内容

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:3000/health`

项目在 `vite.config.ts` 中把 `/api` 代理到了 `http://localhost:3000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/components/SalesAssistant.tsx`

这是整个项目最关键的前端文件。

它把销售助手拆成两个工作标签：

- 销售话术
- 客户画像

这是一种非常典型的业务型 AI 工具设计方式，因为它直接围绕用户的工作任务来组织界面。

### 2. `src/services/api.ts`

这个文件定义了前端当前希望调用的两个能力：

- `generateSalesScript()`
- `analyzeCustomerProfile()`

这让我们能很清楚地看到前端对产品能力的期待。

### 3. `src/types/index.ts`

这里定义了前端理想中的返回结构：

- 销售话术包含重点和异议处理
- 客户画像包含痛点、需求和销售策略

这说明前端希望得到的是“结构化销售辅助结果”，而不是一整段笼统文本。

### 4. `server/routes/sales.ts`

这是当前项目最值得认真看的后端文件。

它一方面已经提供了两个核心接口：

- `/script`
- `/analyze`

另一方面也暴露了当前项目最真实的问题：

- 请求字段没有和前端统一
- 返回结构没有和前端类型统一

这会导致一个典型的原型阶段问题：

界面和交互已经有了，但数据契约还不稳定。

### 5. `server/index.ts`

这个文件确认了当前服务端结构非常轻量：

- 注册 `/api`
- 暴露 `/health`
- 区分 `mock/api` 模式

它很适合帮助学习者理解一个垂直业务 AI 工具后端最小需要哪些部分。

## 八、从产品和技术角度，应该如何理解这个项目

### 1. 从产品角度看

这个项目已经具备销售辅助原型最重要的几个感觉：

- 围绕具体销售任务
- 输入上下文后生成辅助内容
- 输出结果可以直接参考使用

这足以支撑一个有业务感的最小演示。

### 2. 从技术角度看

它当前主要完成的是：

- 前端任务入口设计
- 表单输入与结果展示
- 后端话术生成
- 后端客户分析
- mock/API 模式切换

而真正决定销售系统价值的底层能力还没有落地，例如：

- CRM 联动
- 销售漏斗状态
- 客户跟进历史
- 商机评分
- 团队协同机制

### 3. 从学习角度看

这个项目很适合作为“垂直业务 AI 工具”的学习样本：

1. 先围绕真实业务任务设计入口。
2. 再让输出尽可能结构化。
3. 最后再补齐数据契约与业务流程。

## 九、如果你是产品经理，应该重点观察什么

### 1. 销售辅助产品要围绕销售动作，而不是围绕模型

这个项目比较正确的一点，是它没有做成一个泛化聊天机器人，而是围绕：

- 话术准备
- 客户分析

这更符合销售工作的自然流程。

### 2. 接口契约不统一会直接削弱产品可用性

这个项目最真实的问题不是模型能力，而是前后端没有真正对齐：

- 输入字段不一致
- 返回结构不一致

这会直接导致：

- 页面能打开
- 但结果不完整
- 或者界面预期和后端返回不匹配

这对产品经理是非常重要的一课。

### 3. 企业销售系统真正难的是闭环而不是文案生成

真实销售团队更关心的是：

- 这个客户下一步怎么跟
- 话术是否匹配阶段
- 客户画像是否能转成实际跟进动作
- 数据是否沉淀回 CRM

当前项目正好停留在“前期辅助”阶段，适合你思考它向业务闭环扩展的方向。

## 十、建议你至少做一次改造

如果你要把这个项目真正写进作品集，建议至少做一次高价值改造。

### 改造方向 1：统一前后端字段和返回结构

这是当前最优先的改造。

你需要统一：

- 前端请求字段名
- 后端读取字段名
- 前端期望的结果结构
- 后端实际返回结构

### 改造方向 2：让客户画像结果真正结构化

当前前端希望显示：

- `profile`
- `painPoints`
- `needs`
- `approach`

你可以让后端真正返回这几个字段，而不是只给一段 `analysis` 文本。

### 改造方向 3：让销售话术结果更贴近场景

当前话术虽然有场景概念，但前后端的场景值也存在偏差：

- 前端有 `demo / objection`
- 后端更偏 `meeting / negotiation`

你可以继续统一场景枚举，并让生成内容更贴近真实销售阶段。

### 改造方向 4：增加跟进记录或客户历史

当前结果只停留在页面里，没有沉淀。

你可以增加：

- 跟进历史
- 客户备注
- 话术收藏

### 改造方向 5：接入 CRM 或销售线索流程

如果想往企业级方向推进，可以继续增加：

- 线索录入
- 商机状态管理
- 跟进提醒
- 团队协作分配

## 十一、怎么把这个项目讲成作品集

面试时不要只说“我做了一个 AI 销售助手”，而要讲清楚你对销售工作流的理解。

你可以这样表达：

> 我做了一个 AI 销售助手原型，前端围绕销售话术和客户画像两个核心任务设计了输入表单和结构化结果展示，后端提供了 mock/API 双模式的销售内容生成能力。我在这个项目中重点识别了销售辅助产品从内容生成到业务闭环之间的关键差距，尤其是前后端接口契约统一、结果结构化和与 CRM 体系联动等问题，并设计了后续改造方向。

这种表达会比单纯说“做了个销售页面”更有业务深度。

## 十二、最小复盘模板

完成这个项目后，建议你至少回答下面几个问题：

1. 为什么销售助手更适合围绕任务入口，而不是泛化聊天？
2. 当前项目里，前后端字段不一致会带来什么真实问题？
3. 为什么客户画像不能只返回一段文字，而要结构化输出？
4. 如果只能优先做一次升级，你会先统一接口契约还是先扩新功能？为什么？
5. 企业销售系统最关键的价值是生成话术，还是推动销售动作闭环？

如果这些问题你都能回答清楚，这个项目就真正沉淀成你的产品理解了。

## 十三、这个项目适合作为什么阶段的练习

这个项目很适合作为以下阶段的训练样本：

- 想从通用 AI 工具转向垂直业务 AI 产品
- 想练习销售场景的任务型 AI 工具
- 想理解“文案生成”如何走向“业务辅助”
- 想向 AI 产品经理转型，需要一个业务味更强的案例

如果你前面已经做过客服、会议纪要、Copilot、数据分析等项目，那么这个项目会帮助你进入“销售场景 AI 化”的下一阶段。
