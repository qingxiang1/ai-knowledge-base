# Project11 - AI招聘助手

> 这是一个适合学习 AI 招聘辅助产品原型的项目实战案例。当前项目采用 `React + TypeScript + Vite + Express` 实现，前端提供 `简历评估` 和 `面试问题` 两个标签页，目标是帮助招聘或面试前置准备，但它还不是一个已经具备岗位管理、简历库、候选人流程、面试协同和 ATS 集成能力的企业级招聘系统。

## 一、先理解这个项目在练什么

这个项目最适合训练的是以下三类能力：

1. 如何把招聘流程中的重复性工作拆成明确的 AI 任务入口。
2. 如何让 AI 输出围绕招聘评估结构展开，而不是只返回一段模糊文本。
3. 如何识别招聘助手原型与真实招聘工作台之间的差距。

如果你是初学者，这个项目能帮助你快速理解一个 AI 招聘工具最小 MVP 的典型结构：

- 有清晰的招聘任务入口
- 有结构化输入
- 有结构化评估结果

如果你是技术人员，这个项目适合你练习：

- React 多标签页界面
- 表单状态管理
- 前后端接口联动
- 路由版本并存时的接线判断

如果你是想转型 AI 产品经理，这个项目很有代表性，因为它直接对应招聘场景里的两个高频任务：

- 简历评估
- 面试问题生成

## 二、项目目标与用户场景

这个项目要模拟的问题可以概括为一句话：

> 当招聘人员输入简历和职位描述，或者输入岗位信息后，系统快速输出评估结果或面试问题，帮助招聘团队更高效地完成前置判断。

它适合模拟的典型场景包括：

- 筛选候选人前做简历初评
- 准备一轮结构化面试问题
- 快速梳理候选人与岗位的匹配度
- 为面试官生成问题清单

这类场景很适合 AI 化，因为它们天然具备：

- 输入材料相对标准化
- 输出模板相对固定
- 人工工作重复度高

但要注意，当前项目更准确的定位是“招聘辅助原型”，而不是完整招聘系统。

## 三、当前项目能力概览

根据真实代码，这个项目前端当前已经实现了：

- `简历评估` 标签页
- `面试问题` 标签页
- 结构化评估结果展示
- 结构化面试问题展示

但结合服务端真实接线状态，这个项目存在非常关键的能力边界。

### 1. 前端当前希望得到的能力

前端当前调用的是：

- `POST /api/evaluate-resume`
- `POST /api/interview-questions`

并期望后端返回：

- 简历评估：`score / summary / strengths / weaknesses / matchDetails`
- 面试问题：`questions[]`，每项包含 `type / question / purpose`

### 2. 服务端当前真正挂到入口的能力

`server/index.ts` 当前只挂载了：

- `server/routes/recruit.ts`

这意味着当前实际暴露的接口主要是：

- `POST /api/job-description`
- `POST /api/screen-resume`
- `POST /api/interview-questions`

### 3. 当前最关键的真实问题

仓库中还存在一个：

- `server/routes/recruitment.ts`

这个文件反而更接近前端当前的接口期待，因为它定义了：

- `POST /api/evaluate-resume`
- `POST /api/interview-questions`

而且返回结构也更贴近前端类型。

所以当前项目最真实的判断是：

> 前端页面设计明显是朝 `recruitment.ts` 这套路由结构靠拢的，但服务端启动入口实际接的是 `recruit.ts`，导致简历评估接口按现状大概率不可用，而面试问题接口虽然同名存在，但前后端期望的请求字段和结果结构也并不完全一致。

这也是这个项目最值得学习的地方之一。

## 五、技术栈与实现方式

### 1. 当前真实可运行部分

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 模型 SDK：OpenAI Node SDK
- 本地开发：前端 `5173`，后端 `3000`

### 2. 当前最值得注意的实现特点

这个项目把招聘工作拆成两个典型任务入口：

- 简历评估
- 面试问题

这是一种很合理的 AI 工具设计方式，因为招聘流程天然由多个明确任务组成，而不是一个大而泛的聊天框。

### 3. 当前没有落地的企业级招聘能力

当前项目还没有实现：

- 候选人库
- 岗位管理
- 招聘流程阶段管理
- 面试反馈协作
- ATS 集成
- Offer 流转

所以你在阅读这个项目时，重点应该放在“招聘辅助任务切分是否合理”，而不是它是否像一个完整招聘平台。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置如下：

```env
OPENAI_API_KEY=
PORT=3000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：进入 mock 模式
- 配置 `OPENAI_API_KEY`：进入 API 模式，调用真实模型生成招聘辅助结果

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:3000/health`

项目在 `vite.config.ts` 中把 `/api` 代理到了 `http://localhost:3000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/components/RecruitmentAssistant.tsx`

这是整个项目最关键的前端文件。

它把招聘助手拆成两个清晰任务：

- 简历评估
- 面试问题生成

这种设计非常贴近真实招聘工作流，因为招聘场景里最常见的不是“随便问”，而是：

- 看这个人适不适合岗位
- 这个岗位该怎么面

### 2. `src/services/api.ts`

这个文件非常值得认真看，因为它直接暴露了前端真正希望后端提供的接口：

- `generateJobDescription()`
- `screenResume()`
- `evaluateResume()`
- `generateInterviewQuestions()`

其中前两个来自 `recruit.ts` 的风格，后两个又更像 `recruitment.ts` 的风格。

这说明当前项目的接口设计思路并没有完全统一。

### 3. `src/types/index.ts`

这里定义了前端理想中的结构化结果：

- 简历评估有分数、摘要、优缺点、技能匹配
- 面试问题有类型、问题、目的

这说明前端明显想把招聘输出做成“可决策、可阅读、可复用”的结构化结果。

### 4. `server/routes/recruit.ts`

这是当前真正挂到入口上的路由文件。

它提供了：

- 岗位描述生成
- 简历筛选
- 面试问题生成

但它与前端当前展示页面并不是完全一致的一套路由设计。

### 5. `server/routes/recruitment.ts`

虽然它没有接到当前入口，但这个文件反而更贴近前端当前页面：

- 提供 `evaluate-resume`
- 提供结构化 `interview-questions`

因此这个文件非常值得看，因为它更像“前端期望的后端版本”。

### 6. `server/index.ts`

这个文件暴露了当前项目最关键的接线问题：

- 只注册了 `recruit.ts`
- 没有注册 `recruitment.ts`

你只要看懂这一点，就会明白为什么页面和接口之间存在明显偏差。

## 八、从产品和技术角度，应该如何理解这个项目

### 1. 从产品角度看

这个项目已经具备招聘辅助原型最重要的几个感觉：

- 围绕真实招聘任务设计入口
- 输出结构化结果
- 让招聘人员更快进入下一步判断

这比“做一个万能 HR 聊天机器人”更接近真实工具。

### 2. 从技术角度看

它当前主要完成的是：

- 前端任务型页面
- 结构化结果展示
- 后端招聘辅助接口原型
- mock/API 模式切换

而真正决定招聘产品价值的底层能力还没有落地，例如：

- 候选人流程流转
- 面试记录沉淀
- 多面试官协同
- 候选人比较
- 人才库搜索

### 3. 从学习角度看

这个项目很适合作为“角色工作流 AI 化”的学习样本：

1. 先围绕招聘中的典型任务做入口。
2. 再让输出尽量结构化。
3. 最后再补齐系统层能力和接口契约。

## 九、如果你是产品经理，应该重点观察什么

### 1. 招聘工具比通用聊天更需要结构化输出

招聘人员真正关心的不是“AI说了什么”，而是：

- 候选人值不值得面
- 哪些技能匹配
- 哪些地方要重点追问

当前前端设计已经明显朝这个方向走了。

### 2. 路由接线错误会直接破坏可用性

这个项目最真实的问题之一，就是：

- 前端设计和类型已经定义好了
- 但服务端入口接到了另一套路由

这会造成非常典型的产品问题：

- 页面看起来完成了
- 实际却调不到正确结果

### 3. 企业招聘系统真正难的是协同和流程

真实招聘团队更关心的是：

- 简历谁来筛
- 面试意见怎么汇总
- 候选人怎么推进阶段
- 不同岗位怎么复用问题模板

当前项目正好停留在“前置辅助生成”阶段，适合你思考它怎么往流程协同扩展。

## 十、建议你至少做一次改造

如果你要把这个项目真正写进作品集，建议至少做一次高价值改造。

### 改造方向 1：统一启动入口与前端接口

这是当前优先级最高的改造。

你可以选择：

- 把 `recruitment.ts` 挂到 `server/index.ts`
- 或者把前端改成完全匹配 `recruit.ts` 当前接口

关键是统一：

- 路由路径
- 请求字段
- 返回结构

### 改造方向 2：补齐简历评估完整链路

当前前端使用的是 `evaluate-resume`，但入口没有挂上对应路由。

你可以优先把这个链路打通，让：

- 分数
- 摘要
- 优势
- 不足
- 技能匹配

都真实返回。

### 改造方向 3：统一面试问题结构

当前两个路由对 `interview-questions` 的输入输出设计并不完全一致。

你可以统一：

- 类型枚举
- 问题数量参数
- 返回字段

### 改造方向 4：增加岗位描述入口

当前后端其实已经有 `job-description` 能力，但前端还没有对应页面。

你可以增加：

- 岗位描述生成标签页
- 职位职责与要求输入
- JD 输出与复制

### 改造方向 5：往招聘流程工具扩展

如果想往企业级方向推进，可以继续增加：

- 候选人列表
- 面试反馈记录
- 面试问题模板库
- Offer 决策支持

## 十一、怎么把这个项目讲成作品集

面试时不要只说“我做了一个 AI 招聘助手”，而要讲清楚你对招聘工作流的理解。

你可以这样表达：

> 我做了一个 AI 招聘助手原型，前端围绕简历评估和面试问题两个招聘高频任务设计了结构化输入与结果展示，后端提供了岗位描述、简历筛选和面试问题等辅助能力。我在这个项目中重点识别了招聘辅助产品从单点生成到招聘流程工具的关键差距，尤其是前后端路由接线、接口契约统一和结构化结果可交付性问题，并设计了后续改造方案。

这样的表达会比“做了个招聘页面”更有业务深度。

## 十二、最小复盘模板

完成这个项目后，建议你至少回答下面几个问题：

1. 为什么招聘助手更适合围绕任务入口，而不是泛化聊天？
2. 当前项目中，`recruit.ts` 和 `recruitment.ts` 并存说明了什么问题？
3. 为什么简历评估结果必须结构化，而不能只返回一段文本？
4. 如果只能优先做一次升级，你会先统一路由接线还是扩新功能？为什么？
5. 企业招聘系统最关键的价值是生成内容，还是推动候选人流程协同？

如果这些问题你都能回答清楚，这个项目就真正沉淀成你的产品理解了。

## 十三、这个项目适合作为什么阶段的练习

这个项目很适合作为以下阶段的训练样本：

- 想从通用 AI 工具转向人力资源场景 AI 产品
- 想练习招聘场景的任务型 AI 工具
- 想理解结构化评估结果如何设计
- 想向 AI 产品经理转型，需要一个业务流程味更强的案例

如果你前面已经做过客服、销售、会议纪要、Copilot 等项目，那么这个项目会帮助你进入“招聘场景 AI 化”的下一阶段。
