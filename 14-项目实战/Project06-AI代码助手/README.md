# Project06 - AI代码助手

> 这是一个适合学习 AI 代码助手产品原型的项目实战案例。当前项目采用 `React + TypeScript + Vite + Express` 实现，支持 `生成代码 / 解释代码 / 优化代码 / 修复代码` 四种模式，并提供 mock/API 双模式演示，但它还不是一个已经具备工程上下文、代码仓库理解、测试执行、补全建议和 IDE 深度集成能力的企业级代码助手。

## 一、先理解这个项目在练什么

这个项目最适合训练的是以下三类能力：

1. 如何把通用大模型能力包装成面向开发者的代码工具。
2. 如何把“代码相关请求”拆成明确的操作模式，而不是只做一个聊天框。
3. 如何理解代码助手 Demo 与真实 AI 编程工具之间的能力差距。

如果你是初学者，这个项目能帮助你快速理解一个 AI 编程工具最小 MVP 的典型结构：

- 输入需求或代码
- 选择处理动作
- 查看代码结果和说明

如果你是技术人员，这个项目适合你练习：

- React 表单与状态管理
- 多动作模式切换
- 代码文本展示与复制
- mock/API 双模式切换

如果你是想转型 AI 产品经理，这个项目很适合你理解：

- AI 编程工具为什么必须围绕任务入口设计
- 为什么“能生成代码”不等于“能帮助开发”
- 为什么真实代码助手通常比单次生成复杂得多

## 二、项目目标与用户场景

这个项目要模拟的问题可以概括为一句话：

> 当开发者输入代码需求或代码片段，并选择处理动作后，系统快速返回一版代码结果，帮助用户完成代码生成、理解、优化或修复。

它适合模拟的典型场景包括：

- 生成一个简单函数或模块
- 解释一段已有代码
- 优化一段实现逻辑
- 修复明显 bug

这类场景非常适合做成 AI 工具原型，因为它们天然具备：

- 输入目标明确
- 输出结果直观
- 用户能快速判断是否有帮助

但要注意，当前项目更准确的定位是“代码文本助手原型”，不是完整的 AI 编程环境。

## 三、当前项目能力概览

根据真实代码，这个项目当前已经实现了：

- 语言选择
- `生成 / 解释 / 优化 / 修复` 四种动作模式
- 文本需求输入
- 代码结果展示
- 一键复制结果
- 说明文本展示
- mock/API 双模式

但它当前也有非常明确的边界：

- 不支持文件级上下文
- 不支持项目级理解
- 不支持测试执行
- 不支持代码 diff 展示
- 不支持错误堆栈上传
- API 模式下当前主要把模型结果直接放进 `code` 字段，`explanation` 没有像 mock 模式那样真正拆分

因此，这个项目更适合被理解为：

> 一个面向开发者的单轮代码处理原型，而不是完整代码助手产品。

## 五、技术栈与实现方式

### 1. 当前真实可运行部分

- 前端：React 18 + TypeScript + Vite
- 样式：Tailwind CSS
- 后端：Express + TypeScript
- 模型 SDK：OpenAI Node SDK
- 本地开发：前端 `5173`，后端 `3000`

### 2. 当前最值得注意的实现特点

这个项目最值得学习的地方之一，是它没有把所有需求都塞进一个统一入口，而是先定义了四种操作模式：

- `generate`
- `explain`
- `optimize`
- `fix`

这是一种非常典型的 AI 工具产品化方式，因为它能显著降低用户输入歧义。

### 3. 当前没有落地的企业级代码助手能力

当前项目还没有实现：

- 仓库上下文感知
- 多文件改动建议
- 单元测试生成
- Lint/类型错误联动
- PR Review
- IDE 内联交互

所以你在阅读这个项目时，重点不是它“像不像 Cursor/Claude Code”，而是它有没有把最小代码助手体验表达清楚。

## 六、运行方式

### 1. 安装前端依赖

在项目根目录执行：

```bash
pnpm install
```

### 2. 安装服务端依赖

进入 `server` 目录执行：

```bash
cd server
pnpm install
```

### 3. 配置环境变量

服务端提供了 `.env.example`，你可以复制为 `.env`：

```bash
cd server
cp .env.example .env
```

默认配置如下：

```env
OPENAI_API_KEY=
PORT=3000
```

说明如下：

- 不配置 `OPENAI_API_KEY`：进入 mock 模式，适合学习和演示
- 配置 `OPENAI_API_KEY`：进入 API 模式，调用真实模型处理代码任务

### 4. 启动后端

在 `server` 目录执行：

```bash
pnpm dev
```

或者在项目根目录执行：

```bash
pnpm server
```

### 5. 启动前端

在项目根目录执行：

```bash
pnpm dev
```

### 6. 访问地址

- 前端页面：`http://localhost:5173`
- 后端健康检查：`http://localhost:3000/health`

项目在 `vite.config.ts` 中把 `/api` 代理到了 `http://localhost:3000`。

## 七、这个项目最值得看的代码点

建议你按下面顺序阅读。

### 1. `src/components/CodeAssistant.tsx`

这是整个项目最关键的前端文件。

它把代码助手最核心的交互拆得很清楚：

- 选择语言
- 选择动作
- 输入需求
- 获取结果
- 复制代码

这部分非常适合学习开发者工具产品的最小体验设计。

特别值得注意的是：

- 代码结果被放在深色代码区里展示
- 支持一键复制
- 说明和代码分区展示

这些都是典型的开发者工具 UI 细节。

### 2. `src/services/api.ts`

这个文件封装了 `generateCode()` 请求。

它的价值在于：

- 统一接口入口
- 集中处理错误
- 为后续接入流式返回、鉴权和取消请求留空间

### 3. `src/types/index.ts`

这个文件定义了前端期望的响应结构：

- `code`
- `explanation`
- `language`

这说明前端明显希望后端返回的是“代码结果 + 辅助说明”的双层结构，而不只是单纯一段文本。

### 4. `server/routes/code.ts`

这是当前项目最值得认真看的后端文件。

它完成了几件关键的事：

- 根据动作切换不同系统提示词
- mock 模式返回更完整的代码与说明
- API 模式调用真实模型

其中最值得注意的是：

- mock 模式下 `explanation` 很完整
- API 模式下 `explanation` 目前默认是空字符串

这说明当前项目还处于“真实模型已接入，但结果结构还没完全产品化”的阶段。

### 5. `server/index.ts`

这个文件展示了一个非常轻量的代码助手后端形态：

- 注册 `/api`
- 提供 `/health`
- 区分 `mock/api` 模式

对学习者来说，这个结构足够简单，也非常适合作为入门样板。

## 八、从产品和技术角度，应该如何理解这个项目

### 1. 从产品角度看

这个项目已经具备代码助手原型最重要的几个感觉：

- 面向明确开发任务
- 输出是可直接复制的代码
- 结果有一定说明性

这足以支撑一个开发者工具方向的最小原型。

### 2. 从技术角度看

它当前主要完成的是：

- 前端交互设计
- 代码结果展示
- 后端任务分类处理
- mock/API 双模式切换

而真正决定代码助手价值的底层能力还没有落地，例如：

- 真实工程上下文理解
- 测试与验证
- 代码修改建议
- 多文件关系推理
- 错误定位链路

### 3. 从学习角度看

这个项目很适合作为“AI 开发者工具最小 MVP”的练习样本：

1. 先围绕明确动作定义产品入口。
2. 再把结果做成代码 + 说明的结构。
3. 最后再往上下文化、自动化和工程化演化。

## 九、如果你是产品经理，应该重点观察什么

### 1. 代码助手产品必须围绕任务设计

这个项目最值得肯定的一点，是它没有把一切都交给一个开放式输入框，而是先做了动作约束：

- 生成
- 解释
- 优化
- 修复

这能显著提升用户预期稳定性。

### 2. 开发者工具非常看重结果可消费性

用户不只是想“看到答案”，更想要：

- 能复制
- 能理解
- 能快速判断是否可用

当前项目已经开始用代码区、说明区和复制按钮来解决这个问题。

### 3. 企业级代码助手真正难的是上下文和验证

真实代码助手最难的往往不是生成几行代码，而是：

- 知道当前项目结构
- 理解已有代码约束
- 判断改动是否正确
- 验证结果能不能跑

当前项目正好适合作为“交互层已搭好，上下文层还没展开”的学习样本。

## 十、建议你至少做一次改造

如果你要把这个项目真正写进作品集，建议至少做一次有价值的改造。

### 改造方向 1：补齐 API 模式的说明字段

这是最自然的一步升级。

你可以继续补：

- 让模型返回结构化 JSON
- 单独解析 `code`
- 单独解析 `explanation`
- 增加输出校验

### 改造方向 2：支持代码输入区

当前输入框更偏“描述需求”。

你可以升级为：

- 支持粘贴原始代码
- 区分需求描述与代码内容
- 针对 `explain / optimize / fix` 提供更合理的输入方式

### 改造方向 3：支持 Markdown 代码高亮

当前结果只是文本放进 `pre/code` 区块中。

你可以继续增加：

- 代码高亮
- 按语言切换高亮规则
- 行号显示

### 改造方向 4：增加历史记录

当前每次生成结果都停留在当前页面状态里。

你可以增加：

- 历史请求列表
- 重新编辑
- 对比不同结果

### 改造方向 5：往工程化能力推进

如果想往企业级方向延展，可以继续增加：

- 仓库上下文上传
- 文件级修复建议
- 测试用例生成
- 代码 diff 展示

## 十一、怎么把这个项目讲成作品集

面试时不要只说“我做了一个 AI 代码助手”，而要讲清楚你对开发者工具产品的理解。

你可以这样表达：

> 我做了一个 AI 代码助手原型，前端围绕生成、解释、优化和修复四类典型开发任务设计了操作入口，并实现了代码结果展示、说明展示和一键复制功能。后端实现了 mock/API 双模式代码处理接口。我在这个项目中重点关注代码助手从“模型输出”到“开发者可消费结果”的产品化过程，并识别了结构化输出、工程上下文理解和结果验证等后续演化方向。

这种表达会比“做了个写代码页面”更有产品深度。

## 十二、最小复盘模板

完成这个项目后，建议你至少回答下面几个问题：

1. 为什么代码助手产品更适合按任务模式来设计入口？
2. 当前 mock 模式和 API 模式的能力差距体现在哪里？
3. 为什么代码助手不能只返回一段代码，还需要说明和上下文？
4. 如果只能做一次升级，你会优先补结构化输出、代码输入区还是历史记录？为什么？
5. 企业级代码助手最关键的价值是生成代码，还是理解工程上下文并验证结果？

如果这些问题你都能回答清楚，这个项目就真正沉淀成你的产品理解了。

## 十三、这个项目适合作为什么阶段的练习

这个项目很适合作为以下阶段的训练样本：

- 想从通用 AI 工具转向开发者工具
- 想练习“任务型代码助手”的最小 MVP
- 想理解 AI 编程工具的产品结构
- 想向 AI 产品经理转型，需要一个开发者工具方向的案例

如果你前面已经做过翻译、知识库、Agent、客服、数据分析等项目，那么这个项目会帮助你进入“开发者场景 AI 化”的下一阶段。
