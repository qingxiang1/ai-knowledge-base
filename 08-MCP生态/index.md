# 08 - MCP 生态

> 本章聚焦 MCP（Model Context Protocol）在企业环境中的理解、设计与落地。从协议基础、服务端开发、客户端集成，到 GitHub、飞书、Notion、数据库等生态接入，帮助你把“会调用工具”升级为“会构建标准化工具平台”。

---

## 本章定位

`08-MCP生态` 解决的不是“知道 MCP 是什么”，而是以下 4 个更关键的问题：

1. 如何把零散的工具调用封装成统一、可治理、可复用的协议能力
2. 如何在 Host、Client、Server 三层之间建立稳定的权限、路由、审计与观测机制
3. 如何把内部系统、SaaS 平台和数据库能力接入成企业级工具生态
4. 如何评估 MCP 在成本、稳定性、安全性和组织协同上的真实价值

---

## 企业级学习原则

1. 先明确业务边界，再设计 MCP 工具边界
2. 先建立权限与审计，再开放写操作工具
3. 先跑通最小闭环，再建设工具市场与统一平台
4. 始终将 MCP 视为治理层，而不是单纯协议语法

## 本章学习目标

完成本章后，你应该能够：

- 理解 MCP 的核心抽象，包括 Host、Client、Server、Resources、Prompts、Tools 与 Sampling
- 区分 MCP 与 Function Calling、API Gateway、Plugin、Workflow 的边界与关系
- 设计适用于企业的 MCP Server、MCP Client 和权限治理体系
- 评估飞书、GitHub、Notion、数据库等系统接入 MCP 的收益、风险与改造成本
- 输出一套可评审、可上线、可持续演进的 MCP 接入方案

### 按阶段推进

```text
阶段一：建立统一认知
- 先读 MCP基础，搞清楚 MCP 与 Function Calling 的关系
- 再读 MCP协议，理解初始化、能力协商、请求响应与错误模型

阶段二：打通最小闭环
- 读 MCP服务端开发，完成 1 个只读工具服务
- 读 MCP客户端开发，完成 1 个 Host 或 Agent 侧集成
- 用 Inspector 或桌面客户端完成联调

阶段三：进入生产化
- 读 MCP最佳实践，补齐鉴权、审计、监控、限流、灰度与版本管理
- 选择 1 个业务系统做案例化改造，如 GitHub、飞书或数据库

阶段四：平台化扩展
- 建立工具目录、统一认证、审批机制与调用看板
- 把零散工具升级为企业内部 MCP 能力中心
```

## 阶段验收标准

完成本模块学习后，至少应满足以下验收条件：

- 能解释 MCP 与 Function Calling、普通 API 封装、插件机制的差异
- 能画出 Host、Client、Server 之间的调用链路和权限边界
- 能设计一个只读和一个写操作工具，并给出审批、限流、审计方案
- 能针对一个企业系统输出接入评估，包括收益、风险、依赖和上线门禁
- 能定义 MCP 接入后的关键指标，如调用成功率、P95 延迟、权限拒绝率、审计覆盖率

## 关键术语速查

| 术语                 | 说明                                                   |
| -------------------- | ------------------------------------------------------ |
| MCP                  | Model Context Protocol，面向模型与外部能力的标准化协议 |
| Host                 | 运行模型或承载用户体验的宿主应用                       |
| Client               | 与 MCP Server 通信、管理连接和调用的客户端层           |
| Server               | 对外暴露 Resources、Prompts、Tools 的能力提供方        |
| Resources            | 模型可读取的上下文资源                                 |
| Prompts              | 标准化提示模板                                         |
| Tools                | 模型可调用的结构化工具                                 |
| Sampling             | Server 反向请求模型生成内容的能力                      |
| Capability Discovery | 客户端发现 Server 提供能力的过程                       |
| Audit Log            | 对调用、拒绝、失败和敏感操作的审计记录                 |

## 学习建议

1. 不要把 MCP 只理解为“多一个 SDK”，它本质上是统一工具生态的协议层
2. 企业落地优先从只读工具开始，再逐步开放写操作和自动化执行
3. 优先做高频、低风险、可审计的场景，避免一开始就接入高危系统
4. 每增加一个工具，都要同步补齐 Schema、权限、日志、超时和降级策略
5. 评估成败不要只看“能不能调通”，还要看复用率、治理成本和平台收益
