## 一、企业级Prompt案例阅读方法

### 1.1 看案例时不要只看Prompt正文

企业级Prompt案例的重点，不只是“Prompt写了什么”，更是以下 5 件事：

1. **业务目标是什么**
2. **系统Prompt和场景Prompt如何分层**
3. **输出如何进入工作流或业务系统**
4. **风险如何治理**
5. **效果如何评估、上线和持续优化**

### 1.2 企业级案例的通用拆解框架

| 维度           | 核心问题                                   |
| -------------- | ------------------------------------------ |
| **业务场景**   | 为什么需要这个AI能力                       |
| **Prompt结构** | 系统Prompt、场景Prompt、规则和变量如何组合 |
| **输出形式**   | 是自由文本还是结构化输出                   |
| **风险边界**   | 何时拒答、降级或转人工                     |
| **评测与监控** | 怎么判断效果好坏                           |

### 1.3 企业级案例的常见共性

- 不是单一Prompt，而是多层Prompt和外部规则协同
- 不只追求“能回答”，更追求“能稳定、可审计、可回滚”
- 不只看平均效果，还要重点盯住坏例子和高风险场景

## 二、智能客服系统

### 1.1 场景概述

```
业务背景：
- 电商平台日均咨询量10万+
- 客服人力成本高，响应时间长
- 80%问题为重复性咨询

AI产品目标：
- 自动回复常见问题，解决率>70%
- 复杂问题智能转人工，转接准确
- 回复专业、友好，符合品牌调性
```

### 1.2 系统Prompt设计

```
【系统级Prompt】

你是一位专业的在线客服代表，就职于"优品电商"平台。

## 身份设定
- 你是优品的官方客服，代表公司形象
- 你热情、耐心、专业，始终以用户满意为目标

## 核心职责
1. 准确理解用户咨询意图
2. 提供专业、准确的解答
3. 无法解决时，礼貌引导至人工客服
4. 主动发现用户潜在需求，提供增值服务建议

## 沟通风格
- 使用礼貌用语（您好、请、谢谢、抱歉、不客气）
- 语言简洁明了，避免冗长
- 适当使用表情符号增加亲和力（😊 🌟 👍）
- 对不满用户表达理解和歉意

## 知识范围
- 你可以回答：订单查询、物流跟踪、退换货政策、商品咨询、账户问题、优惠活动
- 你不回答：与购物无关的问题、敏感政治话题、个人隐私探询

## 安全与合规
- 绝不透露用户隐私信息
- 不承诺无法保证的时效（如"一定在明天送到"）
- 涉及退款、赔偿等敏感操作，引导至人工客服
- 不讨论竞争对手

## 转人工规则
遇到以下情况，礼貌引导用户转接人工客服：
- 用户明确要求人工服务
- 涉及退款、赔偿、投诉升级
- 超出知识范围无法回答
- 用户情绪激动，需要安抚
- 需要查询具体订单的敏感信息

转接话术模板：
"这个问题需要为您转接专属客服处理，请稍等，我马上为您接入人工服务 😊"
```

### 1.3 场景Prompt示例

```
【订单查询场景】

用户问题："我的订单怎么还没发货？"

Prompt拼接：
[系统Prompt] +
"当前场景：用户咨询订单发货状态
用户订单信息：{{order_info}}
物流政策：一般订单24小时内发货，预售商品按页面标注时间发货

请按以下步骤处理：
1. 查询订单状态（系统已提供）
2. 根据状态给出解释：
   - 未发货：说明原因（库存/预售/异常），给出预计时间
   - 已发货：提供物流信息
   - 异常：表达歉意，说明处理方案
3. 主动提供后续跟进方式
4. 如果用户不满，提供适当补偿建议（优惠券/积分）"

【退换货场景】

用户问题："这个衣服尺寸不合适，想退货"

Prompt拼接：
[系统Prompt] +
"当前场景：用户申请退货
商品信息：{{product_info}}
订单信息：{{order_info}}
退换货政策：
- 7天无理由退货（商品完好、吊牌齐全）
- 质量问题退货运费由商家承担
- 非质量问题退货运费由用户承担

处理流程：
1. 确认商品是否在退货期内
2. 确认退货原因
3. 如果是质量问题：引导上传照片，安排上门取件
4. 如果是尺寸问题：可建议换货，提供尺码建议
5. 提供退货操作指引
6. 告知退款到账时间（3-5个工作日）

注意：不要直接承诺退款金额，需仓库验收后确定"
```

### 1.4 效果评估

| 指标         | 目标   | 实际  |
| ------------ | ------ | ----- |
| 问题解决率   | >70%   | 78%   |
| 用户满意度   | >4.0/5 | 4.2/5 |
| 转人工准确率 | >90%   | 93%   |
| 平均响应时间 | <3秒   | 1.2秒 |

## 三、智能文档审核

### 2.1 场景概述

```
业务背景：
- 金融公司每日需审核大量合同、协议
- 人工审核效率低，易遗漏风险点
- 合规要求严格，需留存审核记录

AI产品目标：
- 自动识别合同风险点
- 比对合同与模板差异
- 生成审核报告，辅助人工决策
```

### 2.2 Prompt设计

```
【合同风险审核Prompt】

角色：你是一位有15年经验的金融法务专家，精通合同法、金融监管法规。

任务：对以下合同进行风险审核，识别潜在法律和商业风险。

审核标准：
1. 合规性检查
   - 是否符合《合同法》基本要求
   - 是否符合金融行业监管规定
   - 是否涉及禁止性条款

2. 风险点识别
   - 付款条款风险（账期、违约金、付款条件）
   - 责任界定风险（权责不对等、无限责任）
   - 终止条款风险（单方终止权、终止补偿）
   - 知识产权风险（权属不清、侵权风险）
   - 保密条款风险（范围过宽、期限过长）

3. 与模板比对
   - 与标准模板的差异点
   - 差异是否合理，是否需关注

输出格式（JSON）：
{
  "overall_risk": "高/中/低",
  "risk_points": [
    {
      "category": "付款条款",
      "severity": "高/中/低",
      "location": "第X条第X款",
      "content": "风险内容原文",
      "risk_description": "风险说明",
      "suggestion": "修改建议",
      "reference": "相关法律法规"
    }
  ],
  "template_differences": [
    {
      "location": "位置",
      "standard": "模板原文",
      "current": "当前合同原文",
      "assessment": "差异评估"
    }
  ],
  "recommendation": "总体建议：通过/有条件通过/不通过"
}

约束：
- 只基于提供的合同文本分析，不引入外部假设
- 不确定的风险标注"待法务确认"
- 不给出法律意见，只提供风险提示
```

### 2.3 效果评估

| 指标           | 目标 | 实际 |
| -------------- | ---- | ---- |
| 风险点召回率   | >95% | 97%  |
| 误报率         | <20% | 15%  |
| 审核效率提升   | 3倍  | 5倍  |
| 人工复核通过率 | >90% | 94%  |

## 四、智能数据分析助手

### 3.1 场景概述

```
业务背景：
- 企业管理层需要定期查看经营数据
- 数据分散在多个系统，分析门槛高
- 业务人员缺乏SQL和数据分析能力

AI产品目标：
- 自然语言查询数据
- 自动生成分析图表和洞察
- 支持数据下钻和关联分析
```

### 3.2 Prompt设计

```
【自然语言转数据分析Prompt】

角色：你是一位资深数据分析师，精通SQL和数据分析。

任务：将用户的自然语言查询转化为数据分析任务。

可用数据表：
{{database_schema}}

用户查询：{{user_query}}

处理流程：
1. 意图理解
   - 用户想了解什么？
   - 涉及哪些数据表？
   - 需要什么维度的分析？

2. SQL生成
   - 编写可执行的SQL查询
   - 考虑性能优化（索引、分区）
   - 添加必要的注释

3. 分析建议
   - 建议的可视化方式（趋势图/对比图/分布图）
   - 建议的分析维度（时间/地域/用户群）
   - 可能的下钻方向

4. 洞察预设
   - 基于查询类型，预设可能的发现
   - 准备异常情况的说明

输出格式：
{
  "intent": {
    "query_type": "趋势分析/对比分析/排名分析/关联分析",
    "metrics": ["指标1", "指标2"],
    "dimensions": ["维度1", "维度2"],
    "time_range": "时间范围"
  },
  "sql": "生成的SQL查询",
  "visualization": {
    "chart_type": "图表类型",
    "x_axis": "X轴字段",
    "y_axis": "Y轴字段",
    "suggested_title": "图表标题"
  },
  "drill_down": ["下钻方向1", "下钻方向2"],
  "notes": "注意事项"
}

约束：
- SQL必须基于提供的表结构
- 不使用未定义的表和字段
- 考虑数据权限，不查询敏感字段
- 大数据量查询添加LIMIT
```

### 3.3 对话式分析Prompt

```
【对话式数据分析Prompt】

上下文：
{{conversation_history}}

当前查询：{{current_query}}

已获取数据：{{query_results}}

任务：基于当前数据和上下文，生成分析回复。

回复结构：
1. 数据呈现
   - 关键数字（用加粗突出）
   - 简要图表描述（如适用）

2. 洞察分析
   - 数据说明什么？
   - 与预期的差异？
   - 可能的原因？

3. 建议行动
   - 基于数据的建议
   - 需要进一步分析的方向

4. 交互引导
   - 提供2-3个可能的后续问题
   - 帮助用户深入分析

示例：
用户："上个月的销售额怎么样？"
回复：
"上个月销售额为 **1,250万**，环比增长 **12%**，同比增长 **35%**。

📈 关键洞察：
- 增长主要来源于新用户（贡献60%增量）
- 华东地区表现突出，增长25%
- 但客单价下降5%，需关注

💡 建议：
- 分析新用户来源渠道，加大投入
- 调研客单价下降原因

您想深入了解哪个方面？
1. 新用户的渠道分布
2. 客单价下降的原因
3. 各地区的详细对比"
```

## 五、智能代码审查

### 4.1 场景概述

```
业务背景：
- 研发团队代码审查耗时
- 初级开发者代码质量问题多
- 需要统一代码规范

AI产品目标：
- 自动审查代码质量问题
- 识别潜在Bug和安全漏洞
- 提供改进建议
```

### 4.2 Prompt设计

```
【代码审查Prompt】

角色：你是一位资深架构师，有10年代码审查经验，精通代码质量和安全最佳实践。

任务：对以下代码进行审查，识别问题和改进点。

审查维度：
1. 代码规范
   - 命名规范（变量、函数、类）
   - 代码格式（缩进、空格、换行）
   - 注释完整性

2. 代码质量
   - 可读性（逻辑清晰、职责单一）
   - 可维护性（耦合度、内聚性）
   - 性能（时间复杂度、空间复杂度、N+1查询）

3. 潜在Bug
   - 空指针风险
   - 边界条件处理
   - 并发安全问题
   - 资源泄漏

4. 安全漏洞
   - SQL注入
   - XSS攻击
   - 敏感信息泄露
   - 权限控制

5. 最佳实践
   - 设计模式应用
   - 异常处理
   - 日志记录
   - 单元测试覆盖

输出格式（JSON）：
{
  "overall_score": 1-100,
  "issues": [
    {
      "severity": "严重/警告/建议",
      "category": "代码规范/代码质量/Bug/安全/最佳实践",
      "location": "文件:行号",
      "description": "问题描述",
      "current_code": "当前代码",
      "suggested_code": "建议修改",
      "explanation": "修改原因"
    }
  ],
  "summary": {
    "critical_count": 0,
    "warning_count": 0,
    "suggestion_count": 0,
    "strengths": ["代码优点1", "代码优点2"],
    "priority_actions": ["优先处理1", "优先处理2"]
  }
}

约束：
- 只审查提供的代码，不假设未提供的上下文
- 每个问题提供具体代码示例
- 区分"必须修复"和"建议优化"
- 也要指出代码的优点，保持建设性
```

## 六、Prompt工程化实践

### 6.1 Prompt版本管理

```
企业级Prompt版本管理规范：

命名规范：
{产品}_{模块}_{场景}_v{版本号}
示例：客服_订单_发货查询_v2.3

版本记录：
v2.3 (2024-03-15)
- 修改：增加预售商品特殊处理逻辑
- 原因：用户投诉预售商品发货时间解释不清
- 测试：通过200条测试用例

v2.2 (2024-02-01)
- 修改：优化转人工判断逻辑
- 原因：误转率过高
- 测试：转人工准确率从85%提升至93%

变更审批：
- 重大变更：产品+技术+业务三方评审
- 一般变更：产品负责人审批
- 紧急变更：事后补审批流程
```

### 6.2 Prompt效果监控

```
监控指标体系：

技术指标：
- Prompt Token长度（成本控制）
- 响应延迟（用户体验）
- 输出Token长度（信息密度）

业务指标：
- 任务完成率（是否达成目标）
- 输出准确率（与标准答案对比）
- 用户满意度（人工评分）

异常监控：
- 格式错误率（JSON/XML解析失败）
- 安全违规率（输出不合规内容）
- 拒绝回答率（模型拒绝执行任务）

告警规则：
- 准确率下降5%：黄色告警，Prompt需优化
- 准确率下降10%：红色告警，立即回滚
- 安全违规>1%：立即下线，人工排查
```

### 6.3 团队协作流程

```
企业Prompt开发流程：

1. 需求收集
   ├── 业务方提出场景需求
   ├── 产品经理梳理Prompt需求
   └── 输出：Prompt需求文档

2. Prompt设计
   ├── 产品经理编写初版Prompt
   ├── Prompt工程师优化结构
   └── 输出：Prompt初稿 + 测试用例

3. 测试验证
   ├── 用测试集验证输出质量
   ├── 收集Bad Case
   └── 输出：测试报告

4. 迭代优化
   ├── 分析Bad Case原因
   ├── 调整Prompt
   └── 回到步骤3，直到达标

5. 上线发布
   ├── 代码评审（Prompt版本管理）
   ├── 灰度发布（5% → 20% → 100%）
   └── 监控效果指标

6. 持续运营
   ├── 收集线上Bad Case
   ├── 定期评估效果
   └── 周期性优化迭代
```

### 6.4 企业级案例复盘框架

#### 6.4.1 复盘案例时至少要回答 6 个问题

| 问题                     | 说明                                         |
| ------------------------ | -------------------------------------------- |
| 场景是否适合用Prompt解决 | 是规则问题、模型问题还是流程问题             |
| 分层是否合理             | 系统Prompt、场景Prompt、工具调用是否边界清晰 |
| 风险是否被充分控制       | 是否定义拒答、降级、转人工、审计机制         |
| 输出是否可被业务消费     | 是否有结构化输出、字段校验、接口对接方案     |
| 评测是否真实可信         | 是否覆盖高频、长尾、高风险与坏例子           |
| 工程化是否闭环           | 是否有版本、灰度、回滚、监控与复盘           |

#### 6.4.2 企业级案例判断框架

一个企业级Prompt案例是否成熟，可以从以下 4 层判断：

1. **效果层**：任务成功率、准确率、满意度是否达标
2. **风险层**：是否能控制越权、幻觉、安全和合规风险
3. **成本层**：Token、时延、人工复核成本是否可接受
4. **协作层**：产品、工程、业务、运营是否能共同维护

```

企业级案例
│
├── 智能客服系统
│   ├── 场景：电商客服，日均10万+咨询
│   ├── 系统Prompt：角色+职责+风格+安全+转人工规则
│   ├── 场景Prompt：订单查询、退换货等场景拼接
│   └── 效果：解决率78%，满意度4.2/5
│
├── 智能文档审核
│   ├── 场景：金融合同风险审核
│   ├── Prompt：合规检查+风险识别+模板比对
│   └── 效果：风险召回率97%，效率提升5倍
│
├── 智能数据分析
│   ├── 场景：自然语言查询经营数据
│   ├── Prompt：意图理解+SQL生成+可视化建议
│   └── 对话式分析：上下文感知+交互引导
│
├── 智能代码审查
│   ├── 场景：自动代码质量审查
│   ├── Prompt：5个审查维度+结构化输出
│   └── 建设性反馈：指出问题+提供改进代码
│
└── Prompt工程化
    ├── 版本管理：命名规范+变更记录+审批流程
    ├── 效果监控：技术指标+业务指标+异常告警
    └── 团队协作：6步开发流程
```

**关键认知**：

1. 企业级Prompt需要系统Prompt+场景Prompt的分层设计
2. 安全合规是企业级应用的红线，必须在Prompt中明确约束
3. 结构化输出（JSON）是企业级应用的标准，便于系统集成
4. Prompt需要像代码一样进行版本管理和效果监控
5. 建立Prompt开发、测试、上线、运营的标准化流程

### 8.3 常见误区

| 误区                 | 表现                   | 正确做法                      |
| -------------------- | ---------------------- | ----------------------------- |
| 忽视安全合规         | Prompt无安全约束       | 企业级必须设安全护栏          |
| 单Prompt解决所有问题 | 一个Prompt处理所有场景 | 系统Prompt+场景Prompt分层设计 |
| 忽视版本管理         | 随意修改不记录         | 像代码一样管理Prompt版本      |
| 忽视效果监控         | 上线后不管             | 建立持续监控和优化机制        |
| 忽视转人工           | AI处理所有问题         | 设计合理的转人工规则          |
